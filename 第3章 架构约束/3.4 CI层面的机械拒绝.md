# 3.4　CI 层面的机械拒绝

> 本节目标：理解 CI 在 Harness 中的角色，掌握如何设计一条让违规代码物理上无法合并的前端 CI 流水线。

---

## 一、CI 是 Harness 的最后一道防线

上一节讲了 ESLint——它在 Agent 本地工作时实时检查，发现违规立即报错。

但 ESLint 有一个漏洞：**Agent 可以忽略它。**

如果 Agent 没有正确配置本地 ESLint，或者某种原因跳过了检查，违规代码仍然可以被提交。

CI（持续集成）解决这个问题：

```
ESLint（本地）：
  Agent 工作时实时检查
  发现违规 → 报错 → Agent 修复
  依赖：Agent 的本地环境配置正确

CI（服务端）：
  代码提交 PR 时自动触发
  不通过 → PR 物理上无法合并
  不依赖：Agent 的本地环境
  → 这是真正的"机械拒绝"
```

**无论 Agent 的本地环境如何，CI 都会在服务端执行同样的检查。**

---

## 二、前端 Harness CI 流水线的结构

一条完整的前端 Harness CI 流水线通常包含以下阶段：

```
PR 提交
    ↓
┌─────────────────────────────────────┐
│  阶段一：快速检查（约 30 秒）          │
│  - 代码格式检查（Prettier）           │
│  - TypeScript 类型检查（tsc --noEmit）│
│  → 不通过：立即失败，不继续            │
└─────────────────────────────────────┘
    ↓（通过）
┌─────────────────────────────────────┐
│  阶段二：架构约束检查（约 1 分钟）     │
│  - 自定义架构 ESLint（层级规则）      │
│  - 依赖分析（循环依赖检测）           │
│  - 禁用模式检测（AI SDK 直接调用等）  │
│  → 不通过：失败，报告具体违规         │
└─────────────────────────────────────┘
    ↓（通过）
┌─────────────────────────────────────┐
│  阶段三：测试（约 2-5 分钟）          │
│  - 单元测试（Vitest）                │
│  - 组件测试（Testing Library）       │
│  - 测试覆盖率检查                    │
│  → 不通过：失败，报告失败的测试       │
└─────────────────────────────────────┘
    ↓（通过）
┌─────────────────────────────────────┐
│  阶段四：构建验证（约 1-2 分钟）      │
│  - npm run build                    │
│  - Bundle 大小检查（可选）           │
│  → 不通过：失败，报告构建错误         │
└─────────────────────────────────────┘
    ↓（通过）
┌─────────────────────────────────────┐
│  阶段五：质量检查（可选）             │
│  - 代码复杂度分析                    │
│  - 安全漏洞扫描（npm audit）         │
│  → 不通过：Warning（不阻断合并）      │
└─────────────────────────────────────┘
    ↓（通过）
PR 可以合并 ✅
```

**关键设计：阶段越靠前，检查越快。** 快速失败，避免浪费时间。

## 三、检查项分级：阻断 vs 警告

不是所有检查都应该阻断合并，需要明确分级：

```
阻断合并（必须修复才能合并）：
  ✅ 架构层级违规（ARCH-xxx ESLint 规则）
  ✅ TypeScript 类型错误
  ✅ 测试失败
  ✅ 测试覆盖率低于阈值（如 80%）
  ✅ 构建失败（npm run build 报错）
  ✅ 代码格式错误（Prettier 不通过）

警告（显示但不阻断）：
  ⚠️ 代码质量问题（组件过长、复杂度高）
  ⚠️ 轻微的编码规范违反
  ⚠️ Bundle 大小超过建议值
  ⚠️ npm audit 中低危漏洞

仅记录（不影响 PR）：
  📝 性能分析报告
  📝 依赖更新提示
```

**原则：阻断项要少而精。** 阻断项太多，Agent 会陷入无休止的修复循环，效率崩溃。

## 四、CI 错误信息的设计

和 ESLint 一样，CI 的错误信息也要对 Agent 友好。

Agent 在 CI 失败后会读取错误日志，决定如何修复。

```
差的 CI 错误输出：
  ❌ FAILED: lint
  Exit code: 1

好的 CI 错误输出：
  ❌ [ARCH-001] 架构层级违规
  文件：src/components/ChatMessage.tsx，第 8 行
  问题：Components 层直接导入了 API 层 (src/api/aiApi.ts)
  原因：依赖方向规则要求 Components 只能依赖 Hooks 层及以下
  修复：
    1. 在 src/hooks/ 中创建 useMessage Hook 封装数据获取逻辑
    2. 在 ChatMessage 组件中调用 useMessage Hook
  参考：docs/architecture.md#层级约束
```

Agent 读到第二种输出，可以直接开始修复，不需要人介入。

## 五、PR 审查流程与自动合并策略

Harness Engineering 的目标是让大多数 PR **不需要人工深度审查**。

### 可以自动合并的 PR

```
满足以下所有条件时，PR 可以配置为自动合并：
  ✅ 所有 CI 检查通过
  ✅ PR 规模小（改动文件数 < 10，行数 < 200）
  ✅ PR 类型是已知安全类型：
      - 文档更新
      - 测试补充
      - 依赖版本升级（非 major 版本）
      - 自动生成的重构 PR（来自清理 Agent）
```

### 需要人工审查的 PR

```
以下情况需要人工审查：
  ⚠️ 涉及架构变更（新增层级、修改依赖规则）
  ⚠️ 涉及鉴权相关代码（Token 管理、权限验证）
  ⚠️ 大规模改动（文件数 > 10 或行数 > 200）
  ⚠️ 新增外部依赖（引入新的第三方库）
  ⚠️ 修改 ESLint 规则或 CI 配置
```

### 人工审查的节奏

OpenAI 团队的实践：**大多数 PR 在 1 分钟内完成审查。**

这是因为：
- CI 已经验证了代码的正确性
- 架构约束已经保证了结构的合理性
- 人工审查只需确认"结果是否符合预期"，不需要逐行检查代码

```
1 分钟审查流程：
  1. 看 PR 描述（Agent 自动生成）：做了什么？（10秒）
  2. 看 CI 结果：全绿？（5秒）
  3. 看关键文件的改动：结构是否合理？（30秒）
  4. 合并或留评论（15秒）
```

## 六、GitHub Actions 配置示例

以 GitHub Actions 为例，展示一个前端 Harness CI 的基本结构：

```yaml
# .github/workflows/harness-ci.yml

name: Harness CI

on:
  pull_request:
    branches: [main]

jobs:
  # 阶段一：快速检查
  quick-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - name: 格式检查
        run: npx prettier --check "src/**/*.{ts,tsx}"
      - name: TypeScript 类型检查
        run: npx tsc --noEmit

  # 阶段二：架构约束检查
  arch-check:
    needs: quick-check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - name: 架构层级检查（自定义 ESLint 规则）
        run: npx eslint src --rule 'harness/no-api-in-components: error' --rule 'harness/no-store-in-pages: error'
      - name: 循环依赖检测
        run: npx madge --circular src

  # 阶段三：测试
  test:
    needs: arch-check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - name: 运行测试
        run: npx vitest run
      - name: 覆盖率检查
        run: npx vitest run --coverage --coverage.thresholds.lines=80

  # 阶段四：构建验证
  build:
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - name: 构建
        run: npm run build
```

## 本节小结

```
CI 的角色：
  Harness 的最后一道防线
  服务端强制执行，不依赖 Agent 本地环境
  违规代码物理上无法合并

流水线结构（快速失败原则）：
  阶段一：快速检查（Prettier 格式、TypeScript 类型）
  阶段二：架构约束检查（ESLint 层级规则、循环依赖）
  阶段三：测试（Vitest 单元测试、覆盖率）
  阶段四：构建验证（npm run build）
  阶段五：质量检查（可选，Warning 级别）

检查项分级：
  阻断：架构违规、类型错误、测试失败、构建失败
  警告：质量问题、轻微规范违反
  记录：性能报告、依赖提示

PR 审查策略：
  小 PR + CI 全绿 → 自动合并
  架构变更 / 鉴权代码 / 大改动 → 人工审查
  人工审查目标：1 分钟内完成
```
