# 2.2　AGENTS.md 的设计哲学

> 本节目标：掌握 AGENTS.md 的核心设计原则，能够为自己的前端项目写出一份合格的 AGENTS.md。

---

## 一、AGENTS.md 是什么

AGENTS.md 是放在项目根目录的一个 Markdown 文件，是 Agent 开始工作前必读的"项目说明书"。

当 Agent 接到任务时，它会首先读取 AGENTS.md，从中获取：
- 这个项目是什么
- 有哪些关键规则必须遵守
- 需要深入了解某个方面时，去哪里找详细文档

```
Agent 开始任务
    ↓
读取 AGENTS.md（全局视角，约100行）
    ↓
按需读取 docs/ 中的相关文档
    ↓
执行任务
```

---

## 二、核心原则：AGENTS.md 是目录，不是百科全书

这是 AGENTS.md 设计中最重要的一条原则。

### 错误的理解

> "AGENTS.md 应该包含 Agent 需要知道的一切。"

这个想法很直觉，但会导致 2.1 节讲过的"巨型 AGENTS.md"失败。

### 正确的理解

> "AGENTS.md 是一张地图，告诉 Agent 去哪里找需要的信息。"

```
AGENTS.md 的职责边界：

✅ 应该包含：
  - 项目一句话描述
  - 最关键的 5-10 条规则（直接写，不用跳转）
  - 架构概览（一段话 + 指向详细文档的链接）
  - 常用命令（运行、测试、构建、Lint）
  - 深层文档的索引

❌ 不应该包含：
  - 完整的架构文档（放 docs/architecture.md）
  - 详细的编码规范（放 docs/conventions.md）
  - 组件 API 文档（放 docs/components.md）
  - 历史决策记录（放 docs/decisions/）
```

**目标长度：约 100 行。** 超过 150 行就要考虑把内容移到 docs/ 里。

---

## 三、Greg Brockman 的建议：每次 Agent 犯错就更新 AGENTS.md

OpenAI 联合创始人 Greg Brockman 分享了一个非常实用的工作习惯：

> "每当 Agent 犯了一个错误，问自己：我能把防止这个错误的规则写进 AGENTS.md 吗？"

这个习惯背后的逻辑：

```
Agent 犯错
    ↓
分析根因：Agent 为什么会犯这个错？
    ↓
两种可能：
  A. Agent 不知道某条规则  →  把规则写进 AGENTS.md
  B. Agent 知道规则但没遵守 →  考虑用 ESLint 机械强制（第三章）
    ↓
下次同类错误被预防
```

这就是 Harness Engineering 的核心工作方式：**把人的经验编码进系统，而不是每次都重新解释。**

## 四、AGENTS.md 的典型结构

下面是一个针对 AI 内嵌 H5 前端项目的 AGENTS.md 示例：

```markdown
# ai-h5-chat（AI 对话 H5 应用）

## 项目概述
这是一个内嵌在 App WebView 中的 AI 对话 H5 页面，
基于 React + TypeScript，使用 Zustand 管理状态，Vite 构建。

## 架构概览
分层架构：Types → Config → API → Store → Hooks → Components → Pages
依赖方向：只能向下，不能向上。
详细架构：docs/architecture.md

## 层级规则（必须遵守）
- 禁止在 Components 层直接调用 fetch 或 AI SDK
- 禁止在 Pages 层直接操作 Store（必须通过 Hooks）
- 禁止跨层直接调用（必须通过相邻层）
- AI SDK 调用必须封装在 src/api/aiApi.ts 中
详细规则：docs/architecture.md#层级约束

## 编码规范（关键条目）
- 组件文件用 PascalCase，Hook 文件用 camelCase（以 use 开头）
- 所有网络请求必须经过 src/api/ 层，禁止在组件内直接 fetch
- 错误处理统一使用 src/utils/errorHandler.ts
完整规范：docs/conventions.md

## 常用命令
```bash
# 启动开发服务器
npm run dev

# 运行测试
npm test -- --run

# 构建
npm run build

# 运行 ESLint
npm run lint
```

## 文档
docs/README.md（所有文档的入口）
```

## 五、AGENTS.md 的层级结构

一个大型前端项目可能有多个 AGENTS.md，分布在不同目录：

```
项目根目录/
├── AGENTS.md              ← 全局说明书（整个项目的规则）
├── src/
│   ├── AGENTS.md          ← 源码目录的补充规则
│   ├── api/
│   │   └── AGENTS.md      ← API 层的专属规则
│   ├── components/
│   │   └── AGENTS.md      ← 组件层的专属规则
│   └── pages/
│       └── AGENTS.md      ← 页面层的专属规则
└── docs/
    └── AGENTS.md          ← 文档目录的规则（可选）
```

Agent 读取规则：**从当前工作目录向上查找，逐层叠加。**

```
Agent 在 src/components/ 工作时，读取顺序：
  1. src/components/AGENTS.md（最具体）
  2. src/AGENTS.md（中间层）
  3. AGENTS.md（全局，最通用）

规则叠加：下层规则可以补充或覆盖上层规则
```

## 六、实战：为你的第一个前端项目写 AGENTS.md

### 步骤一：明确项目是什么

用一句话回答：这个项目是什么？做什么用的？

示例：
> "这是一个用 React + TypeScript 构建的 AI 对话 H5 页面，内嵌在 App WebView 中，支持流式消息渲染和多轮对话。"

### 步骤二：列出最关键的规则

问自己：如果 Agent 只能记住 5 条规则，哪 5 条最重要？

常见的关键规则类型：
- 架构层级规则（不能反向依赖）
- 禁止使用的模式（比如"禁止在 Component 层直接调用 AI SDK"）
- 必须使用的工具或库（比如"状态管理必须用 Zustand，不用 Redux"）
- 测试要求（比如"每个新 Hook 必须有对应的单元测试"）

### 步骤三：整理常用命令

把运行、测试、构建、Lint 的命令都列出来。Agent 需要这些来验证自己的工作。

### 步骤四：建立 docs/ 索引

在 AGENTS.md 里只写一行：`docs/README.md（所有文档的入口）`

在 docs/README.md 里列出所有文档的分类和链接，Agent 查文档时先读这里。

## 七、AGENTS.md 的维护节奏

AGENTS.md 不是写一次就完事的，它需要随项目演进持续更新。

```
触发更新的时机：

1. Agent 犯了新类型的错误
   → 把防止这个错误的规则加进去

2. 项目架构发生变化
   → 更新架构概览和层级规则

3. 引入了新的工具或库
   → 更新编码规范和常用命令

4. 发现某条规则 Agent 总是忽略
   → 考虑把它移到更靠前的位置，或者改用 ESLint 强制

5. AGENTS.md 超过 150 行
   → 把部分内容移到 docs/ 里，AGENTS.md 只保留链接
```

## 本节小结

```
AGENTS.md 的本质：
  不是百科全书，是地图
  不是越详细越好，是越精准越好

核心原则：
  约 100 行，超过就移到 docs/
  最关键的规则直接写
  详细内容通过链接按需访问

维护习惯（Greg Brockman）：
  Agent 每犯一次错 → 更新一次 AGENTS.md
  把人的经验编码进系统

层级结构：
  根目录 AGENTS.md（全局）
  子目录 AGENTS.md（局部补充）
  从具体到通用，逐层叠加
```
