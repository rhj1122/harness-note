# 2.4　文档自维护机制（Agent for Agent）

> 本节目标：理解为什么文档会腐烂，以及如何用 Agent 来维护文档，让文档保持长期有效。

---

## 一、文档腐烂问题

文档有一个天然的敌人：**时间**。

代码在变，但文档往往跟不上。

```
第 1 周：写了 docs/architecture.md，描述了五层架构
第 3 周：重构，把 API 层拆成了 API 层 + AIProvider 层，但没更新文档
第 6 周：Agent 读到文档，按五层架构写代码
         → 实际是六层，代码结构错误
         → 工程师困惑：Agent 为什么这么写？
         → 查了半天才发现：文档过时了
```

这就是**文档腐烂**（Documentation Rot）。

在 AI 时代，文档腐烂的危害被放大了：

```
人工编码时代：
  工程师读到过时文档 → 凭经验判断 → 可能发现文档有误

AI Agent 时代：
  Agent 读到过时文档 → 严格按文档执行 → 批量生成错误代码
  → 错误被 ESLint/CI 放大，大量 PR 失败
  → 工程师花大量时间排查根因
```

**Agent 不会质疑文档，它只会执行文档。** 所以文档的准确性比以往任何时候都重要。

---

## 二、解法：用 Agent 维护文档

既然 Agent 是问题的放大器，它也可以是解决方案的执行者。

核心思路：

```
不要依赖人来记得更新文档
    ↓
设计一个后台 Agent，定期扫描文档是否过时
    ↓
发现过时文档 → 自动开 PR 更新
    ↓
人只需要审批 PR（而不是主动去维护文档）
```

这就是 **Agent for Agent**：用一个 Agent 来维护另一个 Agent 的工作环境。

---

## 三、文档自维护的三个层次

### 层次一：代码变更触发文档检查

每次代码 PR 合并时，自动触发一个检查：

```
代码 PR 合并
    ↓
触发文档检查 Agent
    ↓
Agent 对比：
  - 这次 PR 改了哪些文件？
  - 这些文件对应的文档是什么？
  - 文档内容是否还准确？
    ↓
如果发现不一致 → 开一个文档更新 PR
```

示例：
```
PR #234 修改了 src/api/aiApi.ts（AI 接口封装）
    ↓
文档检查 Agent 读取 docs/modules/ai-chat.md
    ↓
发现文档里描述的函数签名与代码不符
    ↓
自动开 PR #235：更新 docs/modules/ai-chat.md 中的函数签名
```

### 层次二：定期全量扫描

每周运行一次全量文档扫描：

```
扫描 Agent 遍历所有 docs/ 文件
    ↓
对每份文档，检查：
  - 文档中引用的文件路径是否还存在？
  - 文档中描述的组件/Hook/函数是否还存在？
  - 文档中的命令是否还能运行？
    ↓
生成"文档健康报告"
    ↓
对每个问题开一个修复 PR
```

### 层次三：AGENTS.md 自检

AGENTS.md 是最关键的文档，需要特别保护：

```
每次有新的 ESLint 规则加入
    ↓
检查 AGENTS.md 是否需要更新对应条目
    ↓
如果需要 → 自动在 AGENTS.md 中添加提示行
```

## 四、文档 → 反馈回路 → 文档 的持续闭环

把上面三个层次组合起来，就形成了一个持续运转的闭环：

```
┌─────────────────────────────────────────────────────┐
│              文档自维护闭环                            │
│                                                     │
│  工程师写/更新文档                                    │
│       ↓                                             │
│  Agent 读文档，执行任务                               │
│       ↓                                             │
│  代码变更（Agent 生成的 PR 合并）                      │
│       ↓                                             │
│  文档检查 Agent 触发                                  │
│       ↓                                             │
│  发现文档与代码不一致                                  │
│       ↓                                             │
│  自动开文档更新 PR                                    │
│       ↓                                             │
│  工程师审批（通常 30 秒）                              │
│       ↓                                             │
│  文档更新合并                                         │
│       ↓                                             │
│  下一个 Agent 读到的是最新文档  ←─────────────────────┘
└─────────────────────────────────────────────────────┘
```

**关键设计思想：** 文档的维护责任从"人记得去更新"变成了"系统自动检测并提醒"。

---

## 五、实现文档自维护的技术方案

### 方案一：CI 钩子（最简单，推荐入门）

在 CI 流水线里加一个步骤：

```yaml
# .github/workflows/doc-check.yml 示例结构
# 每次 PR 合并到 main 时触发

steps:
  - name: 检查文档一致性
    # 运行文档检查脚本
    # 脚本逻辑：
    #   1. 获取本次 PR 修改的文件列表
    #   2. 找到对应的文档文件
    #   3. 用 Agent 对比代码和文档
    #   4. 如果不一致，创建 Issue 或 PR
```

### 方案二：定时任务（适合全量扫描）

```yaml
# 每周一凌晨 2 点运行全量文档扫描
# 扫描内容：
#   - 所有 docs/ 文件中的代码引用
#   - 所有 AGENTS.md 中的路径引用
#   - 所有文档中的命令示例（npm run xxx 是否还有效）
# 输出：文档健康报告 + 修复 PR
```

### 方案三：Agent 直接维护（最自动化）

给 Agent 一个持续运行的任务：

```
任务描述：
  你是文档维护 Agent。
  每当有代码 PR 合并，检查相关文档是否需要更新。
  如果需要，直接开 PR 更新文档。
  PR 标题格式：[docs] 更新 xxx 文档以匹配最新代码
```

## 六、文档自维护的边界

不是所有文档都适合自动维护，需要区分：

```
适合自动维护的文档：
  ✅ 组件/Hook 的函数签名（参数、返回值）
  ✅ 文件路径引用（某文件移动了，文档里的路径要更新）
  ✅ 命令示例（npm 脚本变了，文档里的示例要更新）
  ✅ 架构图中的模块名称

需要人工维护的文档：
  ❌ 设计决策（为什么这样设计，需要人来判断）
  ❌ 黄金原则（最佳实践的定义，需要人来决策）
  ❌ AGENTS.md 的核心规则（规则的增删需要人审慎决定）
```

自动维护处理"事实性"内容，人工维护处理"判断性"内容。

## 本节小结

```
文档腐烂的危害在 AI 时代被放大：
  Agent 不会质疑文档，只会严格执行
  过时文档 → 批量错误代码

解法：Agent for Agent
  用后台 Agent 维护文档
  代码变更 → 触发文档检查 → 自动开 PR → 人工审批

三个层次：
  1. 代码变更触发（精准，实时）
  2. 定期全量扫描（全面，周期性）
  3. AGENTS.md 自检（保护最关键的文档）

持续闭环：
  文档 → Agent 执行 → 代码变更 → 文档检查 → 文档更新 → 循环

边界：
  自动维护：事实性内容（路径、签名、命令）
  人工维护：判断性内容（决策、原则、规则）
```
