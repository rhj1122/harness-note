# 2.5　Skills（技能文档）

> 本节目标：理解 Skills 文档是什么、和工具定义有什么区别，以及如何编写有效的 Skills 文档。

---

## 一、一个真实的困境

假设你的前端项目里有这些工具可以用：

```
- read_file        读取文件内容
- write_file       写入文件
- run_command      执行命令
- search_codebase  搜索代码库
- create_pr        创建 Pull Request
- run_tests        运行测试（npm test -- --run）
- check_lint       运行 ESLint
```

工具定义告诉 Agent：**这些工具存在，以及怎么调用它们**（参数、返回值）。

但工具定义**不告诉** Agent：
- 什么时候该用哪个工具？
- 这些工具应该按什么顺序使用？
- 某个场景下，用工具 A 还是工具 B？

结果：Agent 凭自己的"直觉"决定，经常用错。

**Skills 文档就是解决这个问题的。**

---

## 二、Skills vs 工具定义：Why vs How

这是理解 Skills 的关键区分：

```
工具定义（Tool Definition）
  回答：这个工具是什么？怎么调用？
  内容：函数签名、参数说明、返回值格式
  类比：说明书上的"规格参数"

Skills 文档
  回答：什么时候用这个工具？为什么用它？
  内容：使用场景、决策逻辑、使用顺序
  类比：说明书上的"使用指南"
```

举个例子：

```
工具定义（How）：
  search_codebase(query: str, file_pattern: str) -> List[Match]
  在代码库中搜索匹配 query 的内容，可用 file_pattern 过滤文件类型

Skills 文档（Why）：
  当你需要了解某个 Hook 在哪里被调用时，使用 search_codebase。
  当你需要找到某个组件的定义时，先用 search_codebase 找到文件，
  再用 read_file 读取具体内容。
  不要用 search_codebase 来读取文件内容——那是 read_file 的职责。
```

---

## 三、Skills 文档的典型结构

一份 Skills 文档通常包含以下部分：

```markdown
# Skill：[技能名称]

## 适用场景
[描述什么情况下应该使用这个技能/工具]

## 不适用场景
[描述什么情况下不应该使用，以及应该用什么替代]

## 使用步骤
[按顺序描述如何使用]

## 示例
[一个具体的使用示例]

## 常见错误
[Agent 容易犯的错误，以及如何避免]
```

## 四、实际案例：代码搜索 Skill

```markdown
# Skill：在代码库中查找信息

## 适用场景
- 需要找到某个 Hook、组件、工具函数的定义位置
- 需要了解某个模块被哪些地方引用
- 需要确认某个功能是否已经实现

## 不适用场景
- 不要用搜索来"读取"文件内容（用 read_file）
- 不要用搜索来验证代码是否正确（用 run_tests）

## 使用步骤
1. 先用 search_codebase 找到相关文件
2. 用 read_file 读取具体文件内容
3. 如果需要了解调用关系，再次用 search_codebase 搜索函数名

## 示例
任务：找到 useAIStream Hook 的定义

步骤：
  search_codebase("useAIStream")
  → 返回：src/hooks/useAIStream.ts, 第 12 行
  read_file("src/hooks/useAIStream.ts")
  → 读取完整内容

## 常见错误
❌ 错误：用 search_codebase 搜索大段代码内容
   原因：搜索是用来定位的，不是用来读取的
   正确：先搜索定位，再用 read_file 读取
```

## 五、前端项目常用的 Skills 文档

以下是 AI 内嵌 H5 项目中值得编写的 Skills 文档：

### Skill：新增一个 React 组件

```markdown
# Skill：新增一个 React 组件

## 适用场景
- 需要创建一个新的 UI 组件

## 使用步骤
1. 确认组件属于哪一层（Components 还是 Pages）
2. 在对应目录创建 ComponentName.tsx
3. 如果有业务逻辑，抽取到 useXxx Hook，不要写在组件里
4. 运行 check_lint 确认无 ESLint 错误
5. 运行 run_tests 确认测试通过

## 常见错误
❌ 在组件内直接调用 fetch 或 AI SDK
   正确：通过 Hooks 层调用，Hooks 层再调用 API 层
```

### Skill：新增一个 API 接口

```markdown
# Skill：新增一个 API 接口

## 适用场景
- 需要调用新的后端接口或 AI 接口

## 使用步骤
1. 在 src/api/ 目录下找到对应的 API 文件（或新建）
2. 添加接口函数，使用项目统一的 request 封装
3. 在 src/types/ 中定义请求/响应类型
4. 在对应的 Hook 中调用该 API 函数

## 常见错误
❌ 在 Hook 或 Component 中直接写 fetch/axios 调用
   正确：所有网络请求必须封装在 src/api/ 层
```

## 六、Skills 文档放在哪里

Skills 文档放在 `docs/skills/` 下，和其他文档统一管理：

```
项目根目录/
├── AGENTS.md
└── docs/
    ├── README.md          ← 文档总索引（包含 skills 的入口）
    ├── architecture.md
    ├── conventions.md
    └── skills/            ← Skills 文档放这里
        ├── README.md      ← Skills 索引（列出所有 Skill）
        ├── search-code.md ← 代码搜索技能
        ├── add-component.md ← 新增组件技能
        ├── add-api.md     ← 新增 API 接口技能
        ├── write-tests.md ← 编写测试技能
        ├── create-pr.md   ← 创建 PR 技能
        └── debug-issue.md ← 调试问题技能
```

在 `docs/README.md` 里加一行索引：

```markdown
## 工具使用指南
- [Skills 索引](skills/README.md) — 何时用哪个工具
```

AGENTS.md 不需要单独列 skills，只需指向 `docs/README.md` 即可。

## 七、哪些场景需要写 Skills 文档

不是所有工具都需要 Skills 文档，优先为这些场景编写：

```
高优先级（必须写）：
  ✅ Agent 经常用错的工具
  ✅ 有多个相似工具需要区分使用场景的
  ✅ 使用顺序很重要的工具组合
  ✅ 有明确"禁止用法"需要声明的工具

低优先级（可以不写）：
  ⬜ 功能单一、用法显而易见的工具
  ⬜ 只在极少数场景下使用的工具
```

**实用原则：** 当你发现 Agent 第二次用错同一个工具时，就该写 Skills 文档了。

---

## 八、Skills 文档的维护

Skills 文档同样需要维护，触发更新的时机：

```
1. Agent 用错了工具
   → 在对应 Skill 的"常见错误"部分加一条

2. 引入了新工具
   → 写一份新的 Skill 文档

3. 工具的行为发生了变化
   → 更新对应的 Skill 文档

4. 发现两个 Skill 文档有矛盾
   → 合并或明确区分
```

## 本节小结

```
Skills 文档解决的问题：
  工具定义告诉 Agent "怎么用"（How）
  Skills 文档告诉 Agent "什么时候用、为什么用"（Why）

核心区分：
  工具定义 = 规格参数
  Skills   = 使用指南

典型结构：
  适用场景 / 不适用场景 / 使用步骤 / 示例 / 常见错误

存放位置：
  docs/skills/ 目录，在 docs/README.md 中索引
  AGENTS.md 只需指向 docs/README.md

何时写：
  Agent 第二次用错同一个工具时，就该写了
```
