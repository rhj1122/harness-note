# 5.5　反馈回路设计

> 本节目标：把第五章所有工具串联起来，理解短、中、长三条反馈回路的设计，以及如何最小化反馈延迟。

---

## 一、什么是反馈回路

反馈回路是指：**Agent 做了一件事，多久能知道结果是否正确。**

```
反馈回路越短：
  Agent 越快知道自己做错了
  → 越快修复
  → 浪费的时间越少

反馈回路越长：
  Agent 做了很多工作后才发现方向错了
  → 需要大量返工
  → 效率极低
```

**Harness Engineering 的核心目标之一：让每一类问题都有一条尽可能短的反馈回路。**

## 二、三条反馈回路

### 短回路：Agent 本地验证（秒到分钟级）

```
触发：Agent 修改代码后
流程：
  Agent 修改代码
      ↓
  运行静态检查（npm run check）← 秒级
      ↓ 发现问题
  Agent 读取错误信息，立即修复
      ↓
  运行测试（npx vitest run）← 分钟级
      ↓ 发现问题
  Agent 读取失败信息，立即修复
      ↓
  所有检查通过 → 提交 PR

特点：
  - 完全自动，不需要人介入
  - 反馈延迟：秒到分钟
  - 覆盖：类型错误、架构违规、逻辑 bug
```

### 中回路：CI 验证（分钟到十分钟级）

```
触发：Agent 提交 PR 后
流程：
  PR 提交
      ↓
  CI 自动运行（GitHub Actions）
      ↓
  阶段一：格式 + 类型检查（30秒）
  阶段二：架构 ESLint 检查（1分钟）
  阶段三：测试（2-5分钟）
  阶段四：构建验证（1-2分钟）
      ↓
  全部通过 → PR 可合并
  某阶段失败 → Agent 读取 CI 日志，修复后重新推送

特点：
  - 服务端强制执行，不依赖 Agent 本地环境
  - 反馈延迟：5-10 分钟
  - 覆盖：本地漏掉的问题，环境差异导致的问题
```

### 长回路：质量扫描 + Harness Review（天到周级）

```
触发：定时任务（每天/每周）
流程：
  定时触发质量扫描
      ↓
  扫描代码库：重复率、死代码、覆盖率等
      ↓
  生成质量报告
      ↓
  发现质量下降 → 触发清理 Agent → 开清理 PR
  发现 Harness 老化 → 触发 Harness Review → 更新 Harness

特点：
  - 处理积累性问题（单次 PR 看不出来的趋势）
  - 反馈延迟：天到周
  - 覆盖：代码库整体健康状态、Harness 有效性
```

## 四、反馈延迟的影响与最小化策略

### 延迟的代价

```
反馈延迟 10 分钟 vs 10 秒的区别：

10 秒反馈：
  Agent 修改 → 10秒后发现错误 → 立即修复
  Agent 的"工作记忆"还在，修复很快

10 分钟反馈（等 CI）：
  Agent 修改 → 提交 → 等 10 分钟 → 发现错误
  Agent 可能已经开始了其他任务
  需要重新加载上下文，理解之前的修改
  修复效率下降
```

### 最小化策略

**策略一：把检查前移到本地**

```
不好的做法：
  Agent 修改代码 → 直接提交 → 等 CI 发现问题

好的做法：
  Agent 修改代码 → 本地运行 check + test → 确认通过 → 再提交
  → CI 只是最后的保险，不是主要的反馈来源
```

**策略二：快速失败原则**

```
CI 流水线设计：
  最快的检查放最前面（格式检查 30 秒）
  发现问题立即停止，不继续跑后面的阶段
  → Agent 最快 30 秒就能收到反馈，不需要等完整流水线
```

**策略三：精准运行测试**

```
不好的做法：
  每次修改都运行全部测试（可能需要 5 分钟）

好的做法：
  修改了 useAIChat.ts → 只运行 useAIChat.test.ts（10 秒）
  修改了多个文件 → 运行相关测试文件（30 秒）
  提交前 → 运行全部测试（确保没有回归）
```

**策略四：ESLint 错误信息即修复指令**

```
普通 ESLint 错误：
  "Import not allowed"
  → Agent 不知道怎么修，需要查文档，延迟增加

Harness ESLint 错误：
  "[ARCH-001] Components 层不能直接导入 API 层。
   修复：将数据获取逻辑移到 src/hooks/ 中的 useXxx Hook。"
  → Agent 立刻知道怎么修，延迟最小化
```

## 五、反馈回路的健康指标

如何判断反馈回路是否运转良好？

```
短回路健康指标：
  ✅ Agent 提交的 PR，CI 首次通过率 > 80%
     （说明 Agent 在本地已经充分验证）
  ⚠️ 首次通过率 < 50%
     （说明 Agent 没有充分利用本地验证工具）

中回路健康指标：
  ✅ CI 平均运行时间 < 10 分钟
  ✅ CI 失败后，Agent 修复并重新通过的时间 < 30 分钟
  ⚠️ CI 运行时间 > 15 分钟（需要优化流水线）

长回路健康指标：
  ✅ 质量指标趋势稳定或改善
  ✅ Harness Review 每 2 周执行一次
  ⚠️ 质量指标持续恶化（说明清理流程不够有效）
```
