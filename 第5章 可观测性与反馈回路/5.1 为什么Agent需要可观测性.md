# 5.1　为什么 Agent 需要可观测性

> 本节目标：理解可观测性在 Harness Engineering 中的角色——为什么 Agent 必须能"看到"运行结果，才能真正自主工作。

---

## 一、Agent 的感知困境

人类工程师写完代码后，会做什么？

```
人类工程师的验证流程：
  写完代码
    → 在浏览器里打开页面，看看效果
    → 打开控制台，看有没有报错
    → 点击按钮，测试交互是否正常
    → 看网络请求，确认 API 调用正确
    → 运行测试，确认没有破坏已有功能
```

这一切都依赖**感知**——工程师能看到、能点击、能读取日志。

Agent 没有眼睛，没有手，它只能通过工具来感知环境：

```
Agent 的感知方式：
  运行测试 → 读取测试结果
  执行构建 → 读取构建输出
  读取日志 → 分析错误信息
  截图页面 → 分析 UI 状态

如果没有这些工具：
  Agent 写完代码 → 不知道代码是否正确
  → 只能"假设"代码能跑
  → 提交 PR → 人工发现问题
  → 反馈给 Agent → Agent 修复
  → 整个循环非常慢
```

**可观测性 = 给 Agent 装上"眼睛"，让它能自己验证工作结果。**

---

## 二、没有可观测性时的问题

来看一个没有可观测性的典型场景：

```
任务：修复 AI 对话页面的流式渲染 bug

没有可观测性的 Agent：
  1. 读取代码，分析 bug 原因
  2. 修改 useAIStream Hook 的实现
  3. 提交 PR
  4. 等待人工测试

人工测试发现：
  - 流式渲染确实修了
  - 但修复引入了新 bug：消息列表滚动失效了
  - 还有：TypeScript 类型报错（Agent 没有运行 tsc）

Agent 重新修复：
  5. 修复滚动 bug
  6. 修复类型错误
  7. 再次提交 PR

人工测试再次发现：
  - 滚动修了，类型修了
  - 但现在有一个测试失败了

整个过程：3 轮 PR，耗时 2 天
```

```
有可观测性的 Agent：
  1. 读取代码，分析 bug 原因
  2. 修改 useAIStream Hook 的实现
  3. 运行测试 → 发现 2 个测试失败
  4. 修复测试失败
  5. 运行 tsc → 发现类型错误
  6. 修复类型错误
  7. 运行构建 → 成功
  8. 截图页面 → 确认 UI 正常
  9. 提交 PR（CI 全绿）

整个过程：1 轮 PR，耗时 30 分钟
```

**可观测性把"人工发现问题"变成了"Agent 自己发现并修复问题"。**

## 三、前端项目的可观测性层次

前端项目的可观测性分为四个层次：

```
层次一：静态检查（最快，秒级）
  TypeScript 类型检查（tsc --noEmit）
  ESLint 检查（npm run lint）
  → Agent 每次修改后立即运行，发现语法和类型问题

层次二：单元测试（快，秒到分钟级）
  Vitest 单元测试
  React Testing Library 组件测试
  → Agent 修改逻辑后运行，发现功能回归

层次三：构建验证（中等，分钟级）
  npm run build
  → Agent 确认代码能正确打包，没有构建错误

层次四：运行时感知（慢，需要启动服务）
  浏览器截图（Chrome DevTools Protocol）
  页面交互模拟
  → Agent 验证 UI 渲染和交互是否符合预期
```

不是每次修改都需要跑所有层次，Agent 应该根据修改范围选择合适的验证层次：

```
修改了类型定义 → 运行层次一（tsc）
修改了 Hook 逻辑 → 运行层次一 + 层次二（tsc + 测试）
修改了组件 → 运行层次一 + 层次二 + 层次三（tsc + 测试 + 构建）
修改了 UI 样式 → 运行层次四（截图对比）
```

## 四、可观测性是 Agent 自我验证的基础

可观测性不只是"让 Agent 知道代码有没有问题"，它是整个 Harness 反馈回路的基础：

```
完整的 Agent 工作闭环：

  接收任务
      ↓
  读取 Harness（AGENTS.md、docs/）
      ↓
  生成代码
      ↓
  ┌── 自我验证（可观测性）──────────────────┐
  │  运行静态检查 → 发现问题 → 修复         │
  │  运行测试 → 发现问题 → 修复             │
  │  运行构建 → 发现问题 → 修复             │
  │  截图验证 → 发现问题 → 修复             │
  └─────────────────────────────────────────┘
      ↓（所有检查通过）
  提交 PR
      ↓
  CI 验证（服务端再次确认）
      ↓
  人工轻量审查
      ↓
  合并
```

**没有可观测性，Agent 就无法完成"自我验证"这一步，整个闭环就断了。**

## 五、可观测性与 Harness 其他部分的关系

```
上下文工程（第二章）：
  告诉 Agent "应该做什么"
  → 输入端：任务开始前

架构约束（第三章）：
  阻止 Agent "做错误的事"
  → 过程中：代码生成时

可观测性（第五章）：
  让 Agent 知道"做得对不对"
  → 输出端：代码生成后

三者缺一不可：
  只有输入（上下文）：Agent 知道规则，但不知道结果
  只有过程约束（ESLint）：Agent 不会犯架构错误，但可能有逻辑 bug
  只有输出验证（可观测性）：Agent 能发现问题，但不知道规则是什么
```

## 本节小结

```
Agent 的感知困境：
  没有眼睛和手，只能通过工具感知环境
  没有可观测性 = 写完代码不知道对不对

没有可观测性的代价：
  问题只能靠人工发现
  多轮 PR 来回修复，效率极低

前端可观测性四个层次：
  静态检查（tsc + ESLint）→ 秒级
  单元测试（Vitest）→ 分钟级
  构建验证（npm run build）→ 分钟级
  运行时感知（截图、交互）→ 需要启动服务

可观测性的核心价值：
  让 Agent 能自我验证，完成工作闭环
  把"人工发现问题"变成"Agent 自己发现并修复"
```
