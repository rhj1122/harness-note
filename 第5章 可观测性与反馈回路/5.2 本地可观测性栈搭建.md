# 5.2　本地可观测性栈搭建

> 本节目标：掌握前端项目本地可观测性栈的配置方法，让 Agent 能高效运行各层次的验证工具。

---

## 一、本地可观测性栈的目标

本地可观测性栈需要满足三个条件：

```
1. Agent 可以直接调用
   → 工具通过命令行运行，输出结果可被 Agent 读取

2. 反馈足够快
   → 静态检查秒级完成，测试分钟级完成
   → 不能让 Agent 等太久

3. 错误信息对 Agent 友好
   → 输出清晰的文件路径、行号、错误原因
   → Agent 读到后能直接定位并修复
```

---

## 二、静态检查层的配置

### TypeScript 类型检查

```json
// tsconfig.json（推荐的严格配置）
{
  "compilerOptions": {
    "strict": true,           // 开启所有严格检查
    "noUnusedLocals": true,   // 禁止未使用的局部变量
    "noUnusedParameters": true, // 禁止未使用的参数
    "noImplicitReturns": true,  // 函数必须有明确返回值
    "noFallthroughCasesInSwitch": true
  }
}
```

Agent 调用方式：
```bash
npx tsc --noEmit
# --noEmit：只检查类型，不生成文件
# 输出示例：
# src/hooks/useAIChat.ts:23:5 - error TS2345: Argument of type 'string' is not assignable to parameter of type 'Message'.
```

### ESLint 检查

```json
// package.json
{
  "scripts": {
    "lint": "eslint src --ext .ts,.tsx",
    "lint:fix": "eslint src --ext .ts,.tsx --fix"
  }
}
```

Agent 调用方式：
```bash
npm run lint
# 输出示例：
# src/components/ChatMessage.tsx
#   8:1  error  [ARCH-001] Components 层不能直接导入 API 层...
#   ✖ 1 problem (1 error, 0 warnings)
```

### 把两者合并成一个命令

```json
// package.json
{
  "scripts": {
    "check": "tsc --noEmit && eslint src --ext .ts,.tsx"
  }
}
```

Agent 只需运行 `npm run check`，一次性完成所有静态检查。

## 三、测试层的配置

### Vitest 配置

```typescript
// vitest.config.ts
import { defineConfig } from 'vitest/config'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  test: {
    environment: 'jsdom',      // 模拟浏览器环境
    globals: true,             // 全局 describe/it/expect
    setupFiles: ['./src/test/setup.ts'],
    coverage: {
      provider: 'v8',
      reporter: ['text', 'json-summary'],  // text 供 Agent 读取，json 供脚本处理
      thresholds: {
        lines: 80,
        functions: 80,
      }
    }
  }
})
```

```typescript
// src/test/setup.ts
import '@testing-library/jest-dom'
// 其他全局 mock 配置
```

Agent 调用方式：
```bash
# 单次运行（不进入 watch 模式）
npx vitest run

# 运行特定文件
npx vitest run src/hooks/useAIChat.test.ts

# 运行并生成覆盖率报告
npx vitest run --coverage
```

### 让测试输出对 Agent 友好

```typescript
// 好的测试描述：让 Agent 从失败信息就能理解问题
describe('useAIChat', () => {
  it('发送消息后，messages 数组应该包含用户消息', async () => {
    const { result } = renderHook(() => useAIChat())
    await act(async () => {
      await result.current.sendMessage('你好')
    })
    expect(result.current.messages).toContainEqual(
      expect.objectContaining({ role: 'user', content: '你好' })
    )
  })
})

// 失败时的输出：
// ✗ useAIChat > 发送消息后，messages 数组应该包含用户消息
//   AssertionError: expected [] to contain { role: 'user', content: '你好' }
//   → Agent 立刻知道：sendMessage 没有把消息加入 messages 数组
```

## 四、构建验证层的配置

```json
// package.json
{
  "scripts": {
    "build": "vite build",
    "build:check": "vite build 2>&1 | tail -20"
    // 只输出最后 20 行，避免输出太多干扰 Agent
  }
}
```

Agent 调用方式：
```bash
npm run build
# 成功输出：
# ✓ built in 3.42s
# dist/index.html    0.46 kB
# dist/assets/...

# 失败输出：
# error during build:
# src/hooks/useAIChat.ts:45:3: ERROR: No matching export in "src/api/aiApi.ts" for import "streamMessages"
# → Agent 知道：import 了一个不存在的导出，需要修复
```

## 五、Agent 的验证工作流

把以上工具组合成 Agent 的标准验证流程，写进 AGENTS.md 或 skills/ 文档：

```markdown
# Skill：验证代码修改

## 标准验证流程

修改代码后，按以下顺序验证：

### 步骤一：静态检查（必须）
```bash
npm run check
```
如果有错误，先修复所有错误再继续。

### 步骤二：运行相关测试（必须）
```bash
# 只运行修改涉及的文件的测试
npx vitest run src/hooks/useAIChat.test.ts

# 如果不确定影响范围，运行全部测试
npx vitest run
```
如果有测试失败，修复后再继续。

### 步骤三：构建验证（修改了 API 或类型时必须）
```bash
npm run build
```

### 步骤四：提交 PR
所有步骤通过后，才提交 PR。

## 快速验证（小改动时）
```bash
npm run check && npx vitest run
```
```

## 六、Git Worktree 与可观测性的配合

在多 Agent 并行工作时，每个 Agent 在独立的 git worktree 里工作，可观测性工具也在各自的 worktree 里独立运行：

```
主代码库
├── worktree-agent-1/   ← Agent 1 修复流式渲染 bug
│   └── 运行：npm run check && npx vitest run
│
├── worktree-agent-2/   ← Agent 2 实现消息搜索功能
│   └── 运行：npm run check && npx vitest run
│
└── worktree-agent-3/   ← Agent 3 优化组件性能
    └── 运行：npm run check && npx vitest run
```

三个 Agent 同时验证，互不干扰。

配置 worktree 的方式：
```bash
# 为 Agent 创建独立的工作目录
git worktree add ../worktree-agent-1 -b fix/streaming-bug
git worktree add ../worktree-agent-2 -b feat/message-search
git worktree add ../worktree-agent-3 -b perf/component-optimization
```

## 七、可观测性工具的 AGENTS.md 声明

在 AGENTS.md 里声明可用的验证工具，让 Agent 知道如何验证自己的工作：

```markdown
## 验证工具
完成代码修改后，必须按顺序运行以下验证：

1. 静态检查：`npm run check`（tsc + ESLint）
2. 测试：`npx vitest run`（或指定文件）
3. 构建：`npm run build`（修改 API/类型时必须）

详细验证流程：docs/skills/verify-changes.md
```

## 本节小结

```
本地可观测性栈的三个层次：

静态检查层：
  tsc --noEmit（类型检查）
  eslint src（架构和规范检查）
  合并命令：npm run check

测试层：
  npx vitest run（单次运行，不进入 watch 模式）
  npx vitest run --coverage（带覆盖率）
  测试描述要清晰，让失败信息直接指向问题

构建验证层：
  npm run build
  失败时输出清晰的错误位置

Agent 标准验证流程：
  check → vitest run → build（按需）→ 提交 PR

Git Worktree 配合：
  每个 Agent 在独立 worktree 里运行验证，互不干扰

AGENTS.md 声明：
  告诉 Agent 验证工具是什么，按什么顺序运行
```
