# 5.4　Git Worktree 并行工作流

> 本节目标：理解 Git Worktree 机制，掌握配置多 Agent 并行工作环境的方法，实现真正的高吞吐量开发。

---

## 一、为什么需要并行工作

单个 Agent 串行工作的效率瓶颈：

```
串行工作流：

任务 A（修复 bug）
  → Agent 工作 → 提交 PR → 等待 CI → 合并
                                          ↓
                              任务 B（新功能）才能开始
                                          ↓
                              任务 C（优化性能）才能开始

总耗时 = 任务A时间 + 任务B时间 + 任务C时间
```

```
并行工作流：

任务 A（修复 bug）    → Agent 1 工作 → 提交 PR
任务 B（新功能）      → Agent 2 工作 → 提交 PR  ← 同时进行
任务 C（优化性能）    → Agent 3 工作 → 提交 PR

总耗时 ≈ max(任务A时间, 任务B时间, 任务C时间)
```

**并行工作的前提：每个 Agent 必须在独立的环境里工作，互不干扰。**

这就是 Git Worktree 的作用。

---

## 二、Git Worktree 是什么

Git Worktree 允许同一个 Git 仓库同时存在**多个独立的工作目录**，每个目录对应一个分支。

```
没有 Worktree 的情况：
  项目目录/
  └── src/（只有一个工作目录，同一时间只能在一个分支上工作）

使用 Worktree 后：
  项目目录/（主工作目录，main 分支）
  ../worktree-fix-bug/（Agent 1 的工作目录，fix/streaming-bug 分支）
  ../worktree-new-feature/（Agent 2 的工作目录，feat/message-search 分支）
  ../worktree-perf/（Agent 3 的工作目录，perf/render-optimization 分支）
```

关键特性：
- 所有 worktree 共享同一个 `.git` 目录（共享历史、共享对象）
- 每个 worktree 有独立的工作文件和暂存区
- 在一个 worktree 里的修改不会影响其他 worktree

---

## 三、Worktree 的基本操作

```bash
# 创建新的 worktree（同时创建新分支）
git worktree add ../worktree-fix-bug -b fix/streaming-bug

# 创建新的 worktree（基于已有分支）
git worktree add ../worktree-feature feat/message-search

# 查看所有 worktree
git worktree list
# 输出：
# /path/to/project          abc1234 [main]
# /path/to/worktree-fix-bug def5678 [fix/streaming-bug]
# /path/to/worktree-feature ghi9012 [feat/message-search]

# 删除 worktree（任务完成后清理）
git worktree remove ../worktree-fix-bug
```

## 四、多 Agent 并行工作的完整流程

### 步骤一：工程师拆分任务

```
本周任务：
  任务 A：修复 AI 流式渲染重复字符 bug（预计 2 小时）
  任务 B：实现消息搜索功能（预计 4 小时）
  任务 C：优化消息列表渲染性能（预计 3 小时）

这三个任务互相独立，可以并行
```

### 步骤二：为每个任务创建 Worktree

```bash
# 从 main 分支创建三个独立的工作目录
git worktree add ../worktree-fix-streaming -b fix/streaming-bug
git worktree add ../worktree-search -b feat/message-search
git worktree add ../worktree-perf -b perf/list-optimization
```

### 步骤三：为每个 Agent 分配任务

```
Agent 1 的任务描述（在 worktree-fix-streaming 里工作）：
  "修复 src/hooks/useAIStream.ts 中的流式渲染 bug。
   症状：AI 回复的流式文字有时出现重复字符。
   验证方式：运行 npm run check && npx vitest run src/hooks/useAIStream.test.ts"

Agent 2 的任务描述（在 worktree-search 里工作）：
  "在 AI 对话页面实现消息搜索功能。
   需要：在 src/hooks/ 创建 useMessageSearch Hook，在 src/components/ 创建 SearchBar 组件。
   验证方式：运行 npm run check && npx vitest run"

Agent 3 的任务描述（在 worktree-perf 里工作）：
  "优化 src/components/MessageList.tsx 的渲染性能。
   目标：消息超过 100 条时，滚动不卡顿。
   验证方式：运行 npm run check && npx vitest run"
```

### 步骤四：Agent 并行工作

三个 Agent 同时在各自的 worktree 里工作，互不干扰：

```
worktree-fix-streaming/
  Agent 1 修改 useAIStream.ts
  运行测试，发现问题，修复
  提交 PR #301

worktree-search/
  Agent 2 创建 useMessageSearch.ts 和 SearchBar.tsx
  运行测试，全部通过
  提交 PR #302

worktree-perf/
  Agent 3 修改 MessageList.tsx，添加虚拟滚动
  运行测试，全部通过
  提交 PR #303
```

### 步骤五：工程师审查 PR

三个 PR 同时等待审查，工程师可以按优先级逐一处理：

```
PR #301（bug 修复）→ 优先审查，1 分钟，合并
PR #302（新功能）→ 审查，2 分钟，合并
PR #303（性能优化）→ 审查，1 分钟，合并
```

### 步骤六：清理 Worktree

```bash
git worktree remove ../worktree-fix-streaming
git worktree remove ../worktree-search
git worktree remove ../worktree-perf
```

## 五、Worktree 的注意事项

### 注意一：node_modules 需要独立安装

每个 worktree 需要独立的 `node_modules`：

```bash
cd ../worktree-fix-streaming
npm install  # 在新 worktree 里安装依赖
```

或者使用 pnpm，它的硬链接机制可以节省磁盘空间：

```bash
cd ../worktree-fix-streaming
pnpm install  # pnpm 会复用已有的包，只占用少量额外空间
```

### 注意二：端口冲突

多个 worktree 同时启动开发服务器时，需要使用不同端口：

```bash
# worktree-fix-streaming
npm run dev -- --port 5174

# worktree-search
npm run dev -- --port 5175

# worktree-perf
npm run dev -- --port 5176
```

在 Playwright 配置里对应修改 `baseURL`。

### 注意三：任务之间的依赖

如果任务 B 依赖任务 A 的结果，不能并行：

```
可以并行：
  任务 A（修复 bug）和 任务 B（新功能）→ 互相独立

不能并行：
  任务 A（重构 API 层）和 任务 B（基于新 API 实现功能）
  → 任务 B 依赖任务 A，必须等 A 合并后再开始
```

## 六、在 AGENTS.md 中声明 Worktree 规范

```markdown
## 并行工作规范

每个 Agent 在独立的 git worktree 里工作：
  - 分支命名：fix/xxx、feat/xxx、perf/xxx、refactor/xxx
  - 工作目录：../worktree-{分支名}/
  - 完成后提交 PR，不直接合并到 main

创建 worktree：
  git worktree add ../worktree-{任务名} -b {分支名}

清理 worktree（PR 合并后）：
  git worktree remove ../worktree-{任务名}
```

## 本节小结

```
为什么需要并行工作：
  串行工作效率低，任务之间互相等待
  并行工作总耗时 ≈ 最长单个任务的时间

Git Worktree 的作用：
  同一仓库，多个独立工作目录
  每个 Agent 在自己的 worktree 里工作，互不干扰

完整并行流程：
  拆分任务 → 创建 worktree → 分配 Agent → 并行工作
  → 各自提交 PR → 工程师审查 → 合并 → 清理 worktree

注意事项：
  每个 worktree 需要独立安装 node_modules
  多个开发服务器需要使用不同端口
  有依赖关系的任务不能并行

效率提升：
  3 个并行任务 vs 串行：理论上快 3 倍
  实际效果取决于任务拆分的合理性
```
