# 5.3　浏览器自动化（Agent 复现 Bug）

> 本节目标：理解如何让 Agent 通过浏览器自动化工具来验证 UI 层的正确性，以及如何用它复现和验证 bug 修复。

---

## 一、为什么需要浏览器自动化

上一节的可观测性栈（tsc + ESLint + Vitest + build）能覆盖大部分问题，但有一类问题它们覆盖不到：

```
静态检查和单元测试覆盖不到的问题：

UI 渲染问题：
  - 组件在特定数据下渲染出空白
  - 流式文字渲染时出现闪烁
  - 移动端 H5 页面布局错乱

交互问题：
  - 按钮点击后没有反应
  - 滚动到底部时没有触发加载更多
  - 键盘弹起时输入框被遮挡

时序问题：
  - 快速连续发送消息时出现乱序
  - 页面切换时流式请求没有被取消
```

这些问题只有在真实浏览器环境里才能发现和验证。

**浏览器自动化 = 给 Agent 一双"眼睛"，让它能看到页面实际渲染的样子。**

---

## 二、Chrome DevTools Protocol（CDP）

Chrome DevTools Protocol 是 Chrome 浏览器提供的一套调试接口，允许程序控制浏览器：

```
CDP 能做什么：

截图：
  → 获取页面当前的视觉状态
  → Agent 可以"看到"页面长什么样

DOM 快照：
  → 获取页面的 HTML 结构
  → Agent 可以检查元素是否存在、内容是否正确

导航：
  → 打开 URL、前进、后退
  → Agent 可以模拟用户的页面跳转

点击和输入：
  → 模拟用户点击按钮、输入文字
  → Agent 可以测试交互流程

网络拦截：
  → 拦截和修改网络请求
  → Agent 可以模拟 API 返回特定数据
```

## 三、前端项目的浏览器自动化工具选择

### 选项一：Playwright（推荐）

```
优点：
  - 支持 Chrome、Firefox、Safari
  - API 设计现代，TypeScript 原生支持
  - 内置截图、DOM 查询、网络拦截
  - 可以生成测试代码（录制模式）

适合场景：
  - E2E 测试
  - UI 回归测试
  - Agent 验证 UI 修复
```

### 选项二：Puppeteer

```
优点：
  - Google 官方维护，与 Chrome 深度集成
  - CDP 接口更底层，控制更精细

适合场景：
  - 需要精细控制浏览器行为
  - 截图和 PDF 生成
```

本节以 Playwright 为例。

## 五、Agent 用浏览器自动化复现 Bug

### 场景：流式渲染 bug

```
Bug 描述：
  用户发送消息后，AI 回复的流式文字有时会出现重复字符

Agent 的复现流程：
```

```typescript
// e2e/reproduce-streaming-bug.spec.ts
import { test, expect } from '@playwright/test'

test('复现流式渲染重复字符 bug', async ({ page }) => {
  // 1. 打开 AI 对话页面
  await page.goto('/chat')

  // 2. 输入消息并发送
  await page.fill('[data-testid="chat-input"]', '请介绍一下你自己')
  await page.click('[data-testid="send-button"]')

  // 3. 等待流式响应开始
  await page.waitForSelector('[data-testid="ai-message"]')

  // 4. 截图：记录流式渲染过程中的状态
  await page.screenshot({ path: 'screenshots/streaming-in-progress.png' })

  // 5. 等待流式响应完成
  await page.waitForSelector('[data-testid="streaming-indicator"]', { state: 'hidden' })

  // 6. 截图：记录最终状态
  await page.screenshot({ path: 'screenshots/streaming-complete.png' })

  // 7. 获取最终文本内容
  const messageText = await page.textContent('[data-testid="ai-message"]')

  // 8. 检查是否有重复字符（简单检测）
  const hasRepetition = /(.{3,})\1/.test(messageText || '')
  expect(hasRepetition).toBe(false)
})
```

Agent 运行这个测试：
- 如果 bug 存在 → 测试失败，截图记录了问题状态
- 修复后再次运行 → 测试通过，截图确认 UI 正常

## 六、Agent 用截图验证 UI 修复

```typescript
// e2e/verify-ui-fix.spec.ts
import { test, expect } from '@playwright/test'

test('验证消息列表滚动修复', async ({ page }) => {
  await page.goto('/chat')

  // 模拟发送多条消息，让列表超出屏幕
  for (let i = 0; i < 10; i++) {
    await page.fill('[data-testid="chat-input"]', `测试消息 ${i}`)
    await page.click('[data-testid="send-button"]')
    await page.waitForTimeout(100)
  }

  // 截图：检查最新消息是否在视口内（是否自动滚动到底部）
  await page.screenshot({ path: 'screenshots/scroll-after-messages.png' })

  // 验证最后一条消息是否可见
  const lastMessage = page.locator('[data-testid="chat-message"]').last()
  await expect(lastMessage).toBeInViewport()
})
```

## 七、DOM 快照：Agent 检查页面结构

除了截图，Agent 还可以通过 DOM 快照来检查页面结构：

```typescript
// Agent 检查某个元素是否正确渲染
const messageCount = await page.locator('[data-testid="chat-message"]').count()
console.log(`当前消息数量：${messageCount}`)

// Agent 检查某个元素的文本内容
const errorMessage = await page.textContent('[data-testid="error-toast"]')
console.log(`错误提示：${errorMessage}`)

// Agent 检查某个元素是否存在
const isLoadingVisible = await page.isVisible('[data-testid="loading-spinner"]')
console.log(`加载中：${isLoadingVisible}`)
```

## 九、浏览器自动化的使用边界

浏览器自动化功能强大，但也有成本，需要明确使用边界：

```
适合用浏览器自动化的场景：
  ✅ 复现用户报告的 UI bug
  ✅ 验证涉及动画、滚动、布局的修复
  ✅ 验证流式渲染等时序相关功能
  ✅ 回归测试关键用户流程

不适合用浏览器自动化的场景：
  ❌ 验证纯逻辑修改（用单元测试更快）
  ❌ 验证类型错误（用 tsc 更快）
  ❌ 每次小改动都跑（太慢，影响效率）
```

**原则：浏览器自动化是最后一道验证，不是第一道。** 先跑静态检查和单元测试，只有在需要验证 UI 层时才启动浏览器。

## 本节小结

```
为什么需要浏览器自动化：
  静态检查和单元测试覆盖不到 UI 渲染、交互、时序问题
  浏览器自动化让 Agent 能"看到"页面实际状态

工具选择：
  Playwright（推荐）：现代 API，TypeScript 原生支持
  Puppeteer：CDP 底层控制

核心能力：
  截图（看页面状态）
  DOM 快照（检查元素）
  交互模拟（点击、输入）
  网络拦截（模拟 API 数据）

典型用法：
  复现 bug → 写测试脚本 → 运行截图 → 修复 → 再次运行验证

使用边界：
  浏览器自动化是最后一道验证
  先跑 tsc + ESLint + Vitest，再按需启动浏览器
```
