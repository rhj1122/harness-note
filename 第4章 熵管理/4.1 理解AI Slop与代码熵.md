# 4.1　理解 AI Slop 与代码熵

> 本节目标：深入理解 AI Slop 是什么、为什么在前端项目里特别危险，以及熵增的速度为何在 AI 时代被大幅加快。

---

## 一、什么是 AI Slop

AI Slop 是 AI 生成的低质量、难维护代码的统称。

它不是"有 bug 的代码"——AI Slop 往往能跑通，但会让代码库越来越难维护。

前端项目里的 AI Slop 长这样：

### 症状一：重复实现

```typescript
// 某天 Agent 写了这个
// src/hooks/useAuth.ts
export function useAuth() {
  const token = localStorage.getItem('token')
  return { isLoggedIn: !!token, token }
}

// 三周后另一个 Agent 不知道上面这个存在，又写了
// src/hooks/useUserAuth.ts
export function useUserAuth() {
  const [isAuth, setIsAuth] = useState(false)
  useEffect(() => {
    setIsAuth(!!localStorage.getItem('token'))
  }, [])
  return { isAuth }
}

// 又过了两周，第三个 Agent 写了
// src/utils/checkAuth.ts
export function checkAuth() {
  return localStorage.getItem('token') !== null
}
```

现在代码库里有三套认证逻辑，没有人知道该用哪个。

### 症状二：风格不一致

```typescript
// 有的组件这样写
const ChatPage: React.FC = () => { ... }

// 有的这样写
function ChatPage() { ... }

// 有的这样写
export default class ChatPage extends React.Component { ... }

// 有的 Hook 这样命名
const [loading, setLoading] = useState(false)

// 有的这样命名
const [isLoading, setIsLoading] = useState(false)

// 有的这样命名
const [fetchStatus, setFetchStatus] = useState('idle')
```

### 症状三：过度工程化

```typescript
// 任务：给一个按钮加个 loading 状态
// Agent 生成了一个"通用异步操作管理器"

class AsyncOperationManager {
  private operations: Map<string, AsyncOperation> = new Map()
  private subscribers: Set<Subscriber> = new Set()

  register(id: string, operation: AsyncOperation) { ... }
  execute(id: string) { ... }
  subscribe(callback: Subscriber) { ... }
  unsubscribe(callback: Subscriber) { ... }
  getStatus(id: string) { ... }
  // ... 200 行
}

// 实际上只需要
const [isLoading, setIsLoading] = useState(false)
```

### 症状四：死代码

```typescript
// Agent 生成了一个完整的功能，但最终需求变了，只用了其中一部分
// 剩下的代码没有被删除，也没有被引用

export function formatMessageForExport() { ... }  // 从未被调用
export function exportChatHistory() { ... }        // 从未被调用
export const EXPORT_FORMATS = ['pdf', 'txt', 'json']  // 从未被使用
```

## 二、熵增的速度为何在 AI 时代被大幅加快

先理解"熵"在代码库里的含义：

```
代码熵 = 代码库的混乱程度

熵越高：
  - 同一件事有多种做法（不一致性）
  - 模块之间耦合越来越紧（依赖混乱）
  - 死代码越来越多（噪音增加）
  - 没有人能完整理解整个代码库（认知负担爆炸）
```

**为什么 AI 时代熵增速度远超人工编码？**

| 对比维度 | 人工编码 | AI Agent 编码 |
|---------|---------|--------------|
| 代码生成速度 | 慢（人打字） | 极快（秒级生成） |
| 上下文记忆 | 人记得上周写了什么 | Agent 每次任务都是全新开始 |
| 风格一致性 | 同一个人风格自然一致 | 每次生成可能风格不同 |
| 重复实现 | 人会先搜索有没有现成的 | Agent 可能直接生成新的 |
| 死代码清理 | 人会顺手删掉不用的 | Agent 只完成任务，不主动清理 |

结论：**AI 生成代码的速度是人工的 10-100 倍，但产生 Slop 的速度也是 10-100 倍。**

## 三、OpenAI 团队的真实经历

OpenAI 在 100 万行代码实验中，曾经每周花 **20% 的时间**（相当于每周一整天）清理 AI Slop。

他们遇到的具体问题：

```
问题一：模式复制失控
  一个 Agent 用了某种"临时方案"解决了一个问题
  → 后续 Agent 看到这个模式，认为是"标准做法"
  → 这个临时方案被复制到了 30 个地方
  → 清理时需要改 30 个文件

问题二：重复实现爆炸
  同一个工具函数（比如格式化时间戳）
  → 在代码库里出现了 7 个不同的实现
  → 每个实现的行为略有不同
  → 没有人知道该用哪个

问题三：依赖关系腐化
  随着 Slop 积累，模块之间的依赖越来越混乱
  → 改一个地方，不知道会影响哪里
  → 测试越来越难写
  → 最终陷入"谁也不敢动"的状态
```

**解决方案：** 不是靠人工清理，而是设计一套**自动化的熵管理系统**。

## 四、前端项目的熵增热点

不同类型的代码，熵增速度不同。前端项目里最容易积累 Slop 的地方：

```
高风险区域（熵增最快）：
  ├── src/utils/          ← 工具函数，最容易重复实现
  ├── src/hooks/          ← 自定义 Hook，容易出现功能重叠
  ├── src/components/     ← 通用组件，容易出现多个相似版本
  └── src/types/          ← 类型定义，容易出现重复或冲突的类型

中风险区域：
  ├── src/store/          ← 状态管理，容易出现冗余状态
  └── src/api/            ← API 封装，容易出现重复的请求函数

低风险区域：
  └── src/pages/          ← 页面组件，通常一个路由对应一个文件，不容易重复
```

## 本节小结

```
AI Slop 的四种形态：
  重复实现（同一功能多个版本）
  风格不一致（命名、写法混乱）
  过度工程化（简单问题复杂方案）
  死代码（生成了但从未使用）

熵增加速的原因：
  AI 生成速度快 × 没有上下文记忆 × 不主动清理
  → 熵增速度是人工编码的 10-100 倍

前端熵增热点：
  utils/ > hooks/ > components/ > types/

OpenAI 的教训：
  靠人工清理不可持续（每周 20% 时间）
  必须设计自动化的熵管理系统
```
