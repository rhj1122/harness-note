# 4.5　Harness 自身的熵管理

> 本节目标：理解 Harness 本身也会老化，掌握定期维护 Harness 的方法，让约束系统保持长期有效。

---

## 一、Harness 也会腐烂

前四节讲的都是如何管理**代码库**的熵。

但有一个容易被忽视的问题：**Harness 本身也会随时间老化。**

```
Harness 老化的表现：

AGENTS.md 老化：
  - 项目架构变了，但 AGENTS.md 还描述旧架构
  - 某条规则已经被 ESLint 机械化了，但 AGENTS.md 里还重复写着
  - 常用命令变了（比如从 npm 迁移到 pnpm），但文档没更新

ESLint 规则老化：
  - 某条规则检测的违规模式已经不存在了，但规则还在跑
  - 规则的错误信息指向了已经删除的文档链接
  - 新的违规模式出现了，但没有对应的规则

黄金原则老化：
  - 某条原则针对的是旧技术栈（比如 Redux），但项目已经迁移到 Zustand
  - 某条原则的反例已经不会出现了（因为 ESLint 规则已经机械化了它）
  - 新的最佳实践出现了，但黄金原则没有更新
```

**老化的 Harness 比没有 Harness 更危险：** Agent 读到过时的规则，会按错误的标准生成代码，而工程师以为 Harness 在保护代码库，实际上它在误导 Agent。

---

## 二、Harness 老化的根本原因

```
代码库在变化：
  新功能 → 新的架构模式
  技术迁移 → 旧规则失效
  团队成长 → 最佳实践升级

但 Harness 没有跟上：
  AGENTS.md 没有人主动更新
  ESLint 规则没有人定期审查
  黄金原则没有人检查是否还适用
```

解法和代码库的文档腐烂一样：**不能依赖人记得去更新，要设计系统来提醒和触发更新。**

---

## 三、定期 Harness Review

Harness Review 是一个定期（建议每 2 周一次）的检查流程，专门审查 Harness 本身的健康状态。

### Harness Review 的检查清单

```markdown
## AGENTS.md 检查

□ 项目概述是否还准确？
□ 架构概览是否反映了当前的分层结构？
□ 层级规则是否还有效？有没有已经被 ESLint 机械化的规则可以简化？
□ 编码规范条目是否还适用？
□ 常用命令是否还正确？（npm vs pnpm，命令是否变了）
□ 文档索引是否还指向正确的文件？
□ AGENTS.md 是否超过了 100 行？（超过则需要精简）

## ESLint 规则检查

□ 每条自定义规则是否还在检测真实存在的违规模式？
□ 规则的错误信息中的文档链接是否还有效？
□ 有没有新的违规模式需要新增规则？
□ 有没有规则的阈值需要调整？（比如组件行数限制）
□ 规则的分级是否合理？（error vs warn）

## 黄金原则检查

□ 每条原则是否还适用于当前技术栈？
□ 有没有原则已经被 ESLint 规则覆盖，可以从黄金原则中移除？
□ 有没有新的最佳实践需要添加？
□ 反例是否还是真实会出现的场景？

## docs/ 文档检查

□ architecture.md 是否反映了当前架构？
□ tech-choices.md 是否包含了所有当前使用的库？
□ 有没有已经删除的模块还在文档里？
□ skills/ 下的技能文档是否还准确？
```

---

## 四、触发 Harness Review 的时机

除了定期检查，以下事件也应该触发 Harness Review：

```
主动触发（计划内）：
  ✅ 每 2 周一次的定期检查
  ✅ 每次大版本迭代结束后

被动触发（事件驱动）：
  ⚡ 技术栈迁移（换状态管理库、换构建工具等）
  ⚡ 架构重构（新增或删除层级）
  ⚡ 质量指标持续恶化（连续 3 周某指标超阈值）
  ⚡ Agent 反复犯同一类错误（说明 Harness 没有有效预防）
  ⚡ 新工程师加入（他们读 AGENTS.md 时发现的困惑往往是 Harness 老化的信号）
```

---

## 五、Harness Review 的执行方式

### 方式一：人工 Review（最简单）

```
每 2 周，工程师花 30 分钟：
  1. 过一遍 AGENTS.md 检查清单
  2. 过一遍 ESLint 规则检查清单
  3. 过一遍黄金原则检查清单
  4. 发现问题 → 直接更新或开 PR
```

适合：小团队、项目早期

### 方式二：Agent 辅助 Review

```
给 Agent 一个任务：
  "请检查 AGENTS.md、docs/golden-principles.md 和 eslint-rules/ 目录，
   找出可能已经过时的内容。
   对比当前代码库的实际结构，列出需要更新的地方。"

Agent 输出：
  - AGENTS.md 第 12 行：提到了 Redux，但项目已迁移到 Zustand
  - golden-principles.md GP-S01：提到了 Redux Toolkit，已不适用
  - eslint-rules/no-class-components.js：项目已全面使用函数式组件，此规则可删除

工程师审查 Agent 的发现，决定是否更新
```

适合：中大型团队、项目成熟期

### 方式三：自动化检测（最彻底）

```yaml
# .github/workflows/harness-health.yml
# 每月运行一次 Harness 健康检查

name: Harness Health Check

on:
  schedule:
    - cron: '0 9 1 * *'  # 每月 1 号上午 9 点

jobs:
  harness-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: 检查文档链接有效性
        run: npx markdown-link-check docs/**/*.md
      - name: 检查 AGENTS.md 行数
        run: |
          lines=$(wc -l < AGENTS.md)
          if [ $lines -gt 120 ]; then
            echo "⚠️ AGENTS.md 超过 120 行（当前 $lines 行），建议精简"
            exit 1
          fi
      - name: 生成 Harness 健康报告
        run: node scripts/harness-health-check.js
```

## 六、Harness 演进的正确姿势

Harness 不是一成不变的，它应该随项目演进而演进。

```
Harness 演进的方向：

早期（项目刚起步）：
  - AGENTS.md 很简单（20-30 行）
  - 只有基础的 ESLint 规则（格式、类型）
  - 黄金原则很少（5-10 条）
  - 重点：快速建立基础约束

中期（项目成长）：
  - AGENTS.md 逐渐丰富（60-80 行）
  - 自定义架构 ESLint 规则（10-20 条）
  - 黄金原则增加（20-30 条）
  - 重点：把反复出现的问题机械化

成熟期（项目稳定）：
  - AGENTS.md 保持精简（约 100 行）
  - ESLint 规则覆盖主要违规模式
  - 黄金原则稳定，定期审查
  - 重点：维护 Harness 的准确性，防止老化
```

**Harness 的成熟标志：** 工程师花在维护 Harness 上的时间越来越少，但代码质量越来越稳定。

```

  AGENTS.md 描述旧架构

  黄金原则针对旧技术栈

定期 Harness Review（每 2 周）：
  检查 AGENTS.md 准确性
  检查 ESLint 规则有效性
  检查黄金原则适用性

触发时机：
  定期（每 2 周）+ 事件驱动（技术迁移、架构重构、质量恶化）

执行方式：
  人工 Review（小团队）
  Agent 辅助 Review（中大型团队）
  自动化检测（成熟项目）

Harness 演进方向：
  早期：建立基础约束
  中期：把问题机械化
  成熟期：维护准确性，防止老化
```
