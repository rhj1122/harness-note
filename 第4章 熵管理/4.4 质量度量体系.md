# 4.4　质量度量体系

> 本节目标：学会用数字衡量代码库的健康状态，建立可追踪的质量指标，让"代码质量"从模糊感受变成可量化的数据。

---

## 一、为什么需要量化

"代码质量变差了"——这句话太模糊。

```
模糊的感受：
  "感觉代码越来越乱了"
  "好像有很多重复代码"
  "维护起来越来越费劲"

量化的数据：
  "重复代码块从上周的 3 个增加到了 11 个"
  "死代码文件从 2 个增加到了 9 个"
  "平均组件文件长度从 80 行增加到了 140 行"
```

量化的好处：
- **可追踪**：知道质量是在变好还是变差
- **可预警**：某个指标超过阈值时自动报警
- **可决策**：数据支撑"现在需要做清理"的判断
- **可验证**：清理之后能看到指标改善

## 二、前端项目的核心质量指标

### 指标一：重复率（Duplication Rate）

```
定义：代码库中重复代码块占总代码量的比例

工具：jscpd
命令：npx jscpd src --min-lines 10

健康阈值：< 5%
警告阈值：5% - 10%
危险阈值：> 10%

前端重点关注：
  - src/hooks/ 目录的重复率（最容易出现重复 Hook）
  - src/utils/ 目录的重复率（最容易出现重复工具函数）
```

### 指标二：死代码比例（Dead Code Ratio）

```
定义：未被任何地方引用的导出（组件、Hook、函数、类型）占总导出数的比例

工具：ts-prune
命令：npx ts-prune | wc -l（统计未使用导出数量）

健康阈值：< 2%
警告阈值：2% - 5%
危险阈值：> 5%
```

### 指标三：平均文件长度（Average File Length）

```
定义：src/ 目录下各类文件的平均行数

分类统计：
  组件文件（*.tsx）：健康 < 100 行，警告 100-150 行，危险 > 150 行
  Hook 文件（use*.ts）：健康 < 80 行，警告 80-120 行，危险 > 120 行
  工具函数（utils/*.ts）：健康 < 60 行，警告 60-100 行，危险 > 100 行
```

### 指标四：测试覆盖率（Test Coverage）

```
定义：被测试覆盖的代码行数占总代码行数的比例

工具：Vitest coverage
命令：npx vitest run --coverage

分类目标：
  Hooks 层：> 80%（业务逻辑最重要）
  API 层：> 70%
  Components 层：> 60%（UI 测试成本高）
  Utils 层：> 90%（纯函数最容易测试）
```

### 指标五：循环依赖数量（Circular Dependencies）

```
定义：代码库中存在循环依赖的模块对数量

工具：madge
命令：npx madge --circular src | grep -c "→"

健康阈值：0（零容忍）
警告阈值：1-3（需要尽快修复）
危险阈值：> 3（架构已经出问题）
```

### 指标六：TypeScript 严格模式违规数（Type Safety）

```
定义：使用 any、类型断言（as）、非空断言（!）的次数

工具：ESLint + @typescript-eslint 规则
规则：@typescript-eslint/no-explicit-any（warn）

健康趋势：数量随时间减少或持平
危险信号：数量快速增加（说明 Agent 在用 any 绕过类型检查）
```

## 三、质量仪表盘的设计

把以上指标汇总成一个可视化的质量仪表盘：

```markdown
# 代码库质量仪表盘
> 最后更新：2026-04-07（自动生成）

## 总体健康评分：78/100 ⚠️

## 核心指标

| 指标 | 当前值 | 阈值 | 状态 | 趋势（7天）|
|------|--------|------|------|-----------|
| 重复率 | 4.2% | <5% | ✅ 健康 | ↓ -0.8% |
| 死代码比例 | 3.1% | <2% | ⚠️ 警告 | ↑ +1.2% |
| 平均组件长度 | 112行 | <100行 | ⚠️ 警告 | ↑ +8行 |
| 测试覆盖率（Hooks）| 76% | >80% | ⚠️ 警告 | ↓ -2% |
| 循环依赖数量 | 0 | 0 | ✅ 健康 | → 持平 |
| any 使用次数 | 12 | <10 | ⚠️ 警告 | ↑ +3 |

## 需要关注
- 死代码比例上升，建议本周运行清理 Agent
- 平均组件长度超标，有 4 个组件超过 150 行需要拆分
- Hooks 测试覆盖率下降，最近新增的 Hook 缺少测试
```

## 四、趋势监控：质量随时间的变化曲线

单次数据不够，需要追踪趋势：

```
质量趋势的三种模式：

模式一：持续改善 ✅
  重复率：8% → 6% → 4% → 3%
  说明：清理流程在有效运转

模式二：缓慢恶化 ⚠️
  重复率：3% → 4% → 5% → 6%
  说明：Slop 产生速度 > 清理速度，需要加强清理频率

模式三：突然恶化 🚨
  重复率：3% → 3% → 3% → 9%
  说明：某次大量 Agent 任务产生了大量 Slop，需要立即处理
```

**趋势比单次数值更重要。** 一个指标当前值偏高但在持续改善，比当前值正常但在持续恶化要好。

## 五、自动化质量报告的实现

用一个简单的脚本把所有指标汇总：

```javascript
// scripts/quality-report.js

import { execSync } from 'child_process'
import { writeFileSync } from 'fs'

function runCommand(cmd) {
  try {
    return execSync(cmd, { encoding: 'utf-8' })
  } catch (e) {
    return e.stdout || ''
  }
}

// 收集各项指标
const metrics = {
  date: new Date().toISOString().split('T')[0],

  // 重复代码
  duplication: parseDuplicationRate(
    runCommand('npx jscpd src --min-lines 10 --reporters json --silent')
  ),

  // 死代码
  deadCode: runCommand('npx ts-prune 2>/dev/null').split('\n').filter(Boolean).length,

  // 循环依赖
  circularDeps: runCommand('npx madge --circular src 2>/dev/null')
    .split('\n').filter(l => l.includes('→')).length,

  // 测试覆盖率（从 coverage/coverage-summary.json 读取）
  coverage: parseCoverage(),
}

// 追加到历史记录
const history = JSON.parse(readFileSync('reports/quality-history.json', 'utf-8') || '[]')
history.push(metrics)
writeFileSync('reports/quality-history.json', JSON.stringify(history, null, 2))

// 生成 Markdown 报告
generateMarkdownReport(metrics, history)

console.log('质量报告已生成：reports/quality-report.md')
```

## 六、质量指标与 Harness 的联动

质量指标不只是用来"看"的，还要和 Harness 的其他部分联动：

```
指标超过警告阈值
    ↓
触发清理 Agent（4.3 节的周期性清理）
    ↓
清理 Agent 开 PR 修复
    ↓
指标恢复到健康范围

指标持续恶化（连续 3 周）
    ↓
触发 Harness Review（4.5 节）
    ↓
检查是否需要新增黄金原则或 ESLint 规则
    ↓
更新 Harness，从根源预防
```

## 本节小结

```
为什么量化：
  模糊感受 → 可追踪数据
  支撑决策，验证改善效果

六个核心指标：
  重复率（jscpd）
  死代码比例（ts-prune）
  平均文件长度
  测试覆盖率（Vitest coverage）
  循环依赖数量（madge）
  TypeScript any 使用次数

质量仪表盘：
  汇总所有指标，显示当前值 + 阈值 + 趋势

趋势比单次值更重要：
  持续改善 ✅ / 缓慢恶化 ⚠️ / 突然恶化 🚨

与 Harness 联动：
  指标超阈值 → 触发清理 Agent
  持续恶化 → 触发 Harness Review
```
