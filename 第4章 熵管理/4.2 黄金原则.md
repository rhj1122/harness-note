# 4.2　黄金原则（Golden Principles）

> 本节目标：理解黄金原则是什么，掌握编写方法，并能为自己的前端项目制定第一批黄金原则。

---

## 一、什么是黄金原则

黄金原则是**编码进仓库的最佳实践规则集**。

它和 AGENTS.md 里的规则不同：

```
AGENTS.md 里的规则：
  "禁止在 Components 层直接调用 fetch"
  → 这是架构约束，违反了 ESLint 会报错
  → 属于"不能做"的红线

黄金原则：
  "自定义 Hook 应该只返回数据和方法，不返回 JSX"
  "工具函数应该是纯函数，无副作用"
  "组件 props 超过 5 个时，考虑拆分或用 Context"
  → 这是最佳实践，违反了不会报错，但会让代码变差
  → 属于"应该怎么做"的标准
```

**黄金原则 = 定义"好代码长什么样"的标准。**

Agent 在生成代码时，会参考黄金原则来决定"怎么写更好"。

---

## 二、黄金原则 vs AGENTS.md：职责分工

```
AGENTS.md（架构规则）：
  ├── 强制性：违反会被 ESLint/CI 拦截
  ├── 关注：代码放在哪里、依赖方向
  └── 类比：交通法规（红灯必须停）

黄金原则（最佳实践）：
  ├── 引导性：违反不会报错，但会降低代码质量
  ├── 关注：代码怎么写更好、更可维护
  └── 类比：驾驶技巧（平稳加速更省油）

两者配合：
  AGENTS.md 保证代码"放对地方"
  黄金原则保证代码"写得好"
```

## 三、黄金原则的编写方法

一条好的黄金原则需要包含三个要素：

```
1. 规则本身（一句话，可执行）
2. 为什么（原因，让 Agent 理解背后的逻辑）
3. 反例（违反时的样子，帮助 Agent 识别问题）
```

### 写法示例

**差的写法：**
```
- 保持代码整洁
- 组件不要太大
- 避免重复代码
```

这些规则太模糊，Agent 无法执行。

**好的写法：**
```
原则：自定义 Hook 只返回数据和方法，不返回 JSX 元素。

为什么：Hook 是逻辑层，JSX 是视图层，混在一起会破坏分层架构，
        导致逻辑无法在不同 UI 场景下复用。

反例：
  // ❌ 错误
  function useAIChat() {
    const [messages, setMessages] = useState([])
    const MessageList = () => <ul>{messages.map(...)}</ul>  // 返回了 JSX
    return { messages, MessageList }
  }

  // ✅ 正确
  function useAIChat() {
    const [messages, setMessages] = useState([])
    return { messages, sendMessage }  // 只返回数据和方法
  }
```

## 四、前端项目的黄金原则示例

以下是一套适合 AI H5 应用的黄金原则，可以直接参考：

### 组件相关

```markdown
## 组件黄金原则

GP-C01：组件文件不超过 150 行
为什么：超过 150 行通常意味着组件承担了过多职责，应该拆分。
反例：一个 ChatPage.tsx 包含了消息列表、输入框、工具栏、设置面板的全部逻辑。

GP-C02：组件 props 超过 5 个时，改用 interface 定义并考虑拆分
为什么：过多 props 是组件职责不清晰的信号。
反例：<ChatMessage id={} content={} role={} timestamp={} isStreaming={} onRetry={} onCopy={} />

GP-C03：组件内不写业务逻辑，业务逻辑放在 Hooks 层
为什么：组件只负责渲染，业务逻辑在 Hook 里才能被复用和测试。
反例：在 onClick 里直接写 API 调用和状态更新逻辑。

GP-C04：避免在组件内使用 useEffect 处理数据获取，改用自定义 Hook
为什么：useEffect 处理数据获取容易产生竞态条件和内存泄漏，自定义 Hook 更安全。
```

### Hook 相关

```markdown
## Hook 黄金原则

GP-H01：Hook 只返回数据和方法，不返回 JSX
为什么：保持逻辑层和视图层的分离。

GP-H02：Hook 的返回值用对象解构，不用数组解构（除非是 useState 风格的配对）
为什么：对象解构允许调用方按需取值，数组解构强制按顺序，扩展性差。
反例：return [messages, sendMessage, isLoading, error]  // 调用方必须记住顺序

GP-H03：一个 Hook 只做一件事，名称能清晰描述它做什么
为什么：职责单一的 Hook 更容易测试和复用。
反例：useChat（既管理消息，又处理用户信息，又控制 UI 状态）

GP-H04：涉及 AI 流式响应的 Hook，必须处理中断和清理
为什么：用户切换页面时，未完成的流式请求必须被取消，否则会造成内存泄漏。
```

### API 层相关

```markdown
## API 层黄金原则

GP-A01：API 函数只做请求/响应转换，不包含业务判断
为什么：业务判断在 Hook 层，API 层保持纯粹，便于 mock 和测试。
反例：在 sendMessage() 里判断用户是否有权限。

GP-A02：所有 API 函数必须有明确的 TypeScript 返回类型
为什么：类型安全是前端工程质量的基础，any 是 Slop 的温床。
反例：async function getMessages(): Promise<any>

GP-A03：AI 流式 API 使用 AsyncGenerator，不使用回调
为什么：AsyncGenerator 更符合现代 TypeScript 风格，便于 for await...of 消费。
```

### 状态管理相关

```markdown
## Store 黄金原则

GP-S01：Store 只存储需要跨组件共享的状态，局部状态用 useState
为什么：过度使用全局 Store 会让状态流向难以追踪。
反例：把一个只在单个组件内使用的 loading 状态放进 Zustand Store。

GP-S02：Store 的 action 命名用动词，state 命名用名词
为什么：清晰的命名约定让 Agent 生成的代码风格一致。
反例：store.messages（✅）vs store.messageAction（❌）

GP-S03：禁止在 Store 外部直接修改 Store 的状态
为什么：所有状态变更必须通过 Store 定义的 action，保证可追踪性。
```

## 五、黄金原则的存放位置

黄金原则放在 `docs/golden-principles.md`：

```
docs/
├── README.md
├── architecture.md
├── conventions.md
├── golden-principles.md    ← 黄金原则
└── ...
```

在 `docs/README.md` 里加一行：

```markdown
## 代码质量标准
- [黄金原则](golden-principles.md) — 好代码的标准，Agent 生成代码时的参考
```

在 AGENTS.md 里不需要单独列，通过 `docs/README.md` 索引即可。

## 六、黄金原则的维护方式

黄金原则不是一次性写完的，它需要随项目演进持续更新。

### 触发更新的时机

```
1. 发现 AI Slop 的新模式
   → 分析根因：Agent 为什么会这样写？
   → 写一条黄金原则，告诉 Agent 正确的做法

2. Code Review 中反复出现同类问题
   → 说明 Agent 不知道这个最佳实践
   → 把它写成黄金原则

3. 重构时发现某种模式特别难维护
   → 把"避免这种模式"写成黄金原则，附上反例

4. 引入新技术栈
   → 为新技术写对应的黄金原则
```

### 维护节奏

```
每次 Agent 犯了一个"不是架构违规，但写得不好"的错误
    ↓
问自己：这个错误能用一条黄金原则来预防吗？
    ↓ 能
写一条新的黄金原则，加进 docs/golden-principles.md
    ↓
下次 Agent 读到这条原则，就不会再犯同样的错误
```

## 七、黄金原则 vs Linter 规则的选择

有时候你会纠结：这个规则应该写成黄金原则，还是写成 ESLint 规则？

```
判断标准：

能被机械检测的 → 写成 ESLint 规则
  例：Components 层不能 import API 层（可以检测 import 路径）
  例：Hook 文件必须以 use 开头（可以检测文件名）

需要语义理解的 → 写成黄金原则
  例：组件职责是否单一（需要理解代码含义）
  例：Hook 的返回值是否合理（需要判断设计好坏）
  例：是否过度工程化（需要理解业务需求）
```

**原则：能机械化的就机械化，不能机械化的就文档化。**

存放位置：
