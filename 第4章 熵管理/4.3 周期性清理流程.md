# 4.3　周期性清理流程

> 本节目标：理解为什么需要自动化清理，掌握设计周期性清理流程的方法，让代码库的熵增得到持续控制。

---

## 一、为什么需要周期性清理

上一节讲了黄金原则——它告诉 Agent "应该怎么写"。

但黄金原则是**预防性**的，只对新生成的代码有效。

已经存在的 Slop 怎么办？

```
黄金原则的局限：
  ✅ 新代码：Agent 读到原则，按标准生成
  ❌ 旧代码：已经存在的 Slop，黄金原则管不到

周期性清理的作用：
  ✅ 持续扫描已有代码
  ✅ 发现偏离标准的地方
  ✅ 自动开 PR 修复
  → 代码库的熵增被持续对冲
```

**类比：** 黄金原则是"不乱扔垃圾"的规定，周期性清理是"定期打扫"的环卫系统。两者缺一不可。

---

## 二、OpenAI 的节奏设计："技术债像高利贷"

OpenAI 团队用了一个很形象的比喻：

> **技术债像高利贷：每天小额偿还，比攒到最后一次性还要轻松得多。**

对应到实践：

```
错误做法（攒到最后）：
  每天积累 Slop，不管
  → 三个月后，Slop 已经蔓延到整个代码库
  → 需要花一整周做"大扫除"
  → 大扫除期间无法正常开发
  → 大扫除完成后，Slop 很快又积累回来

正确做法（每天小额偿还）：
  每天运行清理 Agent，处理当天产生的 Slop
  → 每次清理量很小（几个 PR）
  → 不影响正常开发节奏
  → 代码库始终保持在可接受的质量水平
```

## 三、周期性清理流程的设计

### 清理流程的三个阶段

```
阶段一：扫描（发现问题）
    ↓
阶段二：分析（判断严重程度）
    ↓
阶段三：修复（自动开 PR）
```

### 阶段一：扫描

后台 Agent 定期扫描代码库，寻找偏离黄金原则的地方：

```
扫描项目（前端版）：

重复实现检测：
  - 搜索功能相似的 Hook（名称相近、逻辑相似）
  - 搜索重复的工具函数（相同的输入输出逻辑）
  - 搜索相似的组件（结构相近的 UI 组件）

死代码检测：
  - 未被任何文件 import 的组件/Hook/工具函数
  - 定义了但从未使用的 TypeScript 类型
  - 注释掉的代码块

风格不一致检测：
  - 命名风格混用（camelCase vs snake_case）
  - 组件写法混用（函数式 vs 类组件）
  - 状态管理混用（useState vs useReducer 用于同类场景）

过度工程化检测：
  - 只被调用一次的抽象（过早抽象）
  - 超过 150 行的组件文件
  - 超过 100 行的 Hook 文件
```

### 阶段二：分析

不是所有扫描结果都需要立即修复，需要按严重程度分级：

```
P0（立即修复）：
  - 死代码（直接删除，无风险）
  - 明显的重复实现（合并，风险低）

P1（本周修复）：
  - 风格不一致（统一，需要测试）
  - 超长文件（拆分，需要仔细）

P2（本月修复）：
  - 过度工程化（简化，需要理解业务）
  - 轻微的最佳实践偏差
```

### 阶段三：修复

对每个问题，清理 Agent 自动开一个针对性的 PR：

```
PR 示例：

PR #312：[cleanup] 删除未使用的 useMessageExport Hook
  - 删除 src/hooks/useMessageExport.ts（从未被 import）
  - 删除对应的测试文件
  - CI 全绿，可自动合并

PR #313：[cleanup] 合并重复的认证 Hook
  - 将 useUserAuth 和 checkAuth 合并到 useAuth
  - 更新所有引用点
  - 需要人工审查（涉及多文件改动）

PR #314：[cleanup] 统一 loading 状态命名规范
  - 将 fetchStatus、isLoading、loading 统一为 isLoading
  - 影响 12 个文件
  - 需要人工审查
```

## 四、自动更新质量等级

除了开 PR，清理流程还应该维护一个**代码库质量报告**：

```markdown
# 代码库质量报告（自动生成，每周更新）

## 本周概况
- 新增 Slop：23 处
- 已清理 Slop：31 处
- 净变化：-8 处（质量提升）

## 当前质量指标
| 指标 | 本周 | 上周 | 趋势 |
|------|------|------|------|
| 重复实现数量 | 3 | 7 | ↓ 改善 |
| 死代码文件数 | 2 | 5 | ↓ 改善 |
| 超长组件数量 | 4 | 4 | → 持平 |
| 风格不一致处 | 8 | 12 | ↓ 改善 |

## 待处理问题
- P0（立即）：2 处
- P1（本周）：5 处
- P2（本月）：11 处
```

这个报告让工程师一眼看出代码库的健康趋势，不需要手动检查。

## 五、前端项目的清理工具

实现周期性清理，可以借助这些工具：

### 死代码检测

```bash
# ts-prune：检测未使用的 TypeScript 导出
npx ts-prune

# 输出示例：
# src/hooks/useMessageExport.ts:1 - exportMessages (unused)
# src/utils/formatDate.ts:15 - formatDateLong (unused)
```

### 重复代码检测

```bash
# jscpd：检测重复代码块
npx jscpd src --min-lines 10 --reporters json

# 输出：找到的重复代码块列表，包含文件路径和行号
```

### 循环依赖检测

```bash
# madge：检测模块循环依赖
npx madge --circular src

# 输出示例：
# Found 2 circular dependencies!
# src/hooks/useAIChat.ts -> src/hooks/useTracking.ts -> src/hooks/useAIChat.ts
```

### 包大小分析

```bash
# 构建后分析 bundle 大小
npm run build -- --report

# 检测是否有不必要的大依赖被引入
```

## 六、把清理流程集成进 CI

周期性清理不只是手动运行，还可以集成进 CI：

```yaml
# .github/workflows/weekly-cleanup.yml

name: Weekly Cleanup Scan

on:
  schedule:
    - cron: '0 2 * * 1'  # 每周一凌晨 2 点

jobs:
  scan:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci

      - name: 死代码扫描
        run: npx ts-prune > reports/dead-code.txt

      - name: 重复代码扫描
        run: npx jscpd src --reporters json --output reports/

      - name: 循环依赖扫描
        run: npx madge --circular src > reports/circular-deps.txt

      - name: 生成质量报告
        run: node scripts/generate-quality-report.js

      - name: 上传报告
        uses: actions/upload-artifact@v4
        with:
          name: quality-report
          path: reports/
```

## 七、清理 PR 的审查策略

清理 PR 和功能 PR 的审查策略不同：

```
死代码删除 PR：
  → 风险极低，CI 全绿即可自动合并
  → 理由：删除未使用的代码不会影响功能

命名统一 PR：
  → 风险低，但需要确认测试覆盖
  → 人工审查：30 秒确认改动范围合理

重复实现合并 PR：
  → 风险中等，需要确认行为一致
  → 人工审查：检查合并后的实现是否覆盖了所有原有场景

大规模重构 PR：
  → 风险高，必须人工仔细审查
  → 建议拆分成多个小 PR
```

## 本节小结

```
为什么需要周期性清理：
  黄金原则只预防新 Slop
  已有的 Slop 需要主动清理

节奏设计：
  "技术债像高利贷，每天小额偿还"
  每天小量清理 > 攒到最后大扫除

清理流程三阶段：
  扫描（发现问题）→ 分析（分级）→ 修复（自动开 PR）

前端清理工具：
  ts-prune（死代码）
  jscpd（重复代码）
  madge（循环依赖）

集成方式：
  GitHub Actions 定时任务（每周一次）
  生成质量报告，追踪趋势

清理 PR 审查：
  死代码删除 → 自动合并
  命名统一 → 快速人工审查
  重复合并 → 仔细人工审查
```
