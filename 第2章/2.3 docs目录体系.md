# 2.3　docs/ 目录体系

> 本节目标：理解深层文档的组织方式，能够为项目设计一套让 Agent 可以精准按需访问的 docs/ 结构。

---

## 一、为什么需要 docs/ 目录

上一节说了：AGENTS.md 是目录，约 100 行，详细内容放 docs/。

那 docs/ 里应该放什么？怎么组织？

先理解 docs/ 存在的根本原因：

```
Agent 执行任务时的信息需求是动态的：

  任务 A（修复 UI bug）
    需要：UI 层规范、组件约定
    不需要：数据库 schema、部署流程

  任务 B（新增 API 端点）
    需要：API 设计规范、Service 层约定、数据库 schema
    不需要：UI 组件约定、前端构建流程

docs/ 的作用：
  把不同类型的信息分门别类存放，
  让 Agent 在需要时能精准找到，
  不需要时不占用上下文。
```

---

## 二、docs/ 的核心分类

一个成熟项目的 docs/ 通常包含以下几类文档：

### 类型一：架构文档（architecture.md）

描述系统的整体结构，是最重要的一类文档。

```markdown
# 架构文档应包含：

1. 系统整体架构图（文字描述或 ASCII 图）
2. 各层的职责定义
3. 层级依赖规则（哪层可以依赖哪层）
4. 关键模块的说明
5. 数据流向描述
```

示例结构：
```
docs/architecture.md

## 系统架构
[架构图]

## 层级定义
- Types 层：纯数据类型，无副作用
- Config 层：配置加载，只依赖 Types
- Repo 层：数据访问，只依赖 Types/Config
- Service 层：业务逻辑，只依赖 Types/Config/Repo
- Runtime 层：应用入口，只依赖 Service 及以下
- UI 层：用户界面，只依赖 Runtime 及以下

## 依赖规则
[详细规则]

## 关键模块
[各模块说明]
```

### 类型二：编码规范（conventions.md）

项目的编码约定，Agent 在写代码时必须遵守。

```markdown
# 编码规范应包含：

1. 命名约定（变量、函数、类、文件的命名规则）
2. 文件组织约定（文件放在哪个目录）
3. 错误处理约定（如何处理异常）
4. 日志约定（用什么库，记录什么级别）
5. 测试约定（测试文件放哪里，命名规则）
6. 注释约定（什么情况下写注释）
```

### 类型三：技术选型（tech-choices.md）

记录项目用了哪些技术，以及为什么选这些技术。

```markdown
# 技术选型文档的价值：

Agent 在遇到新需求时，会倾向于引入新的库或工具。
tech-choices.md 告诉 Agent：
  - 项目已经用了哪些库（不要重复引入类似功能的库）
  - 为什么选这些库（背景和约束）
  - 哪些库是明确禁止的（以及原因）

示例条目：
  HTTP 客户端：使用 httpx（不用 requests，因为需要异步支持）
  日志：使用 structlog（不用 logging，因为需要结构化日志）
  测试：使用 pytest（不用 unittest）
```

### 类型四：设计决策记录（decisions/）

记录重要的架构和设计决策，以及决策背后的原因。

```
docs/decisions/
├── 001-use-postgresql.md
├── 002-layered-architecture.md
├── 003-async-first.md
└── ...
```

每个决策文档的格式：
```markdown
# 决策 001：使用 PostgreSQL

## 状态
已采纳（2026-01-15）

## 背景
[为什么需要做这个决策]

## 决策
[做了什么决策]

## 原因
[为什么这样决策]

## 后果
[这个决策带来的影响]
```

这类文档对 Agent 特别有价值：当 Agent 想"改进"某个设计时，
它能看到当初为什么这样设计，避免推翻有充分理由的决策。

### 类型五：执行计划（plans/）

当前正在进行的开发计划，Agent 执行任务时的参考。

```
docs/plans/
├── current-sprint.md      ← 当前迭代的任务列表
├── feature-auth.md        ← 认证功能的详细计划
└── refactor-service.md    ← Service 层重构计划
```

## 三、文档与代码共存于同一仓库

这是一个重要的工程原则：**文档必须和代码放在同一个 Git 仓库里。**

```
错误做法：
  代码 → GitHub 仓库
  文档 → Confluence / Notion / Google Docs

问题：
  - 文档和代码版本不同步
  - Agent 无法直接读取外部文档系统
  - 代码变了，文档没更新，Agent 读到过时信息
```

```
正确做法：
  代码 + 文档 → 同一个 Git 仓库

好处：
  - 文档随代码一起版本化
  - PR 可以同时包含代码变更和文档更新
  - Agent 可以直接读取文档
  - 文档过时时，Git blame 可以追溯
```

## 四、docs/README.md：文档总索引

docs/README.md 是所有文档的入口，Agent 查文档时先读这里，再按需深入。

```markdown
# docs/README.md

## 架构与设计
- [系统架构](architecture.md) — 分层结构、依赖规则、关键模块
- [技术选型](tech-choices.md) — 用了哪些库，为什么，禁用清单
- [设计决策](decisions/) — 重要决策的背景与原因

## 编码规范
- [编码约定](conventions.md) — 命名、文件组织、错误处理、日志

## 工具使用指南
- [Skills 索引](skills/README.md) — 何时用哪个工具

## 模块说明
- [认证模块](modules/auth.md)
- [支付模块](modules/payment.md)

## 其他
- [开发环境搭建](setup.md)
- [当前执行计划](plans/current-sprint.md)
```

有了这个文件，AGENTS.md 里只需要一行：

```markdown
## 文档
docs/README.md（所有文档的入口）
```

Agent 需要查任何文档时，先读 docs/README.md，找到对应链接，再按需深入。

## 五、入口文档 vs 深层文档的分层设计

docs/ 内部也需要分层，不能把所有内容平铺在一个文件里。

```
分层原则：

入口层（AGENTS.md 引用的）：
  - 每个文档一个主题
  - 开头有摘要（3-5句话）
  - 详细内容在后面，或链接到子文档

深层（按需访问）：
  - 可以很详细
  - 通过入口层的链接访问
  - Agent 只在需要时才读
```

示例：
```
AGENTS.md
  └── 架构概览（3句话）→ 详见 docs/architecture.md

docs/architecture.md
  ├── 摘要（5句话，Agent 快速了解）
  ├── 层级定义（详细）
  ├── 依赖规则（详细）
  └── 各模块说明 → 详见 docs/modules/
      ├── docs/modules/auth.md
      ├── docs/modules/payment.md
      └── docs/modules/notification.md
```

## 五、推荐的 docs/ 目录结构

```
docs/
├── README.md              ← 文档总索引（必备，Agent 查文档的入口）
├── architecture.md        ← 系统架构（必备）
├── conventions.md         ← 编码规范（必备）
├── tech-choices.md        ← 技术选型（必备）
├── api.md                 ← API 文档（有 API 时必备）
├── setup.md               ← 开发环境搭建
├── decisions/             ← 设计决策记录
│   ├── 001-xxx.md
│   └── 002-xxx.md
├── modules/               ← 各模块详细说明（按需创建）
│   ├── auth.md
│   └── payment.md
├── skills/                ← 工具使用指南（Skills 文档）
│   ├── README.md          ← Skills 索引
│   ├── search-code.md
│   └── write-tests.md
└── plans/                 ← 执行计划（可选）
    └── current-sprint.md
```

**最小可用版**（刚起步的项目）：
```
docs/
├── README.md
├── architecture.md
├── conventions.md
└── tech-choices.md
```

## 六、文档质量的判断标准

一份好的 docs/ 文档，Agent 读完后应该能回答这些问题：

```
architecture.md 读完后：
  ✅ 这个系统分几层？每层做什么？
  ✅ 我要写的代码应该放在哪一层？
  ✅ 这一层可以调用哪些其他层？

conventions.md 读完后：
  ✅ 新建一个文件，应该放在哪个目录？
  ✅ 函数命名应该用什么风格？
  ✅ 出现错误时应该怎么处理？

tech-choices.md 读完后：
  ✅ 需要发 HTTP 请求，用哪个库？
  ✅ 需要写日志，用哪个库？
  ✅ 这个需求已经有现成的库了吗？
```

如果 Agent 读完文档还是无法回答这些问题，说明文档需要补充。

## 本节小结

```
docs/ 的核心价值：
  存放 AGENTS.md 放不下的详细信息
  让 Agent 按需精准访问，不浪费上下文

docs/ 的核心文件：
  README.md         → 文档总索引（Agent 查文档的第一站）
  architecture.md   → 系统结构和层级规则
  conventions.md    → 编码约定
  tech-choices.md   → 技术选型和禁用清单
  decisions/        → 设计决策记录（防止 Agent 推翻有理由的决策）
  skills/           → 工具使用指南（Skills 文档）
  plans/            → 执行计划

关键原则：
  文档与代码同仓库（版本化）
  入口轻量，深层详细
  分层链接，按需访问

AGENTS.md 只需一行：
  docs/README.md（所有文档的入口）

最小可用版：
  README.md + architecture.md + conventions.md + tech-choices.md
```
