# 2.1　上下文是稀缺资源

> 本节目标：理解为什么"给 Agent 的信息"需要精心管理，以及管理不好会付出什么代价。

---

## 一、什么是上下文窗口

Agent 在执行任务时，它能"看到"的所有信息加在一起，就是它的**上下文（Context）**。

上下文窗口是有上限的——超出这个上限，模型就会开始"遗忘"早期的内容。

```
上下文窗口 = 工作台面

台面上可以放：
  ├── 任务描述（你告诉它要做什么）
  ├── AGENTS.md（项目说明书）
  ├── 读取的文件内容
  ├── 工具调用的结果
  ├── 对话历史
  └── 错误信息和反馈

台面大小有限。放太多，早期内容会被推出台面，Agent 就"忘了"。
```

现代主流模型的上下文窗口大约在 **10 万到 20 万 token** 之间（1 token ≈ 0.75 个英文单词，约 0.5 个汉字）。

听起来很大，但一个中等规模的代码库，光是把所有文件读进来就能轻松超出这个限制。

## 二、信息太多 vs 信息太少的双重陷阱

上下文管理面临一个两难困境：

### 陷阱一：信息太少

```
Agent 读取的信息不足 → 做出错误决策

典型症状：
  - 重复造轮子（不知道已有类似实现）
  - 违反架构规则（不知道规则在哪）
  - 风格不一致（不知道项目的编码规范）
  - 选错技术方案（不知道项目已有的技术选型）
```

### 陷阱二：信息太多

```
Agent 读取的信息过多 → 上下文窗口被占满，关键信息被挤出

典型症状：
  - Agent "忘记"了任务开始时的关键约束
  - Agent 在大量无关信息中迷失，做出偏离目标的决策
  - 执行速度变慢（处理大量无关信息消耗算力）
  - 成本急剧上升（token 消耗与上下文大小成正比）
```

**正确的做法不是"给越多越好"，而是"给恰好需要的"。**

## 三、OpenAI "一个巨型 AGENTS.md" 为什么失败了

这是一个真实的教训，值得仔细理解。

### 他们做了什么

OpenAI 团队最初的想法很直觉：
> "把所有规则、所有文档、所有约定都写进一个 AGENTS.md，这样 Agent 什么都知道了。"

结果这个 AGENTS.md 越写越长，最终变成了一个几千行的巨型文件。

### 为什么失败

**问题一：上下文污染**

每次 Agent 开始工作，都要把这几千行全部读进上下文。
大量内容与当前任务无关，但仍然占据宝贵的上下文空间。

**问题二：注意力稀释**

研究表明，模型对上下文中间部分的"注意力"远低于开头和结尾。
把关键规则埋在几千行文档的中间，Agent 很可能根本没有"注意到"它。

```
上下文注意力分布（示意）：

开头 ████████████ 高
中间 ████░░░░░░░░ 低（关键规则埋在这里 = 被忽略）
结尾 ████████████ 高
```

**问题三：维护困难**

几千行的单一文件，随着项目演进越来越难以维护。
不同规则之间开始出现矛盾，没有人知道哪条规则是最新的。

### 正确的解法

```
错误做法：一个巨型 AGENTS.md（几千行）
                ↓
正确做法：AGENTS.md 是目录（约100行）+ docs/ 是深层文档

AGENTS.md 只写：
  - 项目是什么（1-2句话）
  - 架构概览（指向 docs/architecture.md）
  - 关键规则（最重要的 5-10 条，直接写）
  - 深层文档的索引（"详细规范见 docs/xxx.md"）

docs/ 里写：
  - 完整的架构文档
  - 详细的编码规范
  - 设计决策记录
  - 等等...
```

这样 Agent 每次都读轻量的 AGENTS.md（快速获取全局视角），
需要深入某个领域时，再按需读取对应的 docs/ 文件。

## 四、上下文工程的核心原则

从上面的分析，可以提炼出上下文工程的三条核心原则：

### 原则一：按需加载

不要一次性把所有信息塞进上下文。
设计一个机制，让 Agent 在需要时才加载相关信息。

```
类比：图书馆 vs 把所有书堆在桌上
  图书馆：需要什么书，去找什么书
  堆在桌上：所有书都在，但找不到需要的那本
```

### 原则二：入口轻量，深层详细

AGENTS.md 是入口，保持轻量（约100行）。
详细内容放在 docs/ 里，通过链接按需访问。

### 原则三：相关性优先

Agent 在执行特定任务时，上下文里应该主要是与这个任务相关的信息。
无关信息越少越好。

## 本节小结

```
上下文窗口 = 有限的工作台面

双重陷阱：
  信息太少 → Agent 做错误决策
  信息太多 → 关键信息被挤出，注意力稀释

OpenAI 的教训：
  巨型 AGENTS.md 失败的原因：
    上下文污染 + 注意力稀释 + 维护困难

正确方向：
  AGENTS.md 是目录（轻量）
  docs/ 是深层文档（详细）
  按需加载，相关性优先
```
