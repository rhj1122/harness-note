# 2.6　渐进式工具发现（Progressive Disclosure）

> 本节目标：理解为什么不能把所有工具一次性塞给 Agent，以及如何设计"按需发现工具"的机制。

---

## 一、问题的起源

随着项目成熟，工具会越来越多：

```
早期项目（5个工具）：
  read_file / write_file / run_tests / check_lint / create_pr

成熟项目（可能有 50+ 个工具）：
  文件操作类：read_file, write_file, delete_file, move_file...
  搜索类：search_code, search_docs, search_issues...
  测试类：run_unit_tests, run_e2e_tests, run_lint, check_types...
  数据库类：query_db, migrate_db, seed_db...
  部署类：build_image, deploy_staging, deploy_prod...
  监控类：get_logs, get_metrics, get_traces...
  ...
```

如果把所有工具定义一次性放进上下文：

```
问题一：上下文爆炸
  50 个工具 × 每个工具定义约 200 token = 10,000 token
  这还没算工具的 Skills 文档
  → 大量上下文空间被工具定义占据，留给实际任务的空间变少

问题二：注意力稀释
  Agent 面对 50 个工具，需要从中选择
  大量无关工具的存在会干扰 Agent 的决策
  → 选错工具的概率上升

问题三：维护困难
  每次加新工具，都要更新那个巨大的工具列表
  → 和"巨型 AGENTS.md"一样的问题
```

---

## 二、渐进式工具发现的核心思想

解法和 AGENTS.md 的设计哲学一脉相承：**按需加载，不要一次性全给。**

```
传统做法（一次性全给）：
  任务开始 → 加载所有 50 个工具定义 → Agent 开始工作

渐进式工具发现：
  任务开始 → 加载少量核心工具 + 一个"工具搜索"工具
                ↓
           Agent 需要某类工具时
                ↓
           调用"工具搜索"找到相关工具
                ↓
           加载该工具的定义
                ↓
           使用工具完成任务
```

关键：**给 Agent 一个"找工具的工具"，而不是把所有工具都给它。**

---

## 三、tool search 机制

这是渐进式工具发现的核心实现：一个专门用来搜索其他工具的工具。

### tool_search 的工作方式

```
工具定义：
  tool_search(query: str) -> List[ToolDescription]
  根据描述搜索可用的工具，返回匹配的工具列表及其使用说明

Agent 使用示例：

  任务：我需要查看应用的错误日志

  Agent 思考：我需要一个查看日志的工具，但我不确定叫什么名字
  Agent 调用：tool_search("查看应用日志")
  返回结果：
    - get_app_logs(service, time_range) → 获取应用日志
    - get_error_logs(level, since) → 获取错误级别日志
    - search_logs(query, time_range) → 在日志中搜索关键词

  Agent 选择：get_error_logs，因为任务是查看错误日志
  Agent 调用：get_error_logs(level="ERROR", since="1h")
```

## 四、工具分层设计

渐进式工具发现需要把工具分层组织：

```
第一层：核心工具（始终在上下文中）
  约 5-10 个，每个 Agent 都会用到的基础工具：
  - read_file / write_file
  - run_command
  - tool_search        ← 这个必须在第一层
  - create_pr

第二层：常用工具（按需加载）
  约 20-30 个，特定类型任务会用到：
  - 测试相关工具
  - 数据库相关工具
  - 监控相关工具

第三层：专用工具（极少使用）
  约 10-20 个，只在特定场景下使用：
  - 部署工具
  - 数据迁移工具
  - 紧急修复工具
```

```
上下文中的工具数量：
  始终存在：5-10 个（核心工具）
  按需加载：0-5 个（当前任务需要的）
  总计：约 10-15 个（而不是 50 个）
```

## 五、工具目录文件

除了 tool_search，还可以用一个静态的工具目录文件来帮助 Agent 发现工具。

工具目录放在 `docs/tools/` 下，和 skills/ 一起归入 docs/ 统一管理：

```
docs/
├── README.md              ← 文档总索引（包含工具目录入口）
├── skills/                ← 工具使用指南（Why）
└── tools/                 ← 工具目录（What/How）
    ├── README.md          ← 工具总目录
    ├── testing.md
    ├── database.md
    ├── observability.md
    └── deployment.md
```

`docs/tools/README.md` 的内容：

```markdown
# 工具目录

## 核心工具（始终可用）
- read_file：读取文件内容
- write_file：写入文件
- run_command：执行 shell 命令
- tool_search：搜索可用工具
- create_pr：创建 Pull Request

## 按类别查找工具

### 测试相关
详见：tools/testing.md
包含：run_unit_tests, run_e2e_tests, check_coverage...

### 数据库相关
详见：tools/database.md
包含：query_db, run_migration, check_schema...

### 监控相关
详见：tools/observability.md
包含：get_logs, get_metrics, get_traces...

### 部署相关
详见：tools/deployment.md
包含：build_image, deploy_staging, rollback...
```

在 `docs/README.md` 里加一行：

```markdown
## 工具
- [工具目录](tools/README.md) — 所有可用工具的分类索引
```

AGENTS.md 依然只需一行：`docs/README.md（所有文档的入口）`

## 六、渐进式工具发现的实战设计

### 步骤一：盘点所有工具

列出项目中所有可用的工具，按类别整理。

### 步骤二：确定核心工具

问自己：哪些工具是 90% 的任务都会用到的？这些进入第一层。

通常不超过 10 个。

### 步骤三：实现 tool_search

最简单的实现：一个搜索工具描述文件的脚本。

```
tool_search 的实现逻辑：
  1. 读取 tools/ 目录下所有工具描述文件
  2. 对每个工具的名称和描述做关键词匹配
  3. 返回匹配度最高的 3-5 个工具
```

### 步骤四：在 AGENTS.md 中声明

AGENTS.md 只需一行：

```markdown
## 文档
docs/README.md（所有文档的入口）
```

Agent 从 docs/README.md 找到工具目录，再用 tool_search 按需加载具体工具。

## 七、第二章总结：上下文工程全景

学完第二章，你已经掌握了上下文工程的完整工具箱：

```
上下文工程工具箱（完整版）

┌─────────────────────────────────────────────────────┐
│  入口层（Agent 每次必读）                              │
│  └── AGENTS.md（约100行，是目录不是百科全书）           │
│       └── 唯一指向：docs/README.md                   │
├─────────────────────────────────────────────────────┤
│  docs/（所有文档统一在这里）                           │
│  ├── README.md（文档总索引，Agent 查文档的第一站）      │
│  ├── architecture.md（架构和层级规则）                 │
│  ├── conventions.md（编码规范）                       │
│  ├── tech-choices.md（技术选型）                      │
│  ├── decisions/（设计决策记录）                       │
│  ├── skills/（工具使用指南：何时用哪个工具）            │
│  └── tools/（工具目录：有哪些工具可用）                │
├─────────────────────────────────────────────────────┤
│  工具发现机制（按需加载）                               │
│  └── tool_search（动态搜索工具定义）                   │
├─────────────────────────────────────────────────────┤
│  自维护机制（后台运行）                                 │
│  └── 文档检查 Agent（保持文档与代码同步）               │
└─────────────────────────────────────────────────────┘

核心原则贯穿始终：
  按需加载，不要一次性全给
  入口轻量，深层详细
  所有文档统一归入 docs/，AGENTS.md 只是入口
```
