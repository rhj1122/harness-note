【概念定位】
Harness Engineering 由 OpenAI 工程师 Ryan Lopopolo 于 2026 年 2 月正式命名，
Mitchell Hashimoto（Terraform/Ghostty 创始人）将其推广。核心理念：

    当 AI 编码智能体成为主要生产力单元时，
    工程师的职责不再是"写代码"，
    而是"设计让智能体可靠地写出并维护代码的约束环境系统"。

    这套约束环境系统的整体 ——
    包括规则、文档、linter、架构边界、反馈回路、可观测性 ——
    统称为 harness（驾具）。

    核心等式：Agent = Model + Harness
    （模型提供智能，harness 使智能变得可靠、可维护）

第二章  上下文工程（Context Engineering）—— 让 Agent 看到正确的信息
================================================================================

2.1  上下文是稀缺资源
     - 什么是上下文窗口，它的容量限制
     - 信息太多 vs 信息太少的双重陷阱
     - OpenAI "一个巨型 AGENTS.md"方案为什么失败了

2.2  AGENTS.md 的设计哲学
     - AGENTS.md 是目录，不是百科全书（约100行原则）
     - Greg Brockman 建议：每次 Agent 犯错就更新 AGENTS.md
     - AGENTS.md 的典型结构与写法
     - 实战：为你的第一个项目写一份 AGENTS.md

2.3  docs/ 目录体系
     - 设计文档、架构图、执行计划的存放规范
     - 文档应与代码共存于同一仓库（版本化）
     - 深层文档 vs 入口文档的分层设计

2.4  文档自维护机制（Agent for Agent）
     - 用后台 Agent 定期扫描过时文档
     - 让 Agent 开 PR 更新文档，人只需审批
     - 文档→反馈回路→文档 的持续闭环

2.5  Skills（技能文档）
     - Skills 与工具定义的区别：Why vs How
     - Skills 如何让 Agent 知道"该何时用哪个工具"
     - 编写有效 Skills 文档的方法

2.6  渐进式工具发现（Progressive Disclosure）
     - 为何不能把所有工具定义一次性塞入上下文
     - tool search 机制：让 Agent 按需发现工具
     - 实战：设计工具发现策略

第三章  架构约束（Architectural Constraints）—— 机械地阻止 Agent 犯错
================================================================================

3.1  为什么需要架构约束
     - Agent 是"模式复制器"：好模式和坏模式都会被复制
     - 没有约束时代码库会发生什么（依赖混乱、耦合爆炸）
     - "机械约束"比"文字说明"更可靠的原因

3.2  分层架构模型
     - OpenAI 内部的层级定义：
       Types → Config → Repo → Service → Runtime → UI
     - 依赖方向规则：只能沿箭头方向流动
     - Providers 横切关注点模式（鉴权、连接器、遥测、特性开关）
     - 如何为自己的项目设计分层架构

3.3  自定义 Linter 的设计
     - Linter 不只标记违规，还告诉 Agent 如何修复（最聪明的设计）
     - Linter 错误信息 = 修复指令 的写法原则
     - 用 Agent 编写 Linter（Codex 写 Linter 本身）
     - 常见架构违规的 Linter 规则示例

3.4  CI 层面的机械拒绝
     - 违反架构规则的代码在 CI 层面被自动拒绝
     - CI 流水线的设计：检查项分级（阻断 vs 警告）
     - PR 审查流程与自动合并策略

3.5  边界划分实战
     - 为项目划定模块边界
     - 识别"横切关注点"并集中管理
     - 反例分析：典型的边界违反案例

4.3  周期性清理流程
     - 后台 Agent 持续扫描偏差
     - 自动更新质量等级
     - 自动开有针对性的重构 PR
     - "技术债像高利贷：每天小额偿还"的节奏设计

4.4  质量度量体系
     - 如何量化代码库的"熵"
     - 关键指标：重复率、耦合度、测试覆盖率
     - 趋势监控：质量随时间的变化曲线

4.5  Harness 自身的熵管理
     - Harness 本身也会老化：AGENTS.md、Linter 规则的维护
     - 定期 Harness Review 的实践

第五章  可观测性与反馈回路（Observability & Feedback Loops）
================================================================================

5.1  为什么 Agent 需要可观测性
     - Agent 无法"看"到结果，必须通过工具感知环境
     - 可观测性是 Agent 自我验证的基础设施

5.2  本地可观测性栈搭建
     - Victoria Logs / Victoria Metrics / Victoria Traces 组合
     - LogQL 和 PromQL 查询能力
     - 每个 git worktree 独立的应用实例配置

5.3  浏览器自动化（Agent 复现 Bug）
     - Chrome DevTools Protocol 接入
     - DOM 快照、截图、导航能力
     - Agent 如何用这些工具复现 Bug 并验证修复

5.4  Git Worktree 并行工作流
     - git worktree 机制：多个 Agent 同时工作互不干扰
     - Worktree 的创建、分配与清理策略
     - 实战：配置多 Agent 并行开发环境

5.5  反馈回路设计
     - 短回路：Agent 执行 → Linter 报错 → Agent 修复
     - 中回路：PR → CI 检查 → 自动合并 or 驳回
     - 长回路：后台质量扫描 → 重构 PR → 人工审批
     - 反馈延迟的影响与最小化策略
