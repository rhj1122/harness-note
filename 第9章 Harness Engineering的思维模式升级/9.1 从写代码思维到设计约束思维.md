# 9.1　从"写代码"思维到"设计约束"思维

> 本节目标：理解 Harness Engineering 要求的根本性思维转变，以及如何在日常工作中培养"设计约束"的习惯。

---

## 一、两种思维模式的对比

### 写代码思维（传统）

```
问题出现
    ↓
我来解决这个问题
    ↓
写代码修复
    ↓
问题解决了（这次）
```

工程师是**问题的直接解决者**。

### 设计约束思维（Harness Engineering）

```
问题出现
    ↓
为什么会出现这个问题？
    ↓
是因为 Agent 不知道某条规则？
还是因为规则没有被机械执行？
    ↓
设计一个约束，让这个问题永远不再出现
    ↓
问题被永久消除
```

工程师是**系统的设计者**，而不是问题的直接解决者。

---

## 二、一个具体的例子

**场景：** Agent 在 ChatMessage 组件里直接调用了 AI SDK。

### 写代码思维的处理方式

```
发现问题
    ↓
修改 ChatMessage 组件，把 AI SDK 调用移到 Hook 里
    ↓
PR 合并，问题解决

结果：
  这次的问题解决了
  但下次 Agent 还会犯同样的错误
  工程师需要反复处理同类问题
```

### 设计约束思维的处理方式

```
发现问题
    ↓
问：为什么 Agent 会这样做？
答：因为没有规则阻止它，而且代码库里有先例

    ↓
设计约束：
  1. 写一条 ESLint 规则：no-ai-sdk-in-components
     错误信息：告诉 Agent 应该通过 useAIProvider() 调用
  2. 更新 AGENTS.md：加一行提示
  3. 修复这次的违规代码

    ↓
结果：
  这次的问题解决了
  以后 Agent 永远不会再犯这个错误
  工程师不需要再处理这类问题
```

**关键差异：** 写代码思维解决"这次"，设计约束思维解决"永远"。

## 三、如何培养"设计约束"的习惯

### 习惯一：每次 Agent 犯错，问"为什么"

```
不要只问：这里怎么修？
要问：Agent 为什么会这样做？

可能的原因：
  A. Agent 不知道这条规则
     → 更新 AGENTS.md 或 docs/

  B. Agent 知道规则但没遵守
     → 写 ESLint 规则机械执行

  C. 规则本身不清晰，Agent 理解错了
     → 重写规则，加上反例

  D. 任务描述不够清晰
     → 改进任务描述模板
```

### 习惯二：评估修复的层次

```
每次修复问题时，问自己：
  "我是在解决症状，还是在消除根因？"

症状修复（治标）：
  改掉这次的违规代码
  → 下次还会出现

根因消除（治本）：
  设计约束，让这类问题永远不再出现
  → 一劳永逸

目标：尽量做根因消除，而不是反复治标
```

### 习惯三：把经验编码进系统

```
人的经验 vs 系统的规则：

人的经验：
  "我知道 AI SDK 不能在组件里直接调用"
  → 只有你知道，新人不知道，Agent 不知道
  → 你离职了，经验就消失了

系统的规则：
  ESLint 规则 + AGENTS.md 条目
  → 所有人都知道，Agent 也知道
  → 永久有效，不依赖任何人

每次你学到一个教训，问自己：
  "这个教训能编码进 Harness 吗？"
  如果能，立即做。
```

## 四、思维转变的三个阶段

### 阶段一：被动响应（初学者）

```
特征：
  Agent 犯错 → 修复代码 → 继续
  没有思考为什么，没有预防措施

感受：
  "感觉在不停地救火"
  "同样的问题反复出现"
```

### 阶段二：主动预防（进阶）

```
特征：
  Agent 犯错 → 修复代码 + 更新 AGENTS.md
  开始建立规则，但主要靠文字说明

感受：
  "Agent 犯错的频率在下降"
  "但还是有一些规则被忽略"
```

### 阶段三：系统设计（成熟）

```
特征：
  Agent 犯错 → 修复代码 + 写 ESLint 规则 + 更新文档
  重要规则全部机械化
  Harness 在持续自我完善

感受：
  "Agent 犯的错越来越少"
  "我的工作重心从'修错误'转向了'设计系统'"
  "代码库在变好，而不是变差"
```

## 五、工程师角色的根本转变

```
传统工程师：
  主要产出：代码
  核心技能：写出正确的代码
  工作重心：实现功能

Harness Engineering 工程师：
  主要产出：约束系统（Harness）
  核心技能：设计让 AI 可靠工作的环境
  工作重心：维护和完善 Harness

类比：
  传统工程师 = 工厂工人（亲手生产产品）
  Harness 工程师 = 工厂设计师（设计生产线，让机器可靠生产）
```

这不是说工程师变得"不重要"了，而是工程师的价值从"执行"转向了"设计"——这是更高层次的工作。

## 本节小结

```
两种思维模式：
  写代码思维：解决"这次"的问题
  设计约束思维：消除"永远"的问题

培养习惯：
  每次犯错问"为什么"
  评估修复是治标还是治本
  把经验编码进系统（ESLint + AGENTS.md）

三个阶段：
  被动响应 → 主动预防 → 系统设计

角色转变：
  从"工厂工人"（亲手写代码）
  到"工厂设计师"（设计让 AI 可靠工作的系统）
```
