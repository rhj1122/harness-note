# 9.3　与其他工程范式的关系

> 本节目标：理解 Harness Engineering 与 DevOps、Platform Engineering、TDD 的关系，找到共同点，避免重复造轮子。

---

## 一、共同的底层逻辑

在深入对比之前，先理解这几个范式的共同点：

```
DevOps、Platform Engineering、TDD、Harness Engineering
都在做同一件事：

  通过"外部系统约束"来提升质量
  而不是依赖"个人自律"

传统做法（依赖个人）：
  "大家要写好代码"
  "大家要做好测试"
  "大家要遵守规范"
  → 效果取决于每个人的自律程度，不稳定

系统约束做法：
  "CI 不通过就无法合并"（DevOps）
  "没有测试就无法提交"（TDD）
  "违反架构规则就报错"（Harness Engineering）
  → 效果不依赖个人，稳定可靠
```

---

## 二、Harness Engineering vs DevOps

### DevOps 是什么

DevOps 关注的是**开发和运维的协作**，核心是：
- 自动化构建、测试、部署
- 持续集成（CI）和持续交付（CD）
- 基础设施即代码

### 关系

```
DevOps 和 Harness Engineering 的关系：

  DevOps 提供了基础设施：
    CI 流水线（GitHub Actions / GitLab CI）
    自动化部署
    监控和告警

  Harness Engineering 在 DevOps 基础上扩展：
    在 CI 里加入架构约束检查
    在 CD 里加入质量门禁
    把 AI Agent 纳入 DevOps 工作流

  简单说：
    DevOps = 让代码从开发到生产的流程自动化
    Harness Engineering = 让 AI Agent 在这个流程里可靠工作
```

### 前端视角

```
你已经在用的 DevOps 实践：
  GitHub Actions CI（第 7.3 节配置的）
  npm run build 自动构建
  
Harness Engineering 在此基础上加了：
  架构约束检查（ESLint 自定义规则）
  质量度量（ts-prune、jscpd、madge）
  周期性清理（定时 Actions）
```

---

## 三、Harness Engineering vs Platform Engineering

### Platform Engineering 是什么

Platform Engineering 关注的是**为开发团队提供内部平台**，核心是：
- 构建内部开发者平台（IDP）
- 标准化开发工具链
- 降低开发者的认知负担

### 关系

```
Platform Engineering 和 Harness Engineering 的关系：

  Platform Engineering 的目标：
    让开发者专注于业务逻辑，不需要关心基础设施
    "你只需要写代码，平台帮你处理其他一切"

  Harness Engineering 的目标：
    让 AI Agent 专注于执行任务，不需要关心约束
    "Agent 只需要写代码，Harness 帮它处理约束"

  两者的共同理念：
    通过平台/系统降低认知负担
    让"使用者"（开发者/Agent）专注于核心工作

  区别：
    Platform Engineering 服务于人类开发者
    Harness Engineering 服务于 AI Agent
    （但两者可以共存，甚至 Harness 可以是平台的一部分）
```

---

## 四、Harness Engineering vs TDD（测试驱动开发）

### TDD 是什么

TDD 的核心流程：
```
先写测试（定义期望行为）
    ↓
写代码让测试通过
    ↓
重构代码（保持测试通过）
```

### 关系

```
TDD 和 Harness Engineering 的共同点：

  都是"约束驱动"的开发方式：
    TDD：测试约束代码的行为（功能正确性）
    Harness：ESLint 约束代码的结构（架构合理性）

  都强调"先定义标准，再写代码"：
    TDD：先写测试（定义功能标准）
    Harness：先写 AGENTS.md 和 ESLint 规则（定义架构标准）

  都通过"外部反馈"驱动修复：
    TDD：测试失败 → 修复代码
    Harness：ESLint 报错 → Agent 修复代码

区别：
  TDD 关注：代码做了什么（功能）
  Harness 关注：代码怎么组织（结构）

  TDD 是开发方法论
  Harness 是 AI 时代的工程系统

两者可以结合：
  TDD 保证功能正确
  Harness 保证架构合理
  → 功能 + 结构双重保障
```

## 五、在前端项目中的实际结合

你的 AI H5 项目可以同时应用这三个范式：

```
DevOps（基础设施层）：
  GitHub Actions CI/CD
  自动构建和部署
  → 保证代码能到达用户

TDD（功能层）：
  Vitest 单元测试
  React Testing Library 组件测试
  → 保证功能正确

Harness Engineering（架构层）：
  AGENTS.md + docs/（上下文工程）
  ESLint 自定义规则（架构约束）
  黄金原则 + 周期性清理（熵管理）
  → 保证代码可维护

三者的分工：
  DevOps：代码能跑
  TDD：代码做对
  Harness：代码好维护
```

## 本节小结

```
共同底层逻辑：
  都通过"外部系统约束"提升质量
  不依赖个人自律，稳定可靠

与 DevOps 的关系：
  DevOps 提供 CI/CD 基础设施
  Harness 在 CI 里加入架构约束和质量门禁
  → Harness 是 DevOps 的扩展

与 Platform Engineering 的关系：
  Platform Engineering 服务于人类开发者
  Harness Engineering 服务于 AI Agent
  → 理念相同，服务对象不同

与 TDD 的关系：
  TDD 约束功能正确性
  Harness 约束架构合理性
  → 互补，可以结合使用

前端项目的结合：
  DevOps（代码能跑）
  + TDD（代码做对）
  + Harness（代码好维护）
  = 完整的质量保障体系
```
