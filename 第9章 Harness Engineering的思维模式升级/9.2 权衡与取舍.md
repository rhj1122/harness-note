# 9.2　权衡与取舍

> 本节目标：理解 Harness 设计中的核心权衡，掌握找到合适平衡点的方法，避免走向两个极端。

---

## 一、Harness 越重 vs 越轻的利弊

Harness 不是越重越好，也不是越轻越好。

```
Harness 太轻（规则太少）：
  优点：搭建快，维护成本低
  缺点：
    Agent 犯错率高
    Slop 快速积累
    代码库质量难以保证

Harness 太重（规则太多）：
  优点：约束严格，质量有保障
  缺点：
    Agent 陷入无休止的修复循环
    每次任务都要修复大量 ESLint 错误
    工程师花大量时间维护 Harness 本身
    开发效率反而下降
```

**目标：找到"刚好够用"的平衡点。**

---

## 二、判断 Harness 是否过重的信号

```
信号一：Agent 的 CI 首次通过率 < 50%
  说明：规则太多或太严，Agent 频繁触发
  处理：检查哪些规则被触发最多，考虑降级为 warn 或删除

信号二：工程师花在维护 Harness 上的时间 > 花在审查 PR 上的时间
  说明：Harness 本身成了负担
  处理：精简规则，删除低价值的约束

信号三：Agent 开始用奇怪的方式绕过规则
  说明：规则太严，Agent 在"钻空子"
  处理：重新审视规则的设计，是否过于死板

信号四：AGENTS.md 超过 150 行
  说明：信息过载，Agent 开始忽略部分内容
  处理：精简 AGENTS.md，把内容移到 docs/
```

---

## 三、判断 Harness 是否过轻的信号

```
信号一：Agent 的 CI 首次通过率 > 95%
  可能说明：规则太少，没有覆盖真正的问题
  （注意：也可能是 Harness 运转良好，需要结合其他信号判断）

信号二：代码库里出现重复实现
  说明：缺少"先搜索再实现"的约束

信号三：同一类错误反复出现
  说明：这类错误应该被机械化，但还没有对应的规则

信号四：工程师花大量时间修复 Agent 的错误
  说明：Harness 没有有效预防这类错误
```

---

## 四、约束的颗粒度：太细碍手，太粗无效

### 太细的约束（碍手）

```
示例：
  规则：函数名必须以动词开头
  规则：变量名不能超过 20 个字符
  规则：每个文件必须有文件头注释
  规则：import 语句必须按字母顺序排列

问题：
  这些规则对代码质量的提升微乎其微
  但 Agent 需要花大量时间满足这些要求
  工程师也需要花时间维护这些规则
  → 高成本，低收益
```

### 太粗的约束（无效）

```
示例：
  规则：保持代码整洁
  规则：遵守最佳实践
  规则：避免重复代码

问题：
  这些规则太模糊，Agent 无法执行
  → 零成本，零收益
```

### 合适的颗粒度

```
好的约束特征：
  ✅ 可以被机械检测（ESLint 能写出来）
  ✅ 违反后代价明显（架构腐化、维护困难）
  ✅ Agent 容易犯这类错误（有真实案例）
  ✅ 修复方向清晰（错误信息能说清楚怎么改）

示例：
  "Components 层不能直接导入 API 层"
  → 可以机械检测（检查 import 路径）
  → 违反后架构腐化
  → Agent 确实容易犯
  → 修复方向清晰（移到 Hook 层）
```

---

## 五、自动化 vs 人工审查的边界

不是所有事情都应该自动化，需要明确边界：

```
适合自动化的：
  ✅ 有明确对错标准的检查（ESLint 规则）
  ✅ 重复性高的操作（格式化、死代码清理）
  ✅ 客观可量化的指标（测试覆盖率、文件行数）
  ✅ 低风险的 PR 合并（文档更新、死代码删除）

需要人工判断的：
  ❌ 架构设计决策（哪种方案更好）
  ❌ 业务逻辑正确性（功能是否符合需求）
  ❌ 用户体验判断（交互是否合理）
  ❌ 安全相关代码（鉴权、加密）
  ❌ 重大重构（影响范围大，风险高）
```

**原则：自动化处理"有标准答案"的问题，人工处理"需要判断"的问题。**

---

## 六、找到平衡点的实践方法

### 方法一：从少到多，逐步增加

```
不要一开始就配置 30 条规则
从 3 条最重要的开始
→ 观察效果
→ 如果 Agent 还在犯某类错误，加新规则
→ 如果某条规则触发太频繁，考虑删除或降级

原则：每条规则都应该有"它存在的理由"
      如果说不出理由，就不需要这条规则
```

### 方法二：定期审查规则的价值

```
每 2 周 Harness Review 时，对每条规则问：
  "这条规则在过去 2 周里，阻止了多少次真实的违规？"

  阻止了 5 次以上 → 保留，价值高
  阻止了 1-4 次 → 保留，但观察
  从未触发 → 考虑删除（可能已经不需要了）
```

### 方法三：听 Agent 的"抱怨"

```
如果 Agent 在同一个任务里触发了同一条规则 3 次以上：
  可能是规则设计有问题（太严格或描述不清）
  → 重新审视这条规则
  → 改进错误信息，或调整规则阈值
```

## 本节小结

```
核心权衡：
  Harness 太轻 → Slop 积累，质量失控
  Harness 太重 → Agent 效率下降，维护成本高
  目标：刚好够用

过重的信号：
  CI 首次通过率 < 50%
  维护 Harness 的时间 > 审查 PR 的时间
  Agent 开始绕过规则

过轻的信号：
  同类错误反复出现
  代码库出现重复实现
  工程师花大量时间修复 Agent 错误

约束颗粒度：
  太细（碍手）：高成本，低收益
  太粗（无效）：零成本，零收益
  合适：可机械检测 + 违反代价明显 + 修复方向清晰

自动化边界：
  有标准答案 → 自动化
  需要判断 → 人工

找平衡的方法：
  从少到多，逐步增加
  定期审查规则价值
  听 Agent 的"抱怨"
```
