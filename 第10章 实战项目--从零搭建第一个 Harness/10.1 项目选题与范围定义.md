## 一、项目基本情况

**项目名：** aiMiniChat
**性质：** 工作项目 + Harness 练手项目（双重目标）
**规模：** 个人，单仓库，GitLab

**技术栈：**
```
运行时：Node v24 + pnpm v10
仓库：GitLab 单仓库
构建：Vite v8（含 Sentry 集成）
语言：TypeScript v6
框架：React v19 + React Router v7 + Zustand v5
样式：Tailwind v4 + Radix UI
测试：Vitest v4 + React Testing Library v16
规范：ESLint v10 + Prettier v3 + Husky v9
```

**核心功能：**
- 若干管理页面
- 一个 AI 助手对话页面（流式输出）

**AI 服务：**
- 开发阶段：本地 VoltAgent 服务（HTTP 接口，流式输出）
- 后续：迁移到公司网关 AI 服务

---

## 二、双重目标的策略

这个项目同时是工作项目和 Harness 练手项目，需要明确优先级：

```
目标一（工作）：按时交差，保证质量
目标二（学习）：Harness 跑起来，积累实践经验

策略：Harness 是"增强手段"，不是"赌注"

  Harness 搭建成功 → AI Agent 帮你写代码，效率更高
  Harness 搭建失败 → 退回手动写代码，项目照常推进

原则：先搭最小可用 Harness，验证能跑，再逐步完善
      不要一开始就搭完整 Harness，风险太高
```

## 四、AI 服务对接说明

```
开发阶段（当前）：
  本地运行 VoltAgent 服务（my-first-agent 工程）
  aiMiniChat 通过 HTTP 接口调用
  支持流式输出（SSE 或 streaming）

对接方式：
  所有 AI 调用统一封装在 src/providers/AIProvider.tsx
  src/api/aiApi.ts 负责具体的 HTTP 请求
  Hooks 层通过 useAIProvider() 调用，不直接接触 HTTP

后续迁移：
  换成公司网关 AI 服务时，只需改 AIProvider 的实现
  Hooks 层和 Components 层代码不需要改动
  → 这正是 Providers 模式的价值
```

## 五、"可维护性"的验收标准

Harness 是否成功，用这些标准来衡量：

```
验收标准一：Agent 能正确放置文件
  给 Agent 一个任务，生成的文件在正确的目录
  （组件在 components/，Hook 在 hooks/，API 在 api/）

验收标准二：Agent 不违反架构规则
  生成的代码通过 ESLint 检查（含自定义架构规则）
  不需要工程师手动修复架构违规

验收标准三：Agent 能自我验证
  Agent 完成任务后，主动运行 check 和 test
  GitLab CI 首次通过率 > 70%

验收标准四：代码风格一致
  AI 生成的代码与手写代码风格一致
  不出现命名混乱、写法不统一的情况
```
