## 一、AGENTS.md 撰写过程

### 初稿要求

参考附录 B 的模板，填写五个部分：
1. 项目概述（aiMiniChat 是什么）
2. 架构分层（七层模型）
3. 关键规则（3-5 条）
4. 常用命令（pnpm 的 dev / check / test / build）
5. 文档入口（docs/README.md）

### Review 反馈

初稿提交后，收到以下反馈并修正：

| 问题 | 修正 |
|------|------|
| `pnpm check` 注释只写了"类型检查工具" | 改为"tsc 类型检查 + ESLint 代码规范检查" |
| 关键规则第1条和第3条重复 | 合并为一条 |
| 缺少 AI SDK 约束 | 确认项目通过 WebSocket 调用 AI，不需要 AI SDK 规则 |
| `docs/README.md` 格式不规范 | 加上括号说明 |

### 关键讨论

**Providers 层是否需要：**
- AIProvider → 不需要（AI 调用是普通 WebSocket，走 api/ 层）
- AuthProvider → 需要（登录态管理是横切关注点）
- ErrorProvider → 需要（Sentry 错误边界是横切关注点）

**判断标准：** 需要 React Context 传递 → Providers；纯函数直接调用 → utils

## 二、docs/ 文档体系

### 创建的文件

```
docs/
├── README.md              ← 文档总索引
├── architecture.md        ← 系统架构（重点文档）
├── conventions.md         ← 编码规范（待完善）
├── golden-principles.md   ← 黄金原则（待完善）
├── pr-template.md         ← PR 描述模板
└── skills/
    └── README.md          ← Skills 索引（待完善）
```

### architecture.md 的关键讨论

**Config 层与常量的归属：**
- 初始设计：常量放在 Config 层
- 问题：常量会被各层引用，但 Store/Hooks/Components 不能依赖 Config 层
- 最终方案：业务常量放 Types 层（最底层，任何层可依赖），Config 层只读取环境变量

**"不能跨层"的表述修正：**
- 初始写法："依赖方向只能向下，不能跨层"
- 问题：Types 是最底层，所有层依赖它都是"跨层"的
- 最终写法："依赖方向只能向下，每一层只能依赖比自己更底层的层"

**utils/ 目录的定位：**
- 不属于七层架构中的任何一层
- 不属于横切关注点（Providers）
- 定位：纯工具函数，无副作用，任何层可依赖
- 约束：不能 import Store、Hooks 层（否则说明它包含了业务逻辑，应该放 Hooks 层）
