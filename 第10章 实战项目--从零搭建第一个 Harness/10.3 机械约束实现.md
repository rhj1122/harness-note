# 10.3　机械约束实现

> 本节记录 ESLint 自定义规则和 CI 配置的实现过程。

---

## 一、ESLint 自定义规则

### 实现的规则

| 规则 | 文件 | 检测内容 | 级别 |
|------|------|---------|------|
| ARCH-001 | `no-network-in-components.js` | Components/Pages 层直接使用 fetch、WebSocket、axios、request | error |
| ARCH-002 | `no-store-in-pages.js` | Pages 层直接 import Store 层 | error |
| ARCH-003 | `no-store-in-utils.js` | utils/ 层 import Store 或 Hooks 层 | error |

### 规则设计的关键讨论

**从 fetch 检测到全面网络请求检测：**
- 初始设计：只检测 `fetch()` 和 `new WebSocket()`
- 问题：项目使用 axios 封装的 `request.js`，直接检测 fetch 不够
- 最终方案：同时检测 fetch、WebSocket、axios 包、`@/utils/request` 路径

**utils/ 层的约束：**
- 起因：讨论中发现 utils/ 容易被塞入业务逻辑（import Store）
- 新增规则：`no-store-in-utils`，禁止 utils/ import Store 和 Hooks 层
- 判断标准：如果需要访问状态 → 不是工具函数，应该放 Hooks 层

### 踩过的坑

| 问题 | 原因 | 解决 |
|------|------|------|
| `context.getFilename is not a function` | ESLint v10 废弃了 `getFilename()` | 改用 `context.filename` |
| `.eslintignore` 警告 | ESLint v10 不再支持 `.eslintignore` 文件 | 删掉文件，用 `eslint.config.js` 里的 `ignores` |
| `eslint-plugin-react` 崩溃 | v7 不兼容 ESLint v10 flat config | 换用 `@eslint-react/eslint-plugin` |

### 规则文件的编写规范

```
每条规则的错误信息必须包含：
  [错误代码] 简短描述
  原因：为什么这样做是错的
  修复：具体的修复步骤
  参考：docs/architecture.md#xxx

使用 context.filename（不是 context.getFilename()）
路径判断用 .replace(/\\/g, '/') 处理 Windows 路径
```

## 二、CI 配置

### 实现的配置

| 文件 | 平台 | 用途 |
|------|------|------|
| `.github/workflows/harness-ci.yml` | GitHub Actions | 当前自测阶段使用 |
| `.gitlab-ci.yml` | GitLab CI | 后续公司项目使用 |

### 四个阶段

```
阶段一：quick-check   → pnpm format:check + pnpm type-check
阶段二：arch-check    → pnpm lint（含自定义 Harness 规则）
阶段三：test          → pnpm test:run
阶段四：build         → pnpm build
```

### 关键讨论

**GitLab CI 的 workflow 触发条件：**
- `$CI_PIPELINE_SOURCE == "merge_request_event"` → 所有 MR 都触发
- `$CI_PIPELINE_SOURCE == "push" && $CI_COMMIT_BRANCH == "main"` → 只有指定分支的 push 触发
- `==` 是精确匹配，`=~` 是正则匹配

**GitHub → GitLab 迁移：**
- 只需要改 CI 配置文件格式，里面跑的命令一行不用改
- 核心差异：GitHub 用 `needs` 声明依赖，GitLab 用 `stages` 顺序自动依赖

## 三、经验总结

```
ESLint 规则从少到多：
  先写 3 条最重要的，验证能跑
  后续发现新的违规模式再加

ESLint v10 的坑：
  很多旧的 API 和插件不兼容
  优先查官方文档，不要照搬旧教程

CI 配置的核心：
  不管用什么平台，跑的命令是一样的
  平台只是"在哪里跑"的区别
```
