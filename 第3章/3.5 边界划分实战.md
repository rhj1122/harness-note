# 3.5　边界划分实战

> 本节目标：学会为真实项目划定模块边界，识别横切关注点，并通过反例分析理解典型的边界违反是什么样的。

---

## 一、什么是模块边界

分层架构解决了"纵向"的依赖问题（哪层依赖哪层）。

但一个真实项目里，同一层内部也会有多个模块，它们之间同样需要边界：

```
Service 层内部（横向边界）：

  UserService      OrderService      PaymentService
       │                │                  │
       └────────────────┴──────────────────┘
                        │
              这些模块之间如何交互？
              谁可以调用谁？
              共享数据怎么处理？
```

**模块边界**定义了：
- 每个模块负责什么（职责范围）
- 模块对外暴露什么接口（公开 API）
- 模块内部隐藏什么（实现细节）

---

## 二、划定模块边界的步骤

### 步骤一：按业务域划分

最自然的边界是**业务域**（Domain）：

```
电商项目的业务域：
  用户域（User）    → 注册、登录、个人信息
  商品域（Product） → 商品列表、详情、库存
  订单域（Order）   → 下单、支付、物流
  通知域（Notification） → 邮件、短信、推送
```

每个业务域对应一个模块，模块内部自治，对外只暴露必要的接口。

### 步骤二：定义每个模块的公开接口

```
OrderModule 对外暴露：
  + placeOrder(userId, items) → Order
  + cancelOrder(orderId) → void
  + getOrderStatus(orderId) → OrderStatus

OrderModule 内部隐藏：
  - calculateDiscount()      ← 内部实现，不对外
  - validateInventory()      ← 内部实现，不对外
  - OrderRepository          ← 数据访问，不对外
```

**原则：最小化公开接口。** 暴露越少，边界越稳定，Agent 越难犯错。

### 步骤三：确定模块间的依赖关系

```
允许的依赖：
  OrderModule → UserModule（下单需要验证用户）
  OrderModule → ProductModule（下单需要检查库存）
  OrderModule → NotificationModule（下单后发通知）

禁止的依赖：
  UserModule → OrderModule（用户模块不应该知道订单的存在）
  ProductModule → OrderModule（商品模块不应该知道订单的存在）
```

画出依赖图，确保没有循环依赖。

### 步骤四：写进文档和 Linter

```
文档（docs/architecture.md）：
  用文字和图描述模块边界和依赖关系

Linter 规则：
  检测禁止的跨模块依赖
  例：UserModule 的文件不能 import OrderModule 的内部实现
```

## 三、识别横切关注点

横切关注点是那些"每个模块都需要，但不属于任何单一模块"的功能。

常见的横切关注点：

```
鉴权（Authentication）
  每个 API 端点都需要验证用户身份
  → 不应该在每个 Service 里各自实现

日志（Logging）
  每个模块都需要记录日志
  → 不应该在每个模块里各自配置日志库

错误处理（Error Handling）
  所有模块的错误都需要统一格式
  → 不应该在每个模块里各自定义错误类型

遥测（Telemetry）
  所有操作都需要记录指标和追踪
  → 不应该在每个模块里各自接入监控

特性开关（Feature Flags）
  某些功能需要动态开关
  → 不应该在每个模块里各自读取配置
```

### 横切关注点的正确处理方式：Providers

```
错误做法：
  UserService:
    import sendgrid  # 直接依赖邮件库
    def register(email):
        sendgrid.send(...)  # 直接调用

  OrderService:
    import sendgrid  # 又一次直接依赖
    def placeOrder():
        sendgrid.send(...)  # 又一次直接调用

问题：
  - 邮件库散落在多个模块
  - 换邮件服务商需要改多处
  - Agent 写新功能时会继续引入直接依赖
```

```
正确做法：
  EmailProvider（在 Providers 层）:
    def send(to, subject, body):
        sendgrid.send(...)  # 只在这一处依赖邮件库

  UserService:
    def register(email, emailProvider):
        emailProvider.send(...)  # 通过 Provider 调用

  OrderService:
    def placeOrder(emailProvider):
        emailProvider.send(...)  # 同一个 Provider

好处：
  - 邮件逻辑集中在一处
  - 换邮件服务商只改 EmailProvider
  - Agent 写新功能时，自然地使用 Provider
```

## 四、反例分析：典型的边界违反案例

### 案例一：模块直接访问另一个模块的内部实现

```python
# ❌ 错误：OrderService 直接访问 UserRepo（跨越了模块边界）
class OrderService:
    def placeOrder(self, userId, items):
        # 直接访问 User 模块的内部数据层
        user = UserRepo.findById(userId)  # 违规！
        ...

# ✅ 正确：通过 UserService 的公开接口访问
class OrderService:
    def placeOrder(self, userId, items):
        # 通过 User 模块的公开接口
        user = self.userService.getUser(userId)  # 正确
        ...
```

**为什么这是问题：**
如果 UserRepo 的实现改变（比如换了数据库），OrderService 也会受影响。
模块边界的意义就是隔离变化。

---

### 案例二：业务逻辑泄漏到 Repo 层

```python
# ❌ 错误：Repo 层包含了业务判断
class OrderRepo:
    def create(self, order):
        # 这是业务逻辑，不应该在 Repo 层
        if order.total > 10000:
            order.requiresApproval = True
        if len(order.items) > 50:
            raise Exception("订单商品数量超限")
        db.insert(order)

# ✅ 正确：Repo 层只做数据操作
class OrderRepo:
    def create(self, order):
        db.insert(order)
        return order

# 业务逻辑在 Service 层
class OrderService:
    def placeOrder(self, userId, items):
        order = Order(userId=userId, items=items)
        if order.total > 10000:
            order.requiresApproval = True
        if len(items) > 50:
            raise BusinessError("订单商品数量超限")
        return self.orderRepo.create(order)
```

### 案例三：循环依赖

```python
# ❌ 错误：UserService 和 OrderService 互相依赖
class UserService:
    def __init__(self, orderService):
        self.orderService = orderService

    def deleteUser(self, userId):
        # 删除用户时取消所有订单
        self.orderService.cancelAllOrders(userId)

class OrderService:
    def __init__(self, userService):
        self.userService = userService

    def placeOrder(self, userId, items):
        # 下单时验证用户
        user = self.userService.getUser(userId)
```

**问题：** A 依赖 B，B 依赖 A，形成循环。
无法单独测试任何一个，修改任何一个都可能影响另一个。

```python
# ✅ 正确：打破循环，通过事件或回调解耦
class UserService:
    def deleteUser(self, userId):
        # 发布事件，不直接调用 OrderService
        self.eventBus.publish("user.deleted", userId)

class OrderService:
    def __init__(self):
        # 订阅事件，不依赖 UserService
        self.eventBus.subscribe("user.deleted", self.cancelAllOrders)
```

### 案例四：共享可变状态

```python
# ❌ 错误：多个模块共享同一个全局状态
GLOBAL_CACHE = {}  # 全局缓存

class UserService:
    def getUser(self, userId):
        if userId in GLOBAL_CACHE:
            return GLOBAL_CACHE[userId]
        ...

class OrderService:
    def placeOrder(self, userId, items):
        # 也在修改同一个全局缓存
        GLOBAL_CACHE[f"order_{userId}"] = ...
```

**问题：** 任何模块都可以修改全局状态，边界形同虚设。

```python
# ✅ 正确：每个模块管理自己的状态
class UserService:
    def __init__(self, cache):
        self._cache = cache  # 注入的、隔离的缓存

class OrderService:
    def __init__(self, cache):
        self._cache = cache  # 不同的缓存实例
```

## 五、边界划分的自检清单

完成边界划分后，用这个清单验证：

```
模块职责：
  □ 每个模块的职责是否可以用一句话描述？
  □ 是否有模块承担了两种以上不相关的职责？

公开接口：
  □ 每个模块的公开接口是否最小化？
  □ 是否有内部实现被意外暴露？

依赖关系：
  □ 是否有循环依赖？
  □ 是否有模块直接访问另一个模块的内部层（如直接访问 Repo）？

横切关注点：
  □ 日志、鉴权、错误处理是否集中在 Providers 层？
  □ 是否有模块直接依赖外部服务（邮件、短信等）而不通过 Provider？

文档和约束：
  □ 模块边界是否写进了 docs/architecture.md？
  □ 是否有 Linter 规则检测禁止的跨模块依赖？
```

## 本节小结

```
模块边界的两个维度：
  纵向：分层架构（Types→Config→Repo→Service→Runtime→UI）
  横向：业务域划分（User/Order/Product/Notification）

划定边界的步骤：
  1. 按业务域划分模块
  2. 定义每个模块的最小公开接口
  3. 确定模块间依赖关系（画图，检查循环）
  4. 写进文档 + 编写 Linter 规则

横切关注点：
  日志、鉴权、错误处理、遥测、特性开关
  → 统一放在 Providers 层，通过依赖注入使用

四类典型违规：
  跨越模块边界直接访问内部实现
  业务逻辑泄漏到 Repo 层
  循环依赖
  共享可变全局状态
```
