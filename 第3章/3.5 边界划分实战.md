# 3.5　边界划分实战

> 本节目标：学会为真实前端项目划定模块边界，识别横切关注点，并通过反例分析理解典型的边界违反是什么样的。

---

## 一、什么是模块边界

分层架构解决了"纵向"的依赖问题（哪层依赖哪层）。

但一个真实前端项目里，同一层内部也会有多个模块，它们之间同样需要边界：

```
Hooks 层内部（横向边界）：

  useAIChat      useUserProfile      usePayment
       │                │                  │
       └────────────────┴──────────────────┘
                        │
              这些 Hooks 之间如何交互？
              谁可以调用谁？
              共享状态怎么处理？
```

**模块边界**定义了：
- 每个模块负责什么（职责范围）
- 模块对外暴露什么接口（公开 API）
- 模块内部隐藏什么（实现细节）

---

## 二、划定模块边界的步骤

### 步骤一：按业务域划分

最自然的边界是**业务域**（Domain）：

```
AI 内嵌 H5 应用的业务域：
  对话域（Chat）      → 消息发送、流式渲染、历史记录
  用户域（User）      → 登录态、个人信息、Token 管理
  AI 配置域（AIConfig）→ 模型选择、参数配置、Prompt 管理
  埋点域（Tracking）  → 用户行为上报、性能监控
```

每个业务域对应一组文件，域内部自治，对外只暴露必要的接口。

### 步骤二：定义每个模块的公开接口

```
ChatModule 对外暴露（Hooks 层）：
  + useAIChat()         → { messages, isStreaming, sendMessage, clearHistory }
  + useChatHistory()    → { history, loadMore, hasMore }

ChatModule 内部隐藏：
  - chatStore           ← 内部 Store，不对外直接暴露
  - streamParser()      ← 内部工具函数，不对外
  - aiApi.ts            ← API 封装，通过 Hook 访问
```

**原则：最小化公开接口。** 暴露越少，边界越稳定，Agent 越难犯错。

### 步骤三：确定模块间的依赖关系

```
允许的依赖：
  ChatModule → UserModule（发消息需要验证登录态）
  ChatModule → AIConfigModule（发消息需要读取 AI 配置）
  TrackingModule → ChatModule（埋点需要监听对话事件）

禁止的依赖：
  UserModule → ChatModule（用户模块不应该知道对话的存在）
  AIConfigModule → ChatModule（AI 配置模块不应该知道对话的存在）
```

画出依赖图，确保没有循环依赖。

### 步骤四：写进文档和 ESLint

```
文档（docs/architecture.md）：
  用文字和图描述模块边界和依赖关系

ESLint 规则：
  检测禁止的跨模块依赖
  例：UserModule 的文件不能直接 import ChatModule 的内部实现
```

## 三、识别横切关注点

横切关注点是那些"每个模块都需要，但不属于任何单一模块"的功能。

前端项目常见的横切关注点：

```
鉴权 Token 管理（AuthProvider）
  每个 API 请求都需要携带 Token
  → 不应该在每个 API 函数里各自处理 Token

请求拦截器（RequestProvider）
  所有请求需要统一的错误处理、重试、超时配置
  → 不应该在每个 fetch 调用里各自配置

埋点/监控（TrackingProvider）
  所有用户操作都需要上报
  → 不应该在每个组件里各自调用埋点 SDK

主题/国际化（ThemeProvider）
  所有组件都需要访问主题变量和语言包
  → 不应该在每个组件里各自读取配置

AI SDK 封装（AIProvider）
  所有 AI 调用需要统一的模型配置、错误处理、Token 计费
  → 不应该在每个 Hook 里各自初始化 AI SDK
```

### 横切关注点的正确处理方式：Providers

```
错误做法：
  useAIChat:
    import OpenAI from 'openai'  // 直接依赖 AI SDK
    const client = new OpenAI({ apiKey: ... })  // 直接初始化

  useAISummary:
    import OpenAI from 'openai'  // 又一次直接依赖
    const client = new OpenAI({ apiKey: ... })  // 又一次初始化

问题：
  - AI SDK 初始化散落在多个 Hook
  - 换 AI 服务商需要改多处
  - Agent 写新功能时会继续引入直接依赖
```

```
正确做法：
  AIProvider（在 Providers 层）:
    const aiClient = new OpenAI({ apiKey: config.aiApiKey })
    export function useAIProvider() {
      return { streamMessage, generateText }  // 只在这一处依赖 AI SDK
    }

  useAIChat:
    const { streamMessage } = useAIProvider()  // 通过 Provider 调用

  useAISummary:
    const { generateText } = useAIProvider()  // 同一个 Provider

好处：
  - AI SDK 初始化集中在一处
  - 换 AI 服务商只改 AIProvider
  - Agent 写新功能时，自然地使用 useAIProvider()
```

## 四、反例分析：典型的边界违反案例

### 案例一：Hook 直接访问另一个模块的内部 Store

```typescript
// ❌ 错误：useAIChat 直接访问 userStore（跨越了模块边界）
function useAIChat() {
  // 直接访问 User 模块的内部 Store
  const userId = useUserStore(s => s.userId)  // 违规！
  // ...
}

// ✅ 正确：通过 UserModule 的公开 Hook 访问
function useAIChat() {
  // 通过 User 模块的公开接口
  const { userId } = useUserProfile()  // 正确
  // ...
}
```

**为什么这是问题：**
如果 userStore 的结构改变（比如 userId 改名为 id），useAIChat 也会受影响。
模块边界的意义就是隔离变化。

---

### 案例二：业务逻辑泄漏到 API 层

```typescript
// ❌ 错误：API 层包含了业务判断
// src/api/aiApi.ts
export async function sendMessage(messages: Message[]) {
  // 这是业务逻辑，不应该在 API 层
  if (messages.length > 100) {
    messages = messages.slice(-100)  // 截断历史，业务决策！
  }
  if (!isUserVip) {
    throw new Error('非 VIP 用户不能使用此功能')  // 权限判断，业务决策！
  }
  return await aiClient.chat(messages)
}

// ✅ 正确：API 层只做请求/响应转换
// src/api/aiApi.ts
export async function sendMessage(messages: Message[]) {
  return await aiClient.chat(messages)  // 只做 API 调用
}

// 业务逻辑在 Hooks 层
// src/hooks/useAIChat.ts
function useAIChat() {
  const { isVip } = useUserProfile()

  async function sendMessage(content: string) {
    if (!isVip) throw new BusinessError('需要 VIP 权限')
    const trimmedMessages = messages.slice(-100)  // 业务决策在 Hook 里
    return await aiApi.sendMessage(trimmedMessages)
  }
}
```

### 案例三：循环依赖

```typescript
// ❌ 错误：useAIChat 和 useTracking 互相依赖
// src/hooks/useAIChat.ts
import { useTracking } from './useTracking'

function useAIChat() {
  const { track } = useTracking()
  // 发消息时埋点
  async function sendMessage(content: string) {
    track('message_sent')
    // ...
  }
}

// src/hooks/useTracking.ts
import { useAIChat } from './useAIChat'  // 循环依赖！

function useTracking() {
  const { messages } = useAIChat()  // 埋点需要读取消息数量
  // ...
}
```

**问题：** A 依赖 B，B 依赖 A，形成循环。
无法单独测试任何一个，修改任何一个都可能影响另一个。

```typescript
// ✅ 正确：打破循环，通过事件或回调解耦
// src/hooks/useAIChat.ts
function useAIChat() {
  async function sendMessage(content: string, onSent?: () => void) {
    // ...
    onSent?.()  // 通过回调通知，不直接依赖 useTracking
  }
}

// src/pages/ChatPage.tsx（在页面层组合）
function ChatPage() {
  const { sendMessage } = useAIChat()
  const { track } = useTracking()

  const handleSend = (content: string) => {
    sendMessage(content, () => track('message_sent'))  // 在页面层组合
  }
}
```

### 案例四：共享可变状态

```typescript
// ❌ 错误：多个模块共享同一个全局变量
// src/globals.ts
export const globalCache: Record<string, unknown> = {}  // 全局缓存

// src/hooks/useAIChat.ts
import { globalCache } from '../globals'
globalCache['lastMessage'] = message  // 直接修改全局状态

// src/hooks/useUserProfile.ts
import { globalCache } from '../globals'
globalCache['userId'] = userId  // 也在修改同一个全局对象
```

**问题：** 任何模块都可以修改全局状态，边界形同虚设。

```typescript
// ✅ 正确：每个模块管理自己的状态（通过 Zustand Store 隔离）
// src/store/chatStore.ts
export const useChatStore = create<ChatStore>()(...)  // Chat 模块自己的 Store

// src/store/userStore.ts
export const useUserStore = create<UserStore>()(...)  // User 模块自己的 Store
```

## 五、边界划分的自检清单

完成边界划分后，用这个清单验证：

```
模块职责：
  □ 每个模块的职责是否可以用一句话描述？
  □ 是否有模块承担了两种以上不相关的职责？

公开接口：
  □ 每个模块的公开 Hook/函数是否最小化？
  □ 是否有内部实现（Store、工具函数）被意外暴露？

依赖关系：
  □ 是否有循环依赖？（可用 madge 工具检测）
  □ 是否有 Hook 直接访问另一个模块的内部 Store？

横切关注点：
  □ Token 管理、请求拦截、埋点是否集中在 Providers 层？
  □ 是否有模块直接初始化 AI SDK 而不通过 AIProvider？

文档和约束：
  □ 模块边界是否写进了 docs/architecture.md？
  □ 是否有 ESLint 规则检测禁止的跨模块依赖？
```

## 本节小结

```
模块边界的两个维度：
  纵向：分层架构（Types→Config→API→Store→Hooks→Components→Pages）
  横向：业务域划分（Chat/User/AIConfig/Tracking）

划定边界的步骤：
  1. 按业务域划分模块
  2. 定义每个模块的最小公开接口（Hook）
  3. 确定模块间依赖关系（画图，检查循环）
  4. 写进文档 + 编写 ESLint 规则

横切关注点（前端版）：
  鉴权 Token（AuthProvider）
  请求拦截（RequestProvider）
  埋点监控（TrackingProvider）
  主题国际化（ThemeProvider）
  AI SDK 封装（AIProvider）
  → 统一放在 Providers 层，通过 React Context 使用

四类典型违规：
  跨越模块边界直接访问内部 Store
  业务逻辑泄漏到 API 层
  循环依赖
  共享可变全局状态
```
