# 3.3　自定义 Linter 的设计

> 本节目标：理解 Harness Engineering 中 Linter 的特殊设计思路，掌握"错误信息即修复指令"的写法，并能为自己的项目编写第一批 Linter 规则。

---

## 一、普通 Linter vs Harness Linter

普通 Linter 的错误信息长这样：

```
ERROR: src/services/user_service.py:42
  Import from 'ui' layer is not allowed in 'service' layer.
```

这对人类工程师够用——他知道怎么修。

但对 Agent 来说，这个信息不够：
- 违反了什么规则？（知道了）
- 为什么不能这样？（不知道）
- 应该怎么修？（**不知道**）

Agent 读到这个错误，可能会：
- 删掉那行 import（治标不治本）
- 换一种方式绕过检查（更糟糕）
- 反复尝试，浪费大量 token

**Harness Linter 的设计原则：错误信息 = 修复指令。**

---

## 二、"错误信息即修复指令"的写法

Harness Linter 的错误信息需要包含三个部分：

```
1. 违反了什么规则（What）
2. 为什么不能这样（Why）
3. 应该怎么修（How）
```

对比示例：

```
普通 Linter 错误：
  ERROR: Import from 'ui' layer is not allowed in 'service' layer.

Harness Linter 错误：
  ERROR [ARCH-001]: Service 层不能依赖 UI 层。
  原因：依赖方向规则要求只能向下依赖（Types→Config→Repo→Service→Runtime→UI）。
  修复：将共享逻辑移到 Service 层或更低层，或通过依赖注入传入。
  参考：docs/architecture.md#层级约束
```

Agent 读到第二种错误信息，立刻知道：
1. 这是架构层级违规（ARCH-001）
2. 依赖方向只能向下
3. 具体的修复方向
4. 去哪里查详细规则

---

## 三、Linter 规则的分类

Harness 中的 Linter 规则通常分为三类：

### 类型一：架构层级规则

检查依赖方向是否正确。

```
规则：每一层只能 import 它下面的层
检查方式：分析 import 语句，判断被导入模块属于哪一层
错误代码：ARCH-001 到 ARCH-010
```

示例规则：
```
ARCH-001：禁止 Service 层 import UI 层
ARCH-002：禁止 Repo 层 import Service 层
ARCH-003：禁止 Types 层 import 任何其他层
ARCH-004：禁止循环依赖
```

### 类型二：编码规范规则

检查代码风格和约定。

```
规则示例：
  CONV-001：日志必须使用 structlog，禁止使用 print() 或 logging.info()
  CONV-002：异常必须被捕获并记录，禁止空的 except 块
  CONV-003：数据库操作必须在 Repo 层，禁止在 Service 层直接写 SQL
  CONV-004：API 响应必须使用统一的 Response 类型
```

### 类型三：质量规则

检查代码质量指标。

```
规则示例：
  QUAL-001：函数长度不超过 50 行
  QUAL-002：函数参数不超过 5 个
  QUAL-003：新增代码必须有对应的测试文件
  QUAL-004：禁止提交 TODO 注释（必须创建 Issue）
```

---

## 四、Linter 错误信息的模板

每条 Linter 规则的错误信息建议遵循这个模板：

```
[错误代码] 简短描述（一句话）
原因：为什么这样做是错的
修复：具体的修复步骤或方向
示例：（可选）正确写法的示例
参考：相关文档的链接
```

实际示例：

```
[CONV-003] Service 层不能直接执行 SQL 查询。
原因：数据库操作必须封装在 Repo 层，Service 层通过 Repo 接口访问数据，
      这样可以解耦业务逻辑和数据访问，便于测试和替换数据库。
修复：
  1. 在对应的 Repo 类中创建新方法封装这个查询
  2. 在 Service 层注入该 Repo，调用新方法
示例：
  # 错误
  class UserService:
      def get_active_users(self):
          return db.execute("SELECT * FROM users WHERE active=1")

  # 正确
  class UserService:
      def get_active_users(self):
          return self.userRepo.findActive()
参考：docs/architecture.md#repo层职责
```

## 五、用 Agent 编写 Linter

这是 Harness Engineering 里一个优雅的自举设计：

**用 Agent（Codex）来编写 Linter 规则本身。**

工作流程：

```
工程师发现一个新的违规模式
    ↓
工程师描述这个违规：
  "我需要一条 Linter 规则，检测 Service 层是否直接执行了 SQL 查询。
   错误信息要告诉 Agent：SQL 查询必须在 Repo 层，并给出修复步骤。"
    ↓
Agent 编写 Linter 规则代码
    ↓
工程师审查规则逻辑是否正确
    ↓
规则加入 CI，永久生效
```

好处：
- 工程师只需描述"检测什么"，不需要自己写 AST 解析代码
- Agent 擅长写这类模式匹配代码
- 大幅降低编写 Linter 规则的门槛

## 六、常见架构违规的 Linter 规则示例

下面是几条实用的规则，展示不同技术栈的写法思路：

### Python 项目（使用 pylint 或自定义检查器）

```python
# 检查 Service 层是否直接导入了数据库模块
# 规则：src/services/ 下的文件不能直接 import sqlalchemy 或 psycopg2

def check_service_db_import(file_path, imports):
    if 'services/' in file_path:
        forbidden = ['sqlalchemy', 'psycopg2', 'pymongo']
        for imp in imports:
            if any(f in imp for f in forbidden):
                return (
                    "[ARCH-002] Service 层不能直接导入数据库驱动。\n"
                    "原因：数据库操作必须封装在 Repo 层。\n"
                    "修复：在 repos/ 目录下创建对应的 Repo 类，"
                    "通过依赖注入在 Service 中使用。\n"
                    "参考：docs/architecture.md#分层规则"
                )
```

### TypeScript 项目（使用 ESLint 自定义规则）

```javascript
// 检查 services/ 目录下是否直接导入了 ui/ 目录的模块
module.exports = {
  create(context) {
    return {
      ImportDeclaration(node) {
        const filePath = context.getFilename();
        const importPath = node.source.value;

        if (filePath.includes('/services/') && importPath.includes('/ui/')) {
          context.report({
            node,
            message:
              '[ARCH-001] Service 层不能依赖 UI 层。\n' +
              '原因：依赖方向只能向下（Service → Repo → Types）。\n' +
              '修复：将共享逻辑移到 Service 层或更低层。\n' +
              '参考：docs/architecture.md#层级约束',
          });
        }
      },
    };
  },
};
```

## 七、Linter 规则的优先级设计

不是所有规则都应该阻断 CI，需要分级：

```
级别一：Error（阻断合并）
  违反架构层级规则
  违反安全规范
  → CI 不通过，PR 无法合并

级别二：Warning（提醒但不阻断）
  代码质量问题（函数过长、参数过多）
  轻微的风格不一致
  → CI 通过，但在 PR 评论中显示警告

级别三：Info（仅记录）
  建议性改进
  → 只在本地显示，不影响 CI
```

**原则：只把真正重要的规则设为 Error。** 规则太严会让 Agent 陷入无休止的修复循环，降低效率。

## 本节小结

```
Harness Linter 的核心设计：
  错误信息 = 修复指令
  包含：What（违反了什么）+ Why（为什么）+ How（怎么修）

三类规则：
  架构层级规则（ARCH-xxx）：检查依赖方向
  编码规范规则（CONV-xxx）：检查代码约定
  质量规则（QUAL-xxx）：检查代码质量指标

错误信息模板：
  [错误代码] 简短描述
  原因：...
  修复：...
  参考：...

用 Agent 写 Linter：
  工程师描述"检测什么" → Agent 写规则代码 → 工程师审查

规则优先级：
  Error（阻断）/ Warning（提醒）/ Info（记录）
  只把真正重要的设为 Error
```
