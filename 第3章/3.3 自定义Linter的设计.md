# 3.3　自定义 ESLint 规则的设计

> 本节目标：理解 Harness Engineering 中 ESLint 的特殊设计思路，掌握"错误信息即修复指令"的写法，并能为自己的前端项目编写第一批 ESLint 规则。

---

## 一、普通 ESLint vs Harness ESLint

普通 ESLint 的错误信息长这样：

```
ERROR: src/components/ChatMessage.tsx:15
  Import from 'api' layer is not allowed in 'components' layer.
```

这对人类工程师够用——他知道怎么修。

但对 Agent 来说，这个信息不够：
- 违反了什么规则？（知道了）
- 为什么不能这样？（不知道）
- 应该怎么修？（**不知道**）

Agent 读到这个错误，可能会：
- 删掉那行 import（治标不治本）
- 换一种方式绕过检查（更糟糕）
- 反复尝试，浪费大量 token

**Harness ESLint 的设计原则：错误信息 = 修复指令。**

---

## 二、"错误信息即修复指令"的写法

Harness ESLint 的错误信息需要包含三个部分：

```
1. 违反了什么规则（What）
2. 为什么不能这样（Why）
3. 应该怎么修（How）
```

对比示例：

```
普通 ESLint 错误：
  ERROR: Import from 'api' layer is not allowed in 'components' layer.

Harness ESLint 错误：
  ERROR [ARCH-001]: Components 层不能直接导入 API 层。
  原因：依赖方向规则要求只能向下依赖（Types→Config→API→Store→Hooks→Components→Pages）。
  修复：将数据获取逻辑移到 Hooks 层（创建 useXxx Hook），在 Component 中调用该 Hook。
  参考：docs/architecture.md#层级约束
```

Agent 读到第二种错误信息，立刻知道：
1. 这是架构层级违规（ARCH-001）
2. 依赖方向只能向下
3. 具体的修复方向（创建 Hook）
4. 去哪里查详细规则

---

## 三、ESLint 规则的分类

Harness 中的 ESLint 规则通常分为三类：

### 类型一：架构层级规则

检查依赖方向是否正确。

```
规则：每一层只能 import 它下面的层
检查方式：分析 import 语句，判断被导入模块属于哪一层
错误代码：ARCH-001 到 ARCH-010
```

示例规则：
```
ARCH-001：禁止 Components 层直接 import API 层
ARCH-002：禁止 Components 层直接 import Store 层（必须通过 Hooks）
ARCH-003：禁止 Pages 层直接 import Store 层（必须通过 Hooks）
ARCH-004：禁止 Types 层 import 任何其他层
ARCH-005：禁止循环依赖
```

### 类型二：编码规范规则

检查代码风格和约定。

```
规则示例：
  CONV-001：禁止在 src/components/ 下直接使用 fetch 或 axios
  CONV-002：禁止在 src/components/ 下直接导入 AI SDK（@ai-sdk/*、openai、anthropic）
  CONV-003：Hook 文件必须以 use 开头（useXxx.ts）
  CONV-004：API 响应必须使用 src/types/ 中定义的类型，禁止使用 any
```

### 类型三：质量规则

检查代码质量指标。

```
规则示例：
  QUAL-001：组件文件长度不超过 150 行
  QUAL-002：函数参数不超过 5 个
  QUAL-003：新增 Hook 必须有对应的测试文件
  QUAL-004：禁止提交 TODO 注释（必须创建 Issue）
```

## 四、ESLint 错误信息的模板

每条 ESLint 规则的错误信息建议遵循这个模板：

```
[错误代码] 简短描述（一句话）
原因：为什么这样做是错的
修复：具体的修复步骤或方向
示例：（可选）正确写法的示例
参考：相关文档的链接
```

实际示例：

```
[CONV-001] Components 层不能直接发起网络请求。
原因：网络请求必须封装在 API 层（src/api/），Components 层通过 Hooks 层访问数据，
      这样可以解耦 UI 和数据获取逻辑，便于测试和复用。
修复：
  1. 在 src/api/ 中创建对应的 API 函数封装请求
  2. 在 src/hooks/ 中创建 useXxx Hook 调用该 API 函数
  3. 在 Component 中调用该 Hook
示例：
  // 错误
  function ChatMessage({ id }) {
    useEffect(() => {
      fetch(`/api/messages/${id}`).then(...)
    }, [id])
  }

  // 正确
  function ChatMessage({ id }) {
    const { message } = useMessage(id)  // 通过 Hook 获取数据
  }
参考：docs/architecture.md#hooks层职责
```

## 五、用 Agent 编写 ESLint 规则

这是 Harness Engineering 里一个优雅的自举设计：

**用 Agent（Codex）来编写 ESLint 规则本身。**

工作流程：

```
工程师发现一个新的违规模式
    ↓
工程师描述这个违规：
  "我需要一条 ESLint 规则，检测 src/components/ 目录下是否直接导入了
   src/api/ 目录的模块。错误信息要告诉 Agent：数据获取必须通过 Hooks 层，
   并给出修复步骤。"
    ↓
Agent 编写 ESLint 规则代码
    ↓
工程师审查规则逻辑是否正确
    ↓
规则加入 CI，永久生效
```

好处：
- 工程师只需描述"检测什么"，不需要自己写 AST 解析代码
- Agent 擅长写这类模式匹配代码
- 大幅降低编写 ESLint 规则的门槛

## 六、常见架构违规的 ESLint 规则示例

### 规则一：禁止 Components 层直接导入 API 层

```javascript
// eslint-rules/no-api-in-components.js
module.exports = {
  meta: {
    type: 'problem',
    docs: {
      description: 'Components 层不能直接导入 API 层',
    },
    messages: {
      noApiInComponents:
        '[ARCH-001] Components 层不能直接导入 API 层。\n' +
        '原因：依赖方向规则要求 Components 只能依赖 Hooks 层及以下。\n' +
        '修复：将数据获取逻辑移到 src/hooks/ 中的 useXxx Hook，在 Component 中调用该 Hook。\n' +
        '参考：docs/architecture.md#层级约束',
    },
  },
  create(context) {
    return {
      ImportDeclaration(node) {
        const filePath = context.getFilename()
        const importPath = node.source.value

        const isInComponents = filePath.includes('/components/')
        const importsFromApi =
          importPath.includes('/api/') || importPath.startsWith('../api')

        if (isInComponents && importsFromApi) {
          context.report({ node, messageId: 'noApiInComponents' })
        }
      },
    }
  },
}
```

### 规则二：禁止在组件中直接使用 AI SDK

```javascript
// eslint-rules/no-ai-sdk-in-components.js
module.exports = {
  meta: {
    type: 'problem',
    messages: {
      noAiSdkInComponents:
        '[CONV-002] Components 层不能直接导入 AI SDK。\n' +
        '原因：AI SDK 调用必须封装在 src/api/aiApi.ts 中，通过 AIProvider 统一管理。\n' +
        '修复：\n' +
        '  1. 在 src/api/aiApi.ts 中添加对应的 AI 调用函数\n' +
        '  2. 在 src/hooks/ 中创建 useAIXxx Hook\n' +
        '  3. 在 Component 中使用该 Hook\n' +
        '参考：docs/architecture.md#ai-provider',
    },
  },
  create(context) {
    const AI_SDK_PACKAGES = ['openai', 'anthropic', '@ai-sdk/', '@anthropic-ai/']

    return {
      ImportDeclaration(node) {
        const filePath = context.getFilename()
        const importPath = node.source.value

        const isInComponents = filePath.includes('/components/')
        const importsAiSdk = AI_SDK_PACKAGES.some(pkg => importPath.startsWith(pkg))

        if (isInComponents && importsAiSdk) {
          context.report({ node, messageId: 'noAiSdkInComponents' })
        }
      },
    }
  },
}
```

### 规则三：禁止 Pages 层直接操作 Store

```javascript
// eslint-rules/no-store-in-pages.js
module.exports = {
  meta: {
    type: 'problem',
    messages: {
      noStoreInPages:
        '[ARCH-003] Pages 层不能直接导入 Store 层。\n' +
        '原因：Pages 层只能依赖 Components 和 Hooks 层，Store 操作必须封装在 Hooks 中。\n' +
        '修复：将 Store 操作移到 src/hooks/ 中的 useXxx Hook，在 Page 中调用该 Hook。\n' +
        '参考：docs/architecture.md#层级约束',
    },
  },
  create(context) {
    return {
      ImportDeclaration(node) {
        const filePath = context.getFilename()
        const importPath = node.source.value

        const isInPages = filePath.includes('/pages/')
        const importsFromStore =
          importPath.includes('/store/') || importPath.includes('Store')

        if (isInPages && importsFromStore) {
          context.report({ node, messageId: 'noStoreInPages' })
        }
      },
    }
  },
}
```

## 七、ESLint 规则的优先级设计

不是所有规则都应该阻断 CI，需要分级：

```
级别一：error（阻断合并）
  违反架构层级规则（ARCH-xxx）
  直接使用 AI SDK（CONV-002）
  → CI 不通过，PR 无法合并

级别二：warn（提醒但不阻断）
  代码质量问题（组件过长、参数过多）
  轻微的风格不一致
  → CI 通过，但在 PR 评论中显示警告

级别三：off（仅本地提示）
  建议性改进
  → 只在编辑器里显示，不影响 CI
```

**原则：只把真正重要的规则设为 error。** 规则太严会让 Agent 陷入无休止的修复循环，降低效率。

## 八、在 ESLint 配置中注册自定义规则

```javascript
// eslint.config.js（ESLint v9 flat config）
import noApiInComponents from './eslint-rules/no-api-in-components.js'
import noAiSdkInComponents from './eslint-rules/no-ai-sdk-in-components.js'
import noStoreInPages from './eslint-rules/no-store-in-pages.js'

export default [
  {
    plugins: {
      'harness': {
        rules: {
          'no-api-in-components': noApiInComponents,
          'no-ai-sdk-in-components': noAiSdkInComponents,
          'no-store-in-pages': noStoreInPages,
        },
      },
    },
    rules: {
      'harness/no-api-in-components': 'error',
      'harness/no-ai-sdk-in-components': 'error',
      'harness/no-store-in-pages': 'error',
    },
  },
]
```

## 本节小结

```
Harness ESLint 的核心设计：
  错误信息 = 修复指令
  包含：What（违反了什么）+ Why（为什么）+ How（怎么修）

三类规则：
  架构层级规则（ARCH-xxx）：检查依赖方向
  编码规范规则（CONV-xxx）：检查代码约定
  质量规则（QUAL-xxx）：检查代码质量指标

错误信息模板：
  [错误代码] 简短描述
  原因：...
  修复：...
  参考：...

用 Agent 写 ESLint 规则：
  工程师描述"检测什么" → Agent 写规则代码 → 工程师审查

规则优先级：
  error（阻断）/ warn（提醒）/ off（本地）
  只把真正重要的设为 error
```
