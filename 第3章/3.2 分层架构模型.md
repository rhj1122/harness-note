# 3.2　分层架构模型

> 本节目标：理解前端项目的分层架构模型，掌握依赖方向规则，并能为自己的前端项目设计合适的分层架构。

---

## 一、前端项目的七层模型

针对 AI 内嵌 H5 应用这类前端项目，我们使用以下七层架构：

```
┌─────────────────────────────────────────┐
│                  Pages                  │  ← 页面层
├─────────────────────────────────────────┤
│               Components               │  ← 组件层
├─────────────────────────────────────────┤
│            Hooks / Composables          │  ← 业务逻辑层
├─────────────────────────────────────────┤
│              Store / State              │  ← 状态管理层
├─────────────────────────────────────────┤
│                   API                   │  ← 接口层
├─────────────────────────────────────────┤
│                  Config                 │  ← 配置层
├─────────────────────────────────────────┤
│                  Types                  │  ← 类型定义层
└─────────────────────────────────────────┘

依赖方向：只能向下，不能向上
Types ← Config ← API ← Store ← Hooks ← Components ← Pages
```

**依赖方向规则：每一层只能依赖它下面的层，不能依赖上面的层。**

## 二、每一层的职责

### Types 层（最底层）

```
职责：定义纯 TypeScript 类型
特征：
  - 只有接口和类型定义（interface、type）
  - 无副作用（不调用任何外部服务）
  - 无业务逻辑
  - 可以被任何层依赖

示例：
  interface Message { id: string; role: 'user' | 'assistant'; content: string }
  interface AIConfig { model: string; temperature: number }
  type ApiError = { code: number; message: string }
```

### Config 层

```
职责：加载和管理配置
特征：
  - 读取环境变量（import.meta.env）
  - 只依赖 Types 层
  - 不包含业务逻辑

示例：
  const appConfig = {
    apiBaseUrl: import.meta.env.VITE_API_BASE_URL,
    aiModel: import.meta.env.VITE_AI_MODEL ?? 'gpt-4o',
  }
```

### API 层（接口层）

```
职责：封装所有网络请求
特征：
  - 封装 HTTP 请求和 AI SDK 调用
  - 只依赖 Types、Config 层
  - 不包含业务逻辑（只做请求/响应转换）
  - Hooks 层通过 API 层访问网络，不直接 fetch

示例：
  // src/api/aiApi.ts
  export async function sendMessage(messages: Message[]): Promise<Message>
  export async function* streamMessage(messages: Message[]): AsyncGenerator<string>

  // src/api/userApi.ts
  export async function getUserInfo(token: string): Promise<UserInfo>
```

### Store 层（状态管理层）

```
职责：管理全局状态
特征：
  - 使用 Zustand / Pinia 等状态管理库
  - 只依赖 API、Types 层
  - 不包含 UI 逻辑
  - 存储跨组件共享的状态

示例：
  // src/store/chatStore.ts（Zustand）
  interface ChatStore {
    messages: Message[]
    isLoading: boolean
    addMessage: (msg: Message) => void
    clearMessages: () => void
  }
```

### Hooks 层（业务逻辑层）

```
职责：封装业务逻辑，连接 Store 和 API
特征：
  - React Hooks（useXxx）或 Vue Composables（useXxx）
  - 只依赖 Store、API、Types 层
  - 不包含 JSX/模板（不渲染 UI）
  - Components 层通过 Hooks 层访问数据和逻辑

示例：
  // src/hooks/useAIChat.ts
  export function useAIChat() {
    const { messages, addMessage } = useChatStore()
    const [isStreaming, setIsStreaming] = useState(false)

    async function sendMessage(content: string) {
      // 调用 API 层，更新 Store
      const stream = streamMessage([...messages, { role: 'user', content }])
      // 处理流式响应...
    }

    return { messages, isStreaming, sendMessage }
  }
```

### Components 层（组件层）

```
职责：UI 组件
特征：
  - 纯展示组件 + 交互处理
  - 只依赖 Hooks、Types 层
  - 不直接调用 API 或操作 Store
  - 不包含业务逻辑（业务逻辑在 Hooks 层）

示例：
  // src/components/ChatMessage.tsx
  function ChatMessage({ message }: { message: Message }) {
    return <div className={...}>{message.content}</div>
  }

  // src/components/ChatInput.tsx
  function ChatInput({ onSend }: { onSend: (text: string) => void }) {
    // 只处理 UI 交互，业务逻辑通过 props 传入
  }
```

### Pages 层（最顶层）

```
职责：页面组合，路由入口
特征：
  - 组合 Components 和 Hooks，构成完整页面
  - 只依赖 Components、Hooks 层
  - 不包含业务逻辑
  - 处理路由参数

示例：
  // src/pages/ChatPage.tsx
  function ChatPage() {
    const { messages, isStreaming, sendMessage } = useAIChat()
    return (
      <div>
        <MessageList messages={messages} />
        <ChatInput onSend={sendMessage} disabled={isStreaming} />
      </div>
    )
  }
```

## 三、Providers：横切关注点模式

有些功能不属于任何单一层，而是被多个层共同使用，比如：

- 鉴权 Token 管理（AuthProvider）
- 请求拦截器（RequestProvider）
- 埋点/监控（TrackingProvider）
- 主题/国际化（ThemeProvider）
- AI SDK 封装（AIProvider）

这类功能叫**横切关注点（Cross-cutting Concerns）**，用 **Providers** 模式处理：

```
┌─────────────────────────────────────────┐
│                  Pages                  │
├─────────────────────────────────────────┤
│               Components               │
├─────────────────────────────────────────┤  ┌──────────────────┐
│            Hooks / Composables          │  │    Providers     │
├─────────────────────────────────────────┤  │                  │
│              Store / State              │  │ - AuthProvider   │
├─────────────────────────────────────────┤  │ - RequestProvider│
│                   API                   │  │ - TrackingProvider│
├─────────────────────────────────────────┤  │ - ThemeProvider  │
│                  Config                 │  │ - AIProvider     │
├─────────────────────────────────────────┤  └──────────────────┘
│                  Types                  │
└─────────────────────────────────────────┘
```

Providers 的规则：
- 每个 Provider 只做一件事（单一职责）
- Provider 可以被任何层使用（通过 React Context 或依赖注入）
- Provider 的实现细节对使用方透明

```
示例：AIProvider

Hooks 层调用：
  const { streamMessage } = useAIProvider()
  → 不关心底层用的是 OpenAI 还是 Anthropic 还是本地模型

好处：
  - 换 AI 服务商只需改 AIProvider 的实现
  - Hooks 层代码不需要改动
  - Agent 写新功能时，直接用 useAIProvider()，不会引入新的 AI SDK 依赖
```

## 四、依赖方向违规的典型案例

理解了分层，再看几个典型的违规案例，加深印象：

### 违规一：Component 层直接调用 API

```typescript
// ❌ 错误：Component 层直接发网络请求
function ChatMessage({ messageId }: { messageId: string }) {
  const [detail, setDetail] = useState(null)

  useEffect(() => {
    // 直接在组件里 fetch，违反分层规则
    fetch(`/api/messages/${messageId}`)
      .then(r => r.json())
      .then(setDetail)
  }, [messageId])

  return <div>{detail?.content}</div>
}

// ✅ 正确：通过 Hook 访问数据
function ChatMessage({ messageId }: { messageId: string }) {
  const { message } = useMessage(messageId)  // Hook 层处理数据获取
  return <div>{message?.content}</div>
}
```

### 违规二：Hook 层直接渲染 JSX

```typescript
// ❌ 错误：Hook 层包含 UI 逻辑
function useAIChat() {
  const [messages, setMessages] = useState<Message[]>([])

  // Hook 不应该返回 JSX！
  const MessageList = () => (
    <ul>{messages.map(m => <li key={m.id}>{m.content}</li>)}</ul>
  )

  return { messages, MessageList }
}

// ✅ 正确：Hook 只返回数据和方法
function useAIChat() {
  const [messages, setMessages] = useState<Message[]>([])
  const sendMessage = async (content: string) => { /* ... */ }
  return { messages, sendMessage }
}
```

### 违规三：Types 层依赖 Config 层（反向依赖）

```typescript
// ❌ 错误：Types 层引用了 Config
import { appConfig } from '../config/appConfig'  // Types 不能依赖 Config

interface AIConfig {
  model: string
  maxTokens: number
  defaultModel: string  // 从 appConfig 读取默认值，违规！
}

// ✅ 正确：常量直接定义在 Types 层
interface AIConfig {
  model: string
  maxTokens: number
}
const DEFAULT_MODEL = 'gpt-4o'  // 常量定义在 Types 层
```

### 违规四：Page 层直接操作 Store

```typescript
// ❌ 错误：Page 层绕过 Hooks 直接操作 Store
function ChatPage() {
  const addMessage = useChatStore(s => s.addMessage)  // 直接访问 Store

  const handleSend = (text: string) => {
    addMessage({ role: 'user', content: text })  // 直接操作 Store，违规
  }
  // ...
}

// ✅ 正确：Page 层通过 Hooks 层操作
function ChatPage() {
  const { sendMessage } = useAIChat()  // 通过 Hook 层

  const handleSend = (text: string) => {
    sendMessage(text)  // Hook 内部处理 Store 操作
  }
  // ...
}
```

## 五、为自己的项目设计分层架构

七层模型是针对 AI H5 应用的参考，不是必须照搬的标准。

### 设计原则

```
原则一：每层职责单一
  每层只做一件事，不越界

原则二：依赖方向单向
  只能向下依赖，不能向上，不能循环

原则三：层数适中
  太少：职责混乱
  太多：过度工程化
  通常 4-6 层适合大多数前端项目
```

### 常见的简化版本

**小型 H5 项目（4层）：**
```
Pages               ← 页面和路由
Components          ← UI 组件
Hooks               ← 业务逻辑 + 状态
API / Types         ← 接口和类型
```

**中型前端项目（5层）：**
```
Pages               ← 页面
Components          ← 组件
Hooks               ← 业务逻辑
Store + API         ← 状态 + 接口
Types + Config      ← 类型 + 配置
```

### 设计步骤

```
步骤一：列出代码的主要职责类型
  （类型定义 / 配置 / 网络请求 / 状态管理 / 业务逻辑 / UI 组件 / 页面）

步骤二：把职责分组，每组对应一层

步骤三：确定依赖方向
  画出箭头：哪层可以调用哪层

步骤四：写进 docs/architecture.md
  用文字 + ASCII 图描述清楚

步骤五：在 AGENTS.md 里写关键规则摘要
  （详细内容指向 docs/architecture.md）
```

## 本节小结

```
前端七层模型：
  Types → Config → API → Store → Hooks → Components → Pages
  依赖方向：只能向下，不能向上

各层职责：
  Types：纯 TypeScript 类型定义
  Config：环境配置加载
  API：网络请求封装（HTTP + AI SDK）
  Store：全局状态管理（Zustand 等）
  Hooks：业务逻辑（useXxx）
  Components：UI 组件（纯展示 + 交互）
  Pages：页面组合（路由入口）

Providers 模式：
  处理横切关注点（鉴权、请求拦截、埋点、主题、AI SDK）
  通过 React Context 被各层使用

为自己的项目设计：
  不必照搬七层，4-6 层适合大多数前端项目
  核心原则：职责单一 + 依赖单向
```
