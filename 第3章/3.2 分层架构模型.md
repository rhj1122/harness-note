# 3.2　分层架构模型

> 本节目标：理解 OpenAI 内部的六层架构模型，掌握依赖方向规则，并能为自己的项目设计合适的分层架构。

---

## 一、OpenAI 内部的六层模型

OpenAI 在构建 100 万行代码项目时，使用了一套六层架构：

```
┌─────────────────────────────────────────┐
│                   UI                    │  ← 用户界面层
├─────────────────────────────────────────┤
│                 Runtime                 │  ← 应用运行层
├─────────────────────────────────────────┤
│                 Service                 │  ← 业务逻辑层
├─────────────────────────────────────────┤
│                  Repo                   │  ← 数据访问层
├─────────────────────────────────────────┤
│                 Config                  │  ← 配置层
├─────────────────────────────────────────┤
│                  Types                  │  ← 数据类型层
└─────────────────────────────────────────┘

依赖方向：只能向下，不能向上
Types ← Config ← Repo ← Service ← Runtime ← UI
```

**依赖方向规则：每一层只能依赖它下面的层，不能依赖上面的层。**

## 二、每一层的职责

### Types 层（最底层）

```
职责：定义纯数据类型
特征：
  - 只有数据结构（struct、class、interface）
  - 无副作用（不调用任何外部服务）
  - 无业务逻辑
  - 可以被任何层依赖

示例：
  User { id, name, email }
  Order { id, userId, items, status }
  ApiError { code, message }
```

### Config 层

```
职责：加载和管理配置
特征：
  - 读取环境变量、配置文件
  - 只依赖 Types 层
  - 不包含业务逻辑

示例：
  DatabaseConfig { host, port, name }
  AppConfig { port, debug, logLevel }
  loadConfig() → AppConfig
```

### Repo 层（Repository，数据仓库层）

```
职责：数据的读写操作
特征：
  - 封装所有数据库操作
  - 只依赖 Types、Config 层
  - 不包含业务逻辑（只做 CRUD）
  - Service 层通过 Repo 层访问数据，不直接操作数据库

示例：
  UserRepo.findById(id) → User
  UserRepo.create(user) → User
  OrderRepo.findByUserId(userId) → List[Order]
```

### Service 层

```
职责：业务逻辑
特征：
  - 实现具体的业务规则
  - 只依赖 Types、Config、Repo 层
  - 不直接操作数据库（通过 Repo）
  - 不处理 HTTP 请求/响应（那是 Runtime 的事）

示例：
  UserService.register(email, password) → User
    （包含：验证邮箱格式、检查是否已注册、加密密码、调用 UserRepo.create）
  OrderService.placeOrder(userId, items) → Order
    （包含：检查库存、计算价格、调用 OrderRepo.create、触发通知）
```

### Runtime 层

```
职责：应用入口，连接外部世界与内部逻辑
特征：
  - 处理 HTTP 请求/响应（路由、中间件）
  - 调用 Service 层处理业务
  - 只依赖 Service 及以下各层
  - 不包含业务逻辑

示例：
  POST /users/register
    → 解析请求体
    → 调用 UserService.register()
    → 返回响应
```

### UI 层（最顶层）

```
职责：用户界面
特征：
  - 展示数据，处理用户交互
  - 只依赖 Runtime 及以下各层
  - 不包含业务逻辑

示例：
  RegisterForm 组件
    → 调用 API（Runtime 层暴露的接口）
    → 展示结果
```

## 三、Providers：横切关注点模式

有些功能不属于任何单一层，而是被多个层共同使用，比如：

- 鉴权（Authentication）
- 外部服务连接器（邮件、短信、支付）
- 遥测/监控（Telemetry）
- 特性开关（Feature Flags）

这类功能叫**横切关注点（Cross-cutting Concerns）**，用 **Providers** 模式处理：

```
┌─────────────────────────────────────────┐
│                   UI                    │
├─────────────────────────────────────────┤
│                 Runtime                 │
├─────────────────────────────────────────┤  ┌──────────────┐
│                 Service                 │  │  Providers   │
├─────────────────────────────────────────┤  │              │
│                  Repo                   │  │ - AuthProvider│
├─────────────────────────────────────────┤  │ - EmailProvider│
│                 Config                  │  │ - TelemetryProvider│
├─────────────────────────────────────────┤  │ - FeatureFlags│
│                  Types                  │  └──────────────┘
└─────────────────────────────────────────┘
```

Providers 的规则：
- 每个 Provider 只做一件事（单一职责）
- Provider 可以被任何层使用（通过依赖注入）
- Provider 的实现细节对使用方透明

```
示例：EmailProvider

Service 层调用：
  emailProvider.send(to, subject, body)
  → 不关心底层用的是 SendGrid 还是 SES

好处：
  - 换邮件服务商只需改 EmailProvider 的实现
  - Service 层代码不需要改动
  - Agent 写新功能时，直接用 emailProvider，不会引入新的邮件库
```

## 四、依赖方向违规的典型案例

理解了分层，再看几个典型的违规案例，加深印象：

### 违规一：Repo 层包含业务逻辑

```python
# ❌ 错误：Repo 层里有业务判断
class OrderRepo:
    def create(self, order):
        # 这是业务逻辑，不应该在 Repo 层
        if order.total > 10000:
            send_alert_email()  # 还调用了邮件服务！
        db.insert(order)

# ✅ 正确：Repo 层只做数据操作
class OrderRepo:
    def create(self, order):
        db.insert(order)
        return order
```

### 违规二：Service 层直接操作数据库

```python
# ❌ 错误：Service 层绕过 Repo 直接写 SQL
class UserService:
    def register(self, email, password):
        db.execute("INSERT INTO users VALUES (...)")  # 直接 SQL

# ✅ 正确：Service 层通过 Repo 访问数据
class UserService:
    def register(self, email, password):
        user = User(email=email, password=hash(password))
        return self.userRepo.create(user)
```

### 违规三：Types 层依赖 Config 层（反向依赖）

```python
# ❌ 错误：Types 层引用了 Config
from config import AppConfig  # Types 不能依赖 Config

class User:
    max_name_length = AppConfig.MAX_NAME_LENGTH  # 违规！

# ✅ 正确：常量直接定义在 Types 层
class User:
    MAX_NAME_LENGTH = 100
```

## 五、为自己的项目设计分层架构

OpenAI 的六层模型是一个参考，不是必须照搬的标准。

### 设计原则

```
原则一：每层职责单一
  每层只做一件事，不越界

原则二：依赖方向单向
  只能向下依赖，不能向上，不能循环

原则三：层数适中
  太少：职责混乱
  太多：过度工程化
  通常 3-5 层适合大多数项目
```

### 常见的简化版本

**Web API 项目（3层）：**
```
Routes/Controllers  ← 处理 HTTP 请求
Services            ← 业务逻辑
Repositories        ← 数据访问
```

**前端项目（3层）：**
```
Pages/Views         ← 页面和路由
Components          ← UI 组件
Services/Stores     ← 数据和状态管理
```

**全栈项目（4层）：**
```
UI                  ← 前端界面
API/Runtime         ← 接口层
Services            ← 业务逻辑
Data/Repo           ← 数据访问
```

### 设计步骤

```
步骤一：列出代码的主要职责类型
  （数据类型定义 / 配置 / 数据访问 / 业务逻辑 / 接口处理 / 界面）

步骤二：把职责分组，每组对应一层

步骤三：确定依赖方向
  画出箭头：哪层可以调用哪层

步骤四：写进 docs/architecture.md
  用文字 + ASCII 图描述清楚

步骤五：在 AGENTS.md 里写关键规则摘要
  （详细内容指向 docs/architecture.md）
```

## 本节小结

```
OpenAI 六层模型：
  Types → Config → Repo → Service → Runtime → UI
  依赖方向：只能向下，不能向上

各层职责：
  Types：纯数据类型
  Config：配置加载
  Repo：数据读写（只做 CRUD）
  Service：业务逻辑
  Runtime：应用入口（HTTP 路由等）
  UI：用户界面

Providers 模式：
  处理横切关注点（鉴权、邮件、监控等）
  通过依赖注入被各层使用

为自己的项目设计：
  不必照搬六层，3-5 层适合大多数项目
  核心原则：职责单一 + 依赖单向
```
