# 3.1　为什么需要架构约束

> 本节目标：理解架构约束的必要性——为什么光靠文档说明不够，必须用机械手段来强制执行规则。

---

## 一、Agent 是"模式复制器"

理解架构约束，先要理解 Agent 的一个本质特征：

**Agent 会复制它看到的模式。**

不管是好模式还是坏模式，只要代码库里存在，Agent 就会学习并复制它。

```
代码库里有一个好的模式：
  EmailProvider 正确地封装了邮件发送逻辑
  → Agent 写新功能时，会参考这个模式，创建类似的 Provider
  → 好模式扩散 ✅

代码库里有一个坏的模式：
  某个 Controller 直接写了 SQL 查询（违反分层规则）
  → Agent 写新功能时，看到这个先例，认为"这样写是可以的"
  → 坏模式扩散 ❌
```

这意味着：**代码库里的第一个违规，会成为后续所有违规的"许可证"。**

---

## 二、没有约束时代码库会发生什么

来看一个真实的演化过程：

### 第 1 个月：一切正常

```
项目刚开始，架构清晰：
  Controllers → Services → Repos → Database
  每层职责明确，依赖方向正确
```

### 第 2 个月：第一个"小小的"违规

```
Agent 在赶一个紧急功能，
在 Controller 里直接调用了数据库查询（跳过了 Service 和 Repo 层）。

工程师审查时没注意，合并了。
```

### 第 3 个月：违规开始扩散

```
另一个 Agent 看到了那个 Controller 里的直接数据库调用，
认为"这是项目里的正常做法"，
在新功能里也这样写了。

现在有 5 个地方直接操作数据库。
```

### 第 6 个月：依赖混乱

```
代码库里出现了：
  - Controller 直接调用数据库
  - Service 直接调用 UI 组件
  - Repo 层包含业务逻辑
  - 循环依赖（A 依赖 B，B 依赖 A）

没有人知道改哪里会影响哪里。
每次修改都可能触发连锁 bug。
```

这就是**依赖混乱**和**耦合爆炸**的样子。

## 三、"文字说明"为什么不够

你可能会想：在 AGENTS.md 里写清楚规则不就行了？

```markdown
## 层级规则（必须遵守）
依赖方向：Types → Config → Repo → Service → Runtime → UI
- 禁止反向依赖
```

这确实有用，但有三个根本性的局限：

### 局限一：Agent 可能没读到

上下文窗口有限，任务复杂时，Agent 可能在执行过程中"忘记"了 AGENTS.md 里的规则。

### 局限二：Agent 可能误判

规则是文字描述的，Agent 需要理解并判断"这段代码是否违反了规则"。
判断本身就可能出错，尤其是边界情况。

### 局限三：规则无法自动执行

文字规则依赖 Agent 的"自觉"，没有强制执行机制。
一旦 Agent 违规，只有人工审查才能发现。

```
文字规则的执行链：
  规则写在文档里
    → Agent 读到（可能没读到）
    → Agent 理解（可能理解错）
    → Agent 遵守（可能忘记）
    → 人工审查发现违规（可能漏掉）

机械约束的执行链：
  规则编码进 Linter
    → 代码提交时自动扫描
    → 违规立即报错（100% 触发）
    → Agent 读到错误信息，自动修复
```

**机械约束不依赖 Agent 的理解和记忆，它是物理上的强制执行。**

---

## 四、"机械约束"的核心思想

机械约束的设计哲学来自一个简单的问题：

> "如何让违规代码在物理上无法进入代码库？"

答案是两道防线：

```
第一道防线：Linter（本地实时检查）
  Agent 写代码时，Linter 实时扫描
  发现违规 → 立即报错 + 给出修复指令
  Agent 读到错误 → 自动修复
  → 大多数违规在本地就被消灭

第二道防线：CI（提交时强制检查）
  代码提交 PR 时，CI 自动运行
  Linter 不通过 → PR 无法合并
  → 即使本地没修复，CI 也会拦截
```

这两道防线合在一起，让违规代码**物理上无法进入主分支**。

---

## 五、架构约束 vs 上下文工程的分工

现在可以清楚地看到两个支柱的分工：

```
上下文工程（第二章）：
  作用：让 Agent 理解规则，主动遵守
  工具：AGENTS.md、docs/、skills/
  特点：依赖 Agent 的理解和意愿
  类比：给员工培训，告诉他规则是什么

架构约束（第三章）：
  作用：让违规代码无法通过，强制执行
  工具：Linter、CI 流水线
  特点：不依赖 Agent，机械执行
  类比：在门口装金属探测器，不管员工懂不懂规则，违禁品就是进不来
```

两者互补，缺一不可：
- 只有上下文工程：Agent 理解规则，但偶尔会犯错，没有兜底
- 只有架构约束：Agent 不理解为什么，会反复触发 Linter，效率低
- 两者结合：Agent 主动遵守（减少触发），Linter 兜底（消灭漏网之鱼）

## 本节小结

```
Agent 是模式复制器：
  好模式和坏模式都会被复制
  第一个违规 = 后续所有违规的许可证

没有约束的演化路径：
  第一个违规 → 扩散 → 依赖混乱 → 耦合爆炸 → 无法维护

文字规则的三个局限：
  Agent 可能没读到 / 理解错 / 忘记
  没有强制执行机制

机械约束的两道防线：
  Linter（本地实时）→ 发现违规立即报错+给修复指令
  CI（提交时强制）→ 不通过则无法合并

与上下文工程的分工：
  上下文工程：让 Agent 主动遵守（培训）
  架构约束：让违规无法通过（门禁）
```
