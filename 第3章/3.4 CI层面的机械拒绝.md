# 3.4　CI 层面的机械拒绝

> 本节目标：理解 CI 在 Harness 中的角色，掌握如何设计一条让违规代码物理上无法合并的 CI 流水线。

---

## 一、CI 是 Harness 的最后一道防线

上一节讲了 Linter——它在 Agent 本地工作时实时检查，发现违规立即报错。

但 Linter 有一个漏洞：**Agent 可以忽略它。**

如果 Agent 没有正确配置本地 Linter，或者某种原因跳过了检查，违规代码仍然可以被提交。

CI（持续集成）解决这个问题：

```
Linter（本地）：
  Agent 工作时实时检查
  发现违规 → 报错 → Agent 修复
  依赖：Agent 的本地环境配置正确

CI（服务端）：
  代码提交 PR 时自动触发
  不通过 → PR 物理上无法合并
  不依赖：Agent 的本地环境
  → 这是真正的"机械拒绝"
```

**无论 Agent 的本地环境如何，CI 都会在服务端执行同样的检查。**

---

## 二、Harness CI 流水线的结构

一条完整的 Harness CI 流水线通常包含以下阶段：

```
PR 提交
    ↓
┌─────────────────────────────────────┐
│  阶段一：快速检查（约 30 秒）          │
│  - 代码格式检查（formatter）          │
│  - 基础 Linter（语法、风格）          │
│  → 不通过：立即失败，不继续            │
└─────────────────────────────────────┘
    ↓（通过）
┌─────────────────────────────────────┐
│  阶段二：架构约束检查（约 1 分钟）     │
│  - 自定义架构 Linter（层级规则）      │
│  - 依赖分析（循环依赖检测）           │
│  - 禁用模式检测                      │
│  → 不通过：失败，报告具体违规         │
└─────────────────────────────────────┘
    ↓（通过）
┌─────────────────────────────────────┐
│  阶段三：测试（约 2-5 分钟）          │
│  - 单元测试                          │
│  - 集成测试                          │
│  - 测试覆盖率检查                    │
│  → 不通过：失败，报告失败的测试       │
└─────────────────────────────────────┘
    ↓（通过）
┌─────────────────────────────────────┐
│  阶段四：质量检查（可选）             │
│  - 代码复杂度分析                    │
│  - 安全漏洞扫描                      │
│  → 不通过：Warning（不阻断合并）      │
└─────────────────────────────────────┘
    ↓（通过）
PR 可以合并 ✅
```

**关键设计：阶段越靠前，检查越快。** 快速失败，避免浪费时间。

## 三、检查项分级：阻断 vs 警告

不是所有检查都应该阻断合并，需要明确分级：

```
阻断合并（必须修复才能合并）：
  ✅ 架构层级违规（ARCH-xxx）
  ✅ 测试失败
  ✅ 测试覆盖率低于阈值（如 80%）
  ✅ 安全漏洞（高危级别）
  ✅ 代码格式错误（formatter 不通过）

警告（显示但不阻断）：
  ⚠️ 代码质量问题（函数过长、复杂度高）
  ⚠️ 轻微的编码规范违反
  ⚠️ 中低危安全提示

仅记录（不影响 PR）：
  📝 性能分析报告
  📝 依赖更新提示
```

**原则：阻断项要少而精。** 阻断项太多，Agent 会陷入无休止的修复循环，效率崩溃。

## 四、CI 错误信息的设计

和 Linter 一样，CI 的错误信息也要对 Agent 友好。

Agent 在 CI 失败后会读取错误日志，决定如何修复。

```
差的 CI 错误输出：
  ❌ FAILED: arch-check
  Exit code: 1

好的 CI 错误输出：
  ❌ [ARCH-002] 架构层级违规
  文件：src/services/user_service.py，第 42 行
  问题：Service 层直接导入了数据库驱动 (psycopg2)
  原因：数据库操作必须封装在 Repo 层
  修复：
    1. 在 src/repos/user_repo.py 中创建对应的查询方法
    2. 在 UserService 中注入 UserRepo，调用该方法
  参考：docs/architecture.md#repo层职责
```

Agent 读到第二种输出，可以直接开始修复，不需要人介入。

## 五、PR 审查流程与自动合并策略

Harness Engineering 的目标是让大多数 PR **不需要人工深度审查**。

### 可以自动合并的 PR

```
满足以下所有条件时，PR 可以配置为自动合并：
  ✅ 所有 CI 检查通过
  ✅ PR 规模小（改动文件数 < 10，行数 < 200）
  ✅ PR 类型是已知安全类型：
      - 文档更新
      - 测试补充
      - 依赖版本升级（非 major 版本）
      - 自动生成的重构 PR（来自清理 Agent）
```

### 需要人工审查的 PR

```
以下情况需要人工审查：
  ⚠️ 涉及架构变更（新增层级、修改依赖规则）
  ⚠️ 涉及安全相关代码（鉴权、加密、权限）
  ⚠️ 大规模改动（文件数 > 10 或行数 > 200）
  ⚠️ 新增外部依赖（引入新的第三方库）
  ⚠️ 数据库 schema 变更
```

### 人工审查的节奏

OpenAI 团队的实践：**大多数 PR 在 1 分钟内完成审查。**

这是因为：
- CI 已经验证了代码的正确性
- 架构约束已经保证了结构的合理性
- 人工审查只需确认"结果是否符合预期"，不需要逐行检查代码

```
1 分钟审查流程：
  1. 看 PR 描述（Agent 自动生成）：做了什么？（10秒）
  2. 看 CI 结果：全绿？（5秒）
  3. 看关键文件的改动：结构是否合理？（30秒）
  4. 合并或留评论（15秒）
```

## 六、GitHub Actions 配置示例

以 GitHub Actions 为例，展示一个 Harness CI 的基本结构：

```yaml
# .github/workflows/harness-ci.yml

name: Harness CI

on:
  pull_request:
    branches: [main]

jobs:
  # 阶段一：快速检查
  quick-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: 格式检查
        run: npm run format:check   # 或 ruff format --check .
      - name: 基础 Linter
        run: npm run lint:basic     # 或 ruff check .

  # 阶段二：架构约束检查
  arch-check:
    needs: quick-check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: 架构层级检查
        run: npm run lint:arch      # 自定义架构 Linter
      - name: 循环依赖检测
        run: npx depcruise src      # 或类似工具

  # 阶段三：测试
  test:
    needs: arch-check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: 运行测试
        run: npm test
      - name: 覆盖率检查
        run: npm run test:coverage -- --threshold 80
```

## 本节小结

```
CI 的角色：
  Harness 的最后一道防线
  服务端强制执行，不依赖 Agent 本地环境
  违规代码物理上无法合并

流水线结构（快速失败原则）：
  阶段一：快速检查（格式、基础 Linter）
  阶段二：架构约束检查（层级规则、循环依赖）
  阶段三：测试（单元、集成、覆盖率）
  阶段四：质量检查（可选，Warning 级别）

检查项分级：
  阻断：架构违规、测试失败、覆盖率不足
  警告：质量问题、轻微规范违反
  记录：性能报告、依赖提示

PR 审查策略：
  小 PR + CI 全绿 → 自动合并
  架构变更 / 安全代码 / 大改动 → 人工审查
  人工审查目标：1 分钟内完成
```
