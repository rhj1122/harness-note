# 6.4　Harness 演进策略

> 本节目标：理解 Harness 的演进路径，掌握何时扩展、何时重构 Harness，以及如何从"偶尔用 AI"平滑迁移到"AI 为主"的工作模式。

---

## 一、Mitchell Hashimoto 的六阶段路径

Mitchell Hashimoto 把工程师使用 AI 的成熟度分为六个阶段，Harness Engineering 对应第五阶段。

```
阶段一：AI 辅助（AI-Assisted）
  你在写代码，AI 偶尔帮你补全
  Harness 需求：无

阶段二：AI 协作（AI-Collaborative）
  AI 生成代码片段，你来整合
  Harness 需求：基础的 AGENTS.md

阶段三：AI 委托（AI-Delegated）
  把完整的小任务委托给 AI
  Harness 需求：AGENTS.md + 基础 ESLint 规则

阶段四：AI 主导（AI-Led）
  AI 主导大部分开发，你做架构决策
  Harness 需求：完整的文档体系 + 架构约束

阶段五：Harness Engineering ← 目标
  AI 在严格约束下自主工作，你维护约束系统
  Harness 需求：完整的 Harness（文档 + 约束 + 可观测性 + 熵管理）

阶段六：自主工程（理论阶段）
  AI 系统自主演进代码库
  Harness 需求：超出当前技术范畴
```

**你不需要一步跳到第五阶段。** 每个阶段都有对应的 Harness 配置，随着熟练度提升逐步演进。

## 二、各阶段的 Harness 配置

### 阶段三（入门）：最小 Harness

```
适合：刚开始把任务委托给 AI，还不确定效果

配置：
  ✅ AGENTS.md（约 25 行）
  ✅ docs/architecture.md（基础分层说明）
  ✅ 3 条核心 ESLint 规则
  ✅ npm run check 脚本

不需要：
  ❌ 完整的 docs/ 体系
  ❌ 黄金原则
  ❌ 周期性清理
  ❌ 浏览器自动化

工作模式：
  每次给 AI 一个小任务
  人工审查每一行代码
  发现问题 → 更新 AGENTS.md
```

### 阶段四（成长）：标准 Harness

```
适合：AI 已经能可靠完成单个功能，开始尝试并行

配置：
  ✅ 完整的 docs/ 体系（README + architecture + conventions + tech-choices）
  ✅ 10-15 条 ESLint 规则
  ✅ 黄金原则（docs/golden-principles.md）
  ✅ 完整的 CI 流水线
  ✅ Vitest 测试配置

不需要：
  ❌ 周期性清理 Agent（手动清理即可）
  ❌ 浏览器自动化（按需使用）
  ❌ Git Worktree（单 Agent 工作）

工作模式：
  每天分配 3-5 个任务给 AI
  1 分钟快速审查 PR
  每周手动检查代码质量
```

### 阶段五（成熟）：完整 Harness

```
适合：AI 是主要生产力，工程师专注于架构和 Harness 维护

配置：
  ✅ 完整的 docs/ 体系
  ✅ 20+ 条 ESLint 规则
  ✅ 黄金原则（持续更新）
  ✅ 完整的 CI 流水线
  ✅ 周期性清理 Agent（自动化）
  ✅ 质量度量仪表盘
  ✅ Git Worktree 并行工作
  ✅ 浏览器自动化（E2E 测试）
  ✅ 定期 Harness Review

工作模式：
  每天分配 10-20 个任务给多个 Agent
  大部分 PR 自动合并
  工程师主要工作：架构决策 + Harness 维护
```

## 三、从当前阶段迁移到下一阶段

### 判断是否准备好升级

```
从阶段三升级到阶段四的信号：
  ✅ AI 完成的任务首次 CI 通过率 > 70%
  ✅ 你已经积累了 10+ 条"Agent 犯过的错误"
  ✅ 你对项目架构有清晰的文档化描述
  ✅ 每周花在审查 AI 代码上的时间 > 2 小时

从阶段四升级到阶段五的信号：
  ✅ AI 完成的任务首次 CI 通过率 > 85%
  ✅ 你已经有完整的 docs/ 体系
  ✅ 代码库质量指标稳定
  ✅ 你开始感觉"手动清理 Slop 太费时间了"
```

### 迁移步骤

```
阶段三 → 阶段四：
  1. 补全 docs/ 体系（约 4 小时）
  2. 把积累的"Agent 犯过的错误"整理成黄金原则
  3. 把最常见的违规模式写成 ESLint 规则
  4. 配置完整的 CI 流水线

阶段四 → 阶段五：
  1. 配置 Git Worktree，开始并行工作
  2. 设置周期性清理 Agent（GitHub Actions 定时任务）
  3. 建立质量度量仪表盘
  4. 配置 Playwright E2E 测试
```

## 四、何时扩展 Harness，何时重构 Harness

### 扩展 Harness（加新规则/文档）

```
触发条件：
  Agent 犯了一个新类型的错误
  → 加一条 ESLint 规则或黄金原则

  新功能引入了新的架构模式
  → 更新 docs/architecture.md

  引入了新的工具或库
  → 更新 docs/tech-choices.md

原则：扩展是增量的，不影响已有的 Harness
```

### 重构 Harness（改变结构）

```
触发条件：
  AGENTS.md 超过 150 行，Agent 开始忽略部分规则
  → 重构：精简 AGENTS.md，把内容移到 docs/

  ESLint 规则超过 30 条，CI 运行时间超过 15 分钟
  → 重构：合并相似规则，删除过时规则

  docs/ 结构混乱，Agent 找不到需要的文档
  → 重构：重新组织 docs/ 目录结构

  技术栈迁移（比如从 Redux 迁移到 Zustand）
  → 重构：更新所有相关文档和规则

原则：重构是结构性变化，需要谨慎，做好测试
```

## 五、Harness 演进的节奏建议

```
每次 Agent 犯错后（立即）：
  → 更新 AGENTS.md 或添加 ESLint 规则
  → 这是最重要的演进时机，不要拖延

每周（30 分钟）：
  → 检查质量指标趋势
  → 把本周积累的"Agent 犯过的错误"整理成规则

每 2 周（1 小时）：
  → Harness Review（检查 Harness 是否老化）
  → 评估是否需要升级到下一阶段

每次大版本迭代后（2-3 小时）：
  → 全面审查 Harness 是否还适合当前项目状态
  → 考虑是否需要重构某个部分
```

## 本节小结

```
六阶段路径：
  阶段三（入门）→ 阶段四（成长）→ 阶段五（成熟）
  不需要一步到位，随熟练度逐步演进

各阶段 Harness 配置：
  阶段三：最小 Harness（AGENTS.md + 3条规则）
  阶段四：标准 Harness（完整 docs + 15条规则 + CI）
  阶段五：完整 Harness（+ 并行 + 清理 + 度量）

升级信号：
  CI 首次通过率 > 70% → 可以升级到阶段四
  CI 首次通过率 > 85% → 可以升级到阶段五

扩展 vs 重构：
  扩展：增量添加规则/文档（频繁，低风险）
  重构：结构性变化（谨慎，需要测试）

演进节奏：
  每次犯错后立即更新
  每周整理规则
  每 2 周 Harness Review
```
