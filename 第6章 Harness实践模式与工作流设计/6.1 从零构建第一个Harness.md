# 6.1　从零构建第一个 Harness

> 本节目标：掌握从零开始为一个前端项目搭建 Harness 的完整步骤，包括项目初始化清单、AGENTS.md 最小可用版，以及第一批 ESLint 规则的选取策略。

---

## 一、什么时候开始搭建 Harness

不需要等项目"足够大"才开始。

```
错误的想法：
  "项目还小，等大了再搭 Harness"
  → 等项目大了，Slop 已经积累，搭 Harness 的成本更高

正确的做法：
  项目一开始就搭最小可用的 Harness
  → 随项目成长逐步完善
  → 始终保持代码库健康
```

**最小可用 Harness 只需要 3 样东西：**
1. 一份 AGENTS.md
2. 一个 docs/ 目录（含 README.md + architecture.md）
3. 基础 ESLint 配置

搭建时间：**约 2 小时**。

---

## 二、项目初始化清单

以一个新的 AI H5 对话应用为例，从零开始：

### 第一步：确定项目技术栈

```
技术栈决策（写进 docs/tech-choices.md）：
  框架：React 18 + TypeScript
  构建：Vite
  状态管理：Zustand
  样式：Tailwind CSS
  测试：Vitest + React Testing Library
  HTTP：axios
  AI SDK：openai（通过 AIProvider 封装）
  代码规范：ESLint + Prettier
```

### 第二步：确定分层架构

```
前端七层架构（写进 docs/architecture.md）：
  Types → Config → API → Store → Hooks → Components → Pages

目录结构：
  src/
  ├── types/          ← 类型定义
  ├── config/         ← 配置（环境变量等）
  ├── api/            ← 网络请求封装
  ├── store/          ← Zustand 状态管理
  ├── hooks/          ← 自定义 Hook（业务逻辑）
  ├── components/     ← UI 组件
  ├── pages/          ← 页面
  ├── providers/      ← 横切关注点（Auth、AI、Tracking 等）
  └── utils/          ← 纯工具函数
```

### 第三步：创建 docs/ 目录

```
docs/
├── README.md           ← 文档总索引（必须先建）
├── architecture.md     ← 分层架构说明
├── conventions.md      ← 编码规范
└── tech-choices.md     ← 技术选型
```

## 三、AGENTS.md 最小可用版

```markdown
# ai-h5-chat

## 项目概述
内嵌在 App WebView 中的 AI 对话 H5 页面。
React 18 + TypeScript + Zustand + Vite。

## 架构
分层：Types → Config → API → Store → Hooks → Components → Pages
依赖只能向下，不能向上。详见：docs/architecture.md

## 关键规则
- Components 层禁止直接调用 fetch 或 AI SDK
- Pages 层禁止直接操作 Store（必须通过 Hooks）
- AI SDK 调用统一封装在 src/providers/AIProvider.tsx
- 所有网络请求统一封装在 src/api/ 目录

## 常用命令
npm run dev          # 启动开发服务器（端口 5173）
npm run check        # tsc + ESLint 检查
npx vitest run       # 运行测试
npm run build        # 构建

## 文档
docs/README.md
```

这份 AGENTS.md 约 25 行，远低于 100 行上限，但包含了 Agent 开始工作所需的全部入口信息。

## 四、第一批 ESLint 规则的选取策略

不要一开始就写 20 条规则，会让 Agent 陷入无休止的修复循环。

**第一批规则的选取原则：只选最重要的 3-5 条。**

### 如何判断"最重要"

```
优先选择：
  ✅ 违反后代价最高的规则（架构层级违规）
  ✅ Agent 最容易犯的错误（直接在组件里 fetch）
  ✅ 一旦扩散就很难清理的模式（AI SDK 直接调用）

暂时不选：
  ⬜ 代码风格类（Prettier 已经处理了）
  ⬜ 质量类（函数长度等，先用黄金原则引导）
  ⬜ 边缘场景（等真正出现问题再加）
```

### 推荐的第一批规则（前端 AI H5 项目）

```javascript
// eslint.config.js

// 规则一：Components 层不能直接导入 API 层
// 原因：这是最常见的架构违规，Agent 很容易犯
'harness/no-api-in-components': 'error'

// 规则二：Components 层不能直接使用 AI SDK
// 原因：AI SDK 必须通过 AIProvider 统一管理
'harness/no-ai-sdk-in-components': 'error'

// 规则三：Pages 层不能直接导入 Store
// 原因：Store 操作必须封装在 Hooks 里
'harness/no-store-in-pages': 'error'
```

这三条规则覆盖了最关键的架构边界，足以保护项目早期的代码质量。

## 五、完整的初始化流程

```bash
# 1. 创建项目
npm create vite@latest ai-h5-chat -- --template react-ts
cd ai-h5-chat
npm install

# 2. 安装必要依赖
npm install zustand axios openai
npm install -D vitest @testing-library/react @testing-library/jest-dom
npm install -D @playwright/test
npm install -D prettier eslint-config-prettier

# 3. 创建目录结构
mkdir -p src/{types,config,api,store,hooks,components,pages,providers,utils}
mkdir -p docs/{decisions,skills,modules}
mkdir -p eslint-rules
mkdir -p e2e

# 4. 创建 Harness 文件
touch AGENTS.md
touch docs/README.md
touch docs/architecture.md
touch docs/conventions.md
touch docs/tech-choices.md
touch docs/golden-principles.md

# 5. 配置 package.json scripts
# 添加：
# "check": "tsc --noEmit && eslint src --ext .ts,.tsx",
# "test": "vitest",
# "test:run": "vitest run"
```

## 六、初始化后的验证

Harness 搭建完成后，用这个清单验证：

```
□ AGENTS.md 存在，约 20-30 行，内容准确
□ docs/README.md 存在，列出了所有文档
□ docs/architecture.md 存在，描述了分层结构
□ ESLint 配置完成，npm run check 能正常运行
□ 测试配置完成，npx vitest run 能正常运行
□ 构建配置完成，npm run build 能正常运行
□ CI 配置完成（.github/workflows/harness-ci.yml）

验证 Agent 能正常工作：
□ 给 Agent 一个简单任务（比如"创建一个 Hello World 组件"）
□ 确认 Agent 读取了 AGENTS.md
□ 确认 Agent 把文件放在了正确的目录
□ 确认 npm run check 通过
```
