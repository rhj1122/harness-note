# 6.5　团队协作中的 Harness Engineering

> 本节目标：理解多人团队如何共同维护 Harness，明确 Harness 的所有权归属，以及规则变更的决策流程。

---

## 一、个人项目 vs 团队项目的差异

前几节的内容主要以单人工程师为视角。团队协作引入了新的挑战：

```
个人项目：
  一个人维护 Harness
  规则变更随时可以做
  没有沟通成本

团队项目（3-10人）：
  多人同时使用 Harness
  规则变更影响所有人
  需要共识和沟通
  需要明确"谁负责什么"
```

---

## 二、Harness 的所有权模型

### 模型一：单一 Owner（小团队推荐）

```
适合：3-5 人团队

设计：
  指定一名工程师作为 Harness Owner
  负责：
    - 维护 AGENTS.md 和 docs/
    - 审查和合并 ESLint 规则变更
    - 定期执行 Harness Review
    - 处理 Harness 相关的问题

其他成员：
  可以提 PR 修改 Harness
  但需要 Owner 审批

优点：决策快，一致性强
缺点：Owner 成为单点，需要备份
```

### 模型二：共同所有权（中型团队推荐）

```
适合：5-10 人团队

设计：
  Harness 的不同部分由不同人负责：
    AGENTS.md + docs/：技术负责人
    ESLint 规则：前端架构师
    CI 配置：DevOps 工程师
    黄金原则：全体工程师共同维护

变更流程：
  任何人都可以提 PR 修改 Harness
  对应负责人审批
  重大变更需要团队讨论

优点：分散风险，每个人都有参与感
缺点：需要更多沟通协调
```

## 三、Harness 规则的决策流程

### 小改动（快速决策）

```
触发：Agent 犯了一个新错误，需要加一条规则

流程：
  发现问题的工程师
      ↓
  提 PR：新增一条 ESLint 规则或黄金原则
      ↓
  Harness Owner 审查（目标：24小时内）
      ↓
  合并

不需要：团队讨论、投票、会议
```

### 中等改动（需要通知）

```
触发：修改现有规则的阈值、更新架构文档

流程：
  提 PR
      ↓
  在 PR 描述里说明：为什么要改、影响范围
      ↓
  通知相关成员（Slack/钉钉）
      ↓
  等待 24 小时，收集反馈
      ↓
  合并（无异议）或讨论（有异议）
```

### 重大改动（需要共识）

```
触发：架构层级变更、技术栈迁移、大规模规则重构

流程：
  提议者写一份简短的设计文档（docs/decisions/ 里）
      ↓
  团队异步讨论（在 PR 评论或文档评论里）
      ↓
  达成共识后，提 PR 实施
      ↓
  所有相关成员审查
      ↓
  合并

原则：重大变更不能单方面决定
```

## 四、多人团队的 AGENTS.md 维护

团队里每个人都可能发现 Agent 犯错，都应该有权更新 AGENTS.md。

### 防止冲突的方法

```
方法一：AGENTS.md 分层（推荐）

根目录 AGENTS.md（全局规则，需要 Owner 审批）：
  - 架构概览
  - 最关键的 5 条规则
  - 文档入口

src/components/AGENTS.md（组件层规则，前端工程师维护）：
  - 组件编写规范
  - 常见错误提示

src/hooks/AGENTS.md（Hook 层规则，前端工程师维护）：
  - Hook 设计规范
  - 常见错误提示

好处：
  - 每个人只维护自己负责的部分
  - 减少冲突
  - 规则更精准（Agent 在哪个目录工作就读哪个 AGENTS.md）
```

```
方法二：AGENTS.md 变更需要 PR

不允许直接 push 到 main 分支修改 AGENTS.md
所有修改必须通过 PR
→ 有记录，可追溯，可讨论
```

## 五、团队 Harness Review

单人项目的 Harness Review 是一个人做。团队项目需要协作：

```
团队 Harness Review 流程（每 2 周，约 1 小时）：

准备（提前 2 天）：
  Harness Owner 生成 Harness 健康报告
  发给团队成员异步阅读

会议（30 分钟）：
  1. 过一遍质量指标趋势（10 分钟）
  2. 讨论本期 Agent 犯的典型错误（10 分钟）
  3. 决定需要新增/修改的规则（10 分钟）

会后（1 周内）：
  各自负责的部分提 PR 更新
  Owner 汇总合并
```

## 六、新成员加入时的 Harness 交接

新工程师加入团队时，Harness 是他们了解项目的最佳入口：

```
新成员入职流程（Harness 视角）：

第一天：
  1. 读 AGENTS.md（了解项目全貌，约 10 分钟）
  2. 读 docs/architecture.md（了解分层架构，约 20 分钟）
  3. 读 docs/conventions.md（了解编码规范，约 15 分钟）

第一周：
  4. 完成一个小任务（感受 Harness 的工作方式）
  5. 如果发现 AGENTS.md 有不清楚的地方，提 PR 改进

新成员的反馈很有价值：
  他们是"第一次读 AGENTS.md 的 Agent"
  他们发现的困惑，往往是 Harness 需要改进的地方
```

## 七、团队规模扩大时的 Harness 演进

```
3 人团队 → 10 人团队的变化：

文档体系：
  3 人：一份 AGENTS.md + 基础 docs/
  10 人：分层 AGENTS.md + 完整 docs/ + 模块级文档

规则数量：
  3 人：5-10 条 ESLint 规则
  10 人：20-30 条 ESLint 规则（覆盖更多场景）

所有权：
  3 人：单一 Owner
  10 人：共同所有权，按模块分工

Review 频率：
  3 人：每月一次
  10 人：每 2 周一次

原则：团队越大，Harness 越需要正式化
```

## 本节小结

```
所有权模型：
  小团队（3-5人）：单一 Owner
  中型团队（5-10人）：共同所有权，按模块分工

规则变更决策：
  小改动：Owner 快速审批（24小时）
  中等改动：通知相关成员，等待反馈
  重大改动：团队共识，设计文档先行

多人维护 AGENTS.md：
  分层 AGENTS.md（全局 + 模块级）
  所有修改通过 PR，有记录可追溯

团队 Harness Review：
  每 2 周，约 1 小时
  提前异步阅读报告，会议聚焦决策

新成员入职：
  AGENTS.md 是最佳入口
  新成员的困惑 = Harness 改进的信号
```
