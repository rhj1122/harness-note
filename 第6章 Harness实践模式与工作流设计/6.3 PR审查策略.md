# 6.3　PR 审查策略

> 本节目标：建立清晰的 PR 审查标准，掌握 1 分钟快速审查的方法，让 PR 审查不成为工作流的瓶颈。

---

## 一、PR 审查的角色转变

传统工程师审查 PR 时，需要：
- 逐行阅读代码，理解逻辑
- 判断实现是否正确
- 检查是否有 bug
- 确认代码风格是否符合规范

这在 Harness Engineering 里**不再是主要工作**。

```
Harness 已经替你做了大部分审查工作：

  ESLint 已经检查了：架构规则、编码规范
  tsc 已经检查了：类型正确性
  Vitest 已经检查了：功能逻辑
  CI 已经确认了：以上所有检查都通过

工程师的审查职责变成了：
  确认"结果是否符合预期"
  而不是"代码是否正确"
```

---

## 二、PR 分类：自动合并 vs 人工审查

### 可以自动合并的 PR

满足以下**所有**条件时，PR 可以配置为自动合并：

```
条件一：CI 全绿
  所有检查通过（tsc + ESLint + 测试 + 构建）

条件二：PR 规模小
  改动文件数 ≤ 10
  改动行数 ≤ 200

条件三：PR 类型安全
  以下类型可以自动合并：
  ✅ 文档更新（docs/ 目录的改动）
  ✅ 测试补充（只新增测试，不改业务代码）
  ✅ 依赖版本升级（patch 和 minor 版本）
  ✅ 清理 Agent 生成的重构 PR（死代码删除、命名统一）
  ✅ 样式调整（只改 CSS/Tailwind 类名）
```

### 需要人工审查的 PR

```
以下情况必须人工审查：
  ⚠️ 涉及架构变更（新增层级、修改依赖规则）
  ⚠️ 涉及 AIProvider 或 AuthProvider（核心横切关注点）
  ⚠️ 新增外部依赖（package.json 新增 dependencies）
  ⚠️ 修改 ESLint 规则或 CI 配置（Harness 本身的变更）
  ⚠️ 大规模改动（文件数 > 10 或行数 > 200）
  ⚠️ 数据结构变更（修改了 src/types/ 中的核心类型）
```

---

## 三、1 分钟审查流程

对于需要人工审查的 PR，目标是在 **1 分钟内**完成。

### 1 分钟审查的四个步骤

```
步骤一（10秒）：看 PR 标题和描述
  Agent 自动生成的 PR 描述应该包含：
  - 做了什么
  - 为什么这样做
  - 影响范围

  快速判断：这个 PR 做的事情是我期望的吗？

步骤二（5秒）：看 CI 状态
  全绿 → 继续
  有失败 → 看失败原因，通常 Agent 会自动修复

步骤三（30秒）：看关键文件的改动
  不需要逐行看，重点看：
  - 新建的文件放在正确的目录了吗？
  - 核心接口（Hook 的返回值、组件的 Props）设计合理吗？
  - 有没有明显的架构违规（ESLint 没有捕获到的）？

步骤四（15秒）：决策
  合理 → 合并
  有问题 → 留一条具体的评论，让 Agent 修复
```

### 留评论的写法

评论要对 Agent 友好，和 ESLint 错误信息一样：**说清楚问题 + 给出修复方向**。

```
差的评论：
  "这里不对"
  "请修改"

好的评论：
  "useMessageSearch 的 results 应该在 query 为空时返回空数组，
   而不是返回全部消息。
   修复：在 Hook 内部加一个判断：if (!query.trim()) return []"
```

---

## 四、PR 描述的标准格式

让 Agent 生成标准格式的 PR 描述，方便快速审查：

```markdown
## 做了什么
[一句话描述]

## 为什么
[背景和原因]

## 改动范围
- 新建：[文件列表]
- 修改：[文件列表]

## 验证
- [ ] npm run check 通过
- [ ] npx vitest run 通过
- [ ] npm run build 通过

## 注意事项
[需要审查者特别关注的地方，没有则省略]
```

在 AGENTS.md 里声明这个格式：

```markdown
## PR 规范
提交 PR 时，描述必须包含：做了什么 / 为什么 / 改动范围 / 验证结果
模板：docs/pr-template.md
```

---

## 五、处理 CI 失败的 PR

当 CI 失败时，工程师不需要自己修复，而是让 Agent 处理：

```
CI 失败的处理流程：

1. 工程师看到 CI 失败的 PR
2. 快速判断：是 Agent 的问题，还是 Harness 的问题？

   Agent 的问题（代码写错了）：
   → 在 PR 评论里告诉 Agent："CI 失败，请查看日志并修复"
   → Agent 读取 CI 日志，自动修复并重新推送

   Harness 的问题（ESLint 规则误报、CI 配置有误）：
   → 工程师修复 Harness
   → 更新相关文档

3. 如果 Agent 连续修复 3 次仍然失败：
   → 工程师介入，分析根本原因
   → 可能需要更新任务描述或 Harness
```

## 六、PR 审查的常见陷阱

### 陷阱一：过度审查

```
症状：工程师花 30 分钟逐行审查每个 PR
原因：不信任 Harness 的检查结果
后果：PR 积压，成为工作流瓶颈

解法：
  信任 CI 全绿的结论
  只关注"结果是否符合预期"，不关注"实现细节是否完美"
  记住：Harness 已经检查了大部分问题
```

### 陷阱二：审查标准不一致

```
症状：有时候快速合并，有时候要求大量修改
原因：没有明确的审查标准
后果：Agent 不知道什么样的代码会被接受

解法：
  把审查标准写进 docs/conventions.md
  "自动合并条件"和"必须人工审查条件"明确文档化
```

### 陷阱三：忽略 PR 积压

```
症状：Agent 提交了 10 个 PR，工程师没有及时审查
原因：工程师忙于其他事情
后果：Agent 等待合并，无法开始依赖这些 PR 的后续任务

解法：
  每天早上固定 10 分钟处理 PR
  设置 PR 积压超过 5 个时的提醒
```

## 本节小结

```
角色转变：
  不再逐行审查代码
  只确认"结果是否符合预期"

PR 分类：
  自动合并：CI 全绿 + 规模小 + 类型安全
  人工审查：架构变更 / 核心依赖 / 大改动 / Harness 变更

1 分钟审查四步：
  看描述（10秒）→ 看 CI（5秒）→ 看关键文件（30秒）→ 决策（15秒）

留评论原则：
  说清楚问题 + 给出修复方向（对 Agent 友好）

CI 失败处理：
  Agent 的问题 → 让 Agent 自己修复
  Harness 的问题 → 工程师修复 Harness

常见陷阱：
  过度审查 / 标准不一致 / 忽略 PR 积压
```
