# 6.2　任务分解与 Agent 调度

> 本节目标：掌握如何把需求拆解成 Agent 可以高效执行的任务，理解任务描述的写法对 Agent 输出质量的影响。

---

## 一、为什么任务分解很重要

Agent 的工作质量很大程度上取决于任务描述的质量。

```
模糊的任务描述：
  "实现 AI 对话功能"

Agent 的困境：
  - 不知道从哪里开始
  - 不知道范围有多大
  - 不知道验收标准是什么
  - 可能生成一个庞大的、难以审查的 PR
```

```
清晰的任务描述：
  "在 src/hooks/ 创建 useAIChat Hook，
   实现发送消息和接收流式响应的功能。
   输入：用户消息字符串
   输出：{ messages, isStreaming, sendMessage, clearHistory }
   验证：运行 npm run check && npx vitest run src/hooks/useAIChat.test.ts"

Agent 的优势：
  - 知道在哪里创建文件
  - 知道接口设计
  - 知道如何验证
  - 生成的 PR 范围可控，易于审查
```

---

## 二、好任务描述的五个要素

```
1. 明确的目标（做什么）
   "创建 useAIChat Hook"
   不是"实现 AI 功能"

2. 明确的位置（在哪里）
   "在 src/hooks/ 目录下"
   不是"在合适的地方"

3. 明确的接口（输入输出）
   "接受 messages 数组，返回 { send, isLoading, response }"
   不是"处理消息"

4. 明确的约束（不能做什么）
   "不要直接调用 AI SDK，通过 useAIProvider() 获取客户端"
   不是"遵守架构规则"

5. 明确的验证方式（怎么确认完成）
   "运行 npx vitest run src/hooks/useAIChat.test.ts，所有测试通过"
   不是"确保代码正确"
```

---

## 三、任务描述模板

```markdown
## 任务：[简短标题]

### 目标
[一句话描述要做什么]

### 位置
- 新建文件：[文件路径]
- 修改文件：[文件路径]（如果是修改任务）

### 接口设计
[函数签名、Props 定义、返回值等]

### 约束
- [不能做的事 1]
- [不能做的事 2]

### 验证
[运行什么命令，期望什么结果]

### 参考
- [相关文档链接]
- [相似实现的参考文件]
```

## 四、实际任务描述示例

### 示例一：创建新 Hook

```markdown
## 任务：创建 useMessageSearch Hook

### 目标
实现消息搜索功能，支持在历史消息中按关键词搜索。

### 位置
- 新建文件：src/hooks/useMessageSearch.ts
- 新建测试：src/hooks/useMessageSearch.test.ts

### 接口设计
```typescript
function useMessageSearch() {
  return {
    query: string,
    setQuery: (q: string) => void,
    results: Message[],
    isSearching: boolean,
  }
}
```

### 约束
- 不要直接访问 chatStore，通过 useChatStore() Hook 获取消息列表
- 搜索逻辑放在 Hook 内部，不要放在 Store 里
- 搜索应该是实时的（query 变化时自动触发）

### 验证
运行：npx vitest run src/hooks/useMessageSearch.test.ts
期望：所有测试通过，npm run check 无报错

### 参考
- 类似实现参考：src/hooks/useAIChat.ts
- 消息类型定义：src/types/message.ts
```

### 示例二：修复 Bug

```markdown
## 任务：修复流式渲染重复字符 bug

### 目标
修复 useAIStream Hook 中流式响应偶尔出现重复字符的问题。

### 位置
- 修改文件：src/hooks/useAIStream.ts

### 问题描述
当 AI 流式响应速度较快时，onChunk 回调可能被重复调用，
导致同一段文字被追加两次。

### 约束
- 不要修改 AIProvider 的实现
- 修复方案应该在 Hook 层处理，不要改 API 层

### 验证
运行：npx vitest run src/hooks/useAIStream.test.ts
期望：新增的"重复字符"测试用例通过，原有测试不回归

### 参考
- 相关代码：src/hooks/useAIStream.ts 第 45-67 行
- 问题复现：e2e/reproduce-streaming-bug.spec.ts
```

## 五、任务粒度：多大合适

```
太大的任务（一个 PR 改动 > 20 个文件）：
  ❌ Agent 难以保持一致性
  ❌ 工程师难以审查
  ❌ 出错后难以定位问题

太小的任务（只改一行代码）：
  ❌ 任务描述的开销 > 实际工作量
  ❌ 大量 PR 增加管理负担

合适的任务粒度：
  ✅ 一个完整的功能单元（一个 Hook、一个组件、一个 API 函数）
  ✅ 改动文件数 3-10 个
  ✅ 工程师能在 1-2 分钟内审查完
```

## 六、并行任务 vs 串行任务的判断

### 可以并行的任务

```
判断标准：任务之间没有依赖关系

示例（可以并行）：
  任务 A：创建 useMessageSearch Hook
  任务 B：优化 MessageList 组件的渲染性能
  任务 C：修复输入框在 iOS 上的键盘遮挡问题

  → 三个任务互相独立，分配给三个 Agent 同时执行
```

### 必须串行的任务

```
判断标准：任务 B 依赖任务 A 的输出

示例（必须串行）：
  任务 A：重构 API 层，统一错误处理格式
  任务 B：更新所有 Hook，适配新的错误格式

  → 任务 B 依赖任务 A 的结果，必须等 A 合并后再开始

示例（必须串行）：
  任务 A：在 src/types/ 中定义 SearchResult 类型
  任务 B：创建 useMessageSearch Hook（使用 SearchResult 类型）

  → 任务 B 依赖任务 A 定义的类型
```

### 依赖关系图

在分配任务前，先画出依赖关系图：

```
本周任务依赖图：

类型定义（A）
    ↓
API 封装（B）← 依赖 A
    ↓
useAIChat Hook（C）← 依赖 B
    ↓
ChatPage 组件（D）← 依赖 C

同时可以并行：
  消息搜索（E）← 独立
  性能优化（F）← 独立
  Bug 修复（G）← 独立

执行顺序：
  第一批（并行）：A + E + F + G
  第二批（并行，等 A 完成）：B
  第三批（等 B 完成）：C
  第四批（等 C 完成）：D
```

## 七、任务调度的实践节奏

```
每天早上（10 分钟）：
  1. 查看昨天 Agent 提交的 PR
  2. 快速审查并合并（或留评论）
  3. 根据合并结果，分配今天的新任务

每天下午（5 分钟）：
  1. 检查 Agent 是否卡住（CI 失败超过 2 次）
  2. 如果卡住，分析原因，更新 Harness 或调整任务描述

每周一（30 分钟）：
  1. 规划本周任务，画依赖关系图
  2. 确定哪些可以并行，哪些必须串行
  3. 准备好任务描述，等待分配
```

## 本节小结

```
任务描述的五个要素：
  目标（做什么）+ 位置（在哪里）+ 接口（输入输出）
  + 约束（不能做什么）+ 验证（怎么确认完成）

任务粒度：
  改动 3-10 个文件，工程师 1-2 分钟能审查完

并行 vs 串行：
  无依赖关系 → 并行（分配给多个 Agent）
  有依赖关系 → 串行（等上游任务合并后再开始）

调度节奏：
  每天早上审查 PR + 分配新任务
  每周一规划任务依赖图
```
