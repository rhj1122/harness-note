# 1.1　什么是 Harness Engineering？

> 本节目标：彻底搞清楚这个概念从哪来、解决什么问题、为什么重要。

---

## 一、"harness" 这个词从哪来？

**harness** 是英文里的"马具"——套在马身上的那套皮带、缰绳、轭具。

马很强壮，但如果没有马具，它的力量是随机的、不可控的。
套上马具之后，马的力量才能被引导到正确的方向，稳定地拉动马车。

```
没有马具的马  →  力量强大，但方向随机，无法驾驭
套上马具的马  →  同样的力量，但可预测、可控制、可持续工作
```

**AI 大模型就是那匹马。**

模型的能力极强，但如果没有约束，它会：
- 今天用这种方式写组件，明天换另一种
- 在不该用某个模式的地方用了它
- 把整个前端代码库越写越乱，直到没人能维护

**Harness（驾具）就是套在 AI 身上的那套约束系统。**

---

## 二、从"写代码"到"驾驭 AI 写代码"

### 传统工程师的工作

```
工程师  →  理解需求  →  亲手写代码  →  提交
```

工程师是**直接生产者**，代码质量取决于工程师个人的水平和习惯。

### Vibe Coding 阶段（2023-2025 年主流）

```
工程师  →  描述需求  →  AI 生成代码  →  工程师粘贴/微调  →  提交
```

工程师变成了**提示词写手 + 代码审核员**。
效率提升了，但代码质量开始参差不齐。
这个阶段叫 **vibe coding**——凭感觉驾驭 AI，没有系统性约束。

### Harness Engineering 阶段（2026 年起）

```
Harness Engineer  →  设计约束系统（harness）
                         ↓
                    AI Agent 在 harness 内自主执行任务
                         ↓
                    机械反馈（ESLint/CI）自动检查
                         ↓
                    Agent 自动修复
                         ↓
                    工程师审查结果（而非审查每一行代码）
```

工程师变成了**系统设计者**，不再逐行写代码，而是设计让 AI 可靠工作的环境。

---

## 三、为什么 Vibe Coding 不够用？

Vibe coding 有一个致命弱点：**它不可持续**。

想象一个前端项目用 vibe coding 开发了 3 个月：

- 第 1 周：AI 生成了一套用户认证逻辑，用了 `useAuth` Hook
- 第 3 周：AI 又生成了另一套，因为它"忘了"第 1 周的那套，这次直接在组件里处理 token
- 第 6 周：代码库里有 3 套认证逻辑，互相不兼容
- 第 3 个月：没有人敢动任何一行认证相关代码，因为不知道会破坏什么

这就是 **AI Slop**（AI 垃圾代码）的蔓延过程。

> Mitchell Hashimoto 说得很直接：
> "Vibe coding 只适合可丢弃的周末项目。
>  一旦你需要维护代码，vibe coding 就会让你付出惨痛代价。"

**可维护性的代价**就是：没有约束的 AI 生成代码，会以指数级速度积累技术债。

---

## 四、Harness Engineering 解决的核心问题

| 问题 | Harness Engineering 的解法 |
|------|---------------------------|
| AI 每次生成风格不一致 | 用 ESLint 规则强制统一风格，违规自动报错 |
| AI 不知道项目的架构边界 | 用 AGENTS.md + 架构文档告诉它边界在哪 |
| AI 会复制坏模式 | 用黄金原则文档定义什么是好模式 |
| AI 生成的代码越来越乱 | 用周期性清理 Agent 持续对抗代码熵 |
| AI 无法验证自己的结果 | 用可观测性工具让 Agent 能"看到"运行结果 |

一句话总结：

> **Harness Engineering = 设计让 AI Agent 可靠、可持续地生产可维护代码的约束环境系统。**

---

## 五、OpenAI 内部实验：3 人 × 零手写代码 × 100 万行

这是 Harness Engineering 最有说服力的证据。

**背景：** OpenAI 内部一个小团队，用自己开发的 Codex agent 系统，构建了一个真实的生产级应用。

**结果：**
- 团队规模：**3 名工程师**
- 手写代码行数：**0 行**（所有代码由 Agent 生成）
- 生成代码总量：**超过 100 万行**
- 代码质量：**可维护、可扩展、通过所有 CI 检查**

**为什么能做到？**

因为他们没有用 vibe coding，而是先花时间设计了一套完整的 harness：
- 精心设计的 AGENTS.md
- 严格的分层架构约束
- 自定义 linter 规则（违规时告诉 Agent 怎么修）
- 完整的 CI 流水线
- 本地可观测性栈

这 3 名工程师的工作不是写代码，而是**维护和改进这套 harness**。

> 效率数据：每名工程师的产出相当于传统模式下 **3-10 名工程师**。

## 本节小结

```
harness（马具）隐喻
    ↓
AI 模型 = 强大但需要约束的"马"
harness = 让 AI 力量可控、可预测的"驾具"
    ↓
工程师角色转变：
    写代码者  →  设计约束系统的架构师
    ↓
核心等式：Agent = Model + Harness
```
