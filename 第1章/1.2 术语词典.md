# 1.2　术语词典

> 本节目标：建立你的 Harness Engineering 词汇表。这些术语在后续每一章都会反复出现，先认识它们，后面学起来会顺很多。

---

## 一、核心角色：Agent / Model / LLM / Codex

这四个词经常混用，但含义有细微差别：

| 术语 | 含义 | 类比 |
|------|------|------|
| **LLM**（大语言模型） | 底层的 AI 模型，只负责"预测下一个词" | 发动机 |
| **Model**（模型） | 同 LLM，泛指具体的模型版本（如 GPT-4o、Claude） | 某款发动机型号 |
| **Agent**（智能体） | 模型 + 工具调用能力 + 自主执行循环 | 装了发动机的机器人 |
| **Codex** | OpenAI 推出的专为编码设计的 Agent 系统（2026年新版） | 专门干工程活的机器人 |

关键区别：

- **LLM/Model** 是被动的，你问它才回答
- **Agent** 是主动的，它会自己调用工具、读文件、执行命令、循环直到完成任务

> 核心等式再强调一次：`Agent = Model + Harness`
> 模型提供智能，harness 决定这个智能是否可靠。

---

## 二、三种编码范式的区别

这是理解 Harness Engineering 定位的关键。

### Vibe Coding（氛围编码）
- **谁在写代码：** AI，人来粘贴和微调
- **约束程度：** 几乎没有
- **适合场景：** 周末项目、原型验证、可丢弃的脚本
- **致命弱点：** 代码库会随时间快速腐化，不可维护

### Agentic Engineering（智能体工程）
- **谁在写代码：** AI Agent 自主完成整个任务
- **约束程度：** 有一些（工具定义、基本提示词）
- **适合场景：** 自动化单一任务（写测试、生成文档）
- **致命弱点：** 多个 Agent 协作时容易产生冲突和混乱

### Harness Engineering（驾驭工程）
- **谁在写代码：** AI Agent，但在严格的 harness 约束下
- **约束程度：** 系统性、机械性约束（linter、CI、架构规则）
- **适合场景：** 需要长期维护的生产级项目
- **核心优势：** 代码质量可持续，Agent 犯的错误会被永久修复

```
Vibe Coding          →  人驾驭 AI（靠感觉）
Agentic Engineering  →  AI 自主执行（有基本约束）
Harness Engineering  →  AI 在机械约束系统内自主执行（可持续、可维护）
```

---

## 三、Harness（驾具）

**Harness** 是本课程最核心的概念，值得单独解释清楚。

Harness 不是一个文件，也不是一个工具，而是一套**系统**，包含：

```
Harness 的组成
├── 上下文文档
│   ├── AGENTS.md          ← Agent 的"说明书"
│   ├── docs/              ← 架构文档、设计文档
│   └── skills/            ← 工具使用指南
├── 机械约束
│   ├── Linter 规则         ← 自动检查代码规范
│   └── CI 流水线           ← 自动拒绝违规代码
├── 反馈回路
│   ├── 可观测性工具         ← 让 Agent 能"看到"运行结果
│   └── 自动修复流程         ← Agent 根据反馈自我纠正
└── 熵管理
    ├── 黄金原则             ← 最佳实践规则集
    └── 周期性清理 Agent     ← 持续对抗代码腐化
```

一句话：**Harness = 让 AI Agent 可靠工作的整套约束环境**。

---

## 四、AGENTS.md

AGENTS.md 是放在项目根目录的一个 Markdown 文件，是 **Agent 的"说明书"**。

当 Agent 开始工作时，它会首先读取 AGENTS.md，了解：
- 这个项目是干什么的
- 架构是怎么分层的
- 有哪些规则必须遵守
- 常用命令是什么
- 深层文档在哪里找

**关键设计原则：AGENTS.md 是目录，不是百科全书。**

- 控制在约 100 行以内
- 只写"入口信息"，详细内容放在 docs/ 里
- 每次 Agent 犯错，就更新 AGENTS.md 防止重犯

> 我们会在第二章专门深入讲 AGENTS.md 的设计。

## 五、Context / Context Window（上下文 / 上下文窗口）

**Context（上下文）** 是 Agent 在执行任务时"能看到的所有信息"的总和：
- 你给它的任务描述
- 它读取的文件内容
- 工具调用的结果
- 对话历史

**Context Window（上下文窗口）** 是模型单次能处理的最大信息量，超出就会"忘记"早期内容。

这是一个**稀缺资源**。

```
上下文窗口就像工作台面：
- 台面有限，放太多东西就乱
- 放太少又找不到需要的信息
- Harness Engineering 的核心任务之一就是管理好这个台面
```

上下文管理不好的后果：
- Agent 忘记了项目的架构规则
- Agent 重复造轮子（因为没看到已有的实现）
- Agent 生成的代码与项目风格不一致

## 六、Linter / CI

### Linter（代码检查器）
自动扫描代码，发现不符合规范的地方并报错。

在 Harness Engineering 里，Linter 有一个特殊设计：
> **Linter 的错误信息 = 修复指令**

普通 Linter 只说"这里有问题"，Harness 里的 Linter 会说：
"这里违反了分层规则，`Service` 层不能直接依赖 `UI` 层，请将共享逻辑移到 `Repo` 层。"

这样 Agent 读到错误信息就知道怎么修，不需要人介入。

### CI（持续集成，Continuous Integration）
每次提交代码时自动运行的检查流水线：
- 运行所有 Linter
- 运行测试
- 检查架构约束
- 不通过则自动拒绝合并

CI 是 Harness 的**最后一道机械防线**——违规代码物理上无法进入主分支。

---

## 七、Git Worktree（代码库并行副本）

Git Worktree 是 Git 的一个功能，允许同一个代码库同时存在**多个独立的工作目录**。

在 Harness Engineering 里的用途：

```
主代码库
├── worktree-agent-1/   ← Agent 1 在这里工作（任务A）
├── worktree-agent-2/   ← Agent 2 在这里工作（任务B）
└── worktree-agent-3/   ← Agent 3 在这里工作（任务C）
```

多个 Agent 可以**同时并行工作**，互不干扰，最后各自提 PR。

这是实现高吞吐量 AI 开发的关键基础设施。

---

## 八、Entropy（熵）—— 代码库的"混乱度"

熵是物理学概念，在这里借用来描述**代码库的混乱程度**。

熵越高 = 代码越乱 = 越难维护 = 越容易出 bug

**AI 时代的熵增速度远超人工编码时代：**

| 场景 | 熵增速度 |
|------|---------|
| 纯人工编码 | 慢（人会自然地保持一致性） |
| Vibe Coding | 快（AI 每次生成风格不同） |
| 无 Harness 的 Agent | 极快（Agent 大量生成，风格混乱） |
| 有 Harness 的 Agent | 受控（约束系统持续对抗熵增） |

> OpenAI 团队曾经每周花 **20% 的时间**清理 AI 产生的代码熵。
> 引入 Harness 后，这个比例大幅下降。

---

## 九、AI Slop（AI 垃圾代码）

**AI Slop** 指 AI 生成的低质量、难维护的代码，特征：

- 重复实现（同一功能有 3 个版本）
- 过度工程化（简单问题用了复杂方案）
- 风格不一致（命名、结构、注释风格混乱）
- 无意义的注释（"// 这个函数做了 X"，但 X 显而易见）
- 死代码（生成了但从未被调用的函数）

AI Slop 是熵的主要来源，Harness Engineering 的熵管理模块专门对付它。

## 十、三个层级的关系

```
Prompt Engineering（提示词工程）
    ↓ 解决：如何写出好的单次提示词

Context Engineering（上下文工程）
    ↓ 解决：如何管理 Agent 在整个任务过程中看到的信息

Harness Engineering（驾驭工程）
    ↓ 解决：如何设计让 Agent 持续可靠工作的整套系统
```

这是一个**层层递进**的关系：

- Prompt Engineering 是基础，解决单次交互质量
- Context Engineering 是中层，解决任务执行过程中的信息管理
- Harness Engineering 是顶层，解决整个工程生命周期的可维护性

> 你可以把 Harness Engineering 理解为：
> **把 Context Engineering 系统化、机械化、持续化的工程学科。**
