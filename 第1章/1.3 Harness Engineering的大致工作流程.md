# 1.3　Harness Engineering 的大致工作流程

> 本节目标：在脑子里建立一张完整的地图——没有 harness 时会发生什么，有了 harness 之后整个工作闭环是怎么转起来的。

---

## 一、没有 Harness 时会发生什么

我们先看一个真实的反面教材，感受一下 AI Slop 是怎么蔓延的。

### 场景：一个用 Vibe Coding 开发的 AI 内嵌 H5 应用

**第 1 周**

```
工程师：帮我写一个用户登录功能
AI：好的，这里是 useAuth.ts，用了 JWT + localStorage...
```

代码生成，功能跑通，皆大欢喜。

**第 3 周**

```
工程师：帮我给 AI 对话页面加权限验证
AI：好的，这里是 ChatGuard.tsx，用了 sessionStorage + cookie...
```

注意：AI 不记得第 1 周用了 JWT + localStorage，这次用了完全不同的方案。

**第 6 周**

```
工程师：帮我修复一个 bug，用户登录后跳转失败
AI：我看了一下，这里有三套认证逻辑互相冲突...
```

现在代码库里有：
- `useAuth.ts`（JWT + localStorage 方案）
- `ChatGuard.tsx`（sessionStorage + cookie 方案）
- `authMiddleware.ts`（第 4 周某个 Agent 又写了一套）

**第 3 个月**

没有人敢动认证相关的代码。每次改动都会触发神秘 bug。
新加入的工程师看着三套互相矛盾的实现，不知道该信哪个。

这就是**熵增失控**的样子。

## 二、有 Harness 时的工作闭环

有了 harness，同样的场景会完全不同。

### 完整闭环图

```
┌─────────────────────────────────────────────────────┐
│                   Harness 工作闭环                    │
│                                                     │
│  1. 工程师定义约束                                    │
│     └─ 更新 AGENTS.md / 架构文档 / ESLint 规则        │
│              ↓                                      │
│  2. Agent 读取 Harness，理解约束                      │
│     └─ 读 AGENTS.md → 读相关文档 → 开始执行任务        │
│              ↓                                      │
│  3. Agent 执行任务，生成代码                           │
│     └─ 在 git worktree 里独立工作                     │
│              ↓                                      │
│  4. 机械反馈（自动触发）                               │
│     └─ ESLint 扫描 → CI 运行 → 测试执行               │
│              ↓                                      │
│  5. Agent 根据反馈自动修复                             │
│     └─ 读取错误信息 → 理解修复指令 → 修改代码           │
│     └─ 循环直到所有检查通过                            │
│              ↓                                      │
│  6. 人工审查（轻量）                                   │
│     └─ 审查结果，不审查每一行代码                       │
│     └─ 通过 → 合并；不通过 → 反馈给 Agent              │
│              ↓                                      │
│  7. 如果 Agent 犯了新错误                              │
│     └─ 更新 Harness（回到步骤 1）                      │
└─────────────────────────────────────────────────────┘
```

### 关键点：步骤 4 和 5 是自动的

这是 Harness Engineering 最重要的设计思想：

> **让机器来拒绝错误，而不是让人来发现错误。**

ESLint 发现违规 → 报出带修复指令的错误信息 → Agent 读到后自己修复 → 再次检查 → 通过

整个修复循环**不需要人介入**。人只在最后做轻量审查。

## 三、三大支柱预览

Harness 的整套系统建立在三大支柱上：

### 支柱一：上下文工程（Context Engineering）

**解决的问题：** Agent 需要看到正确的信息才能做出正确的决策

核心工具：
- `AGENTS.md`：项目说明书，Agent 每次工作前必读
- `docs/` 目录：深层架构文档、设计文档
- `skills/` 文档：工具使用指南

**类比：** 给新员工的入职手册 + 公司规章制度

### 支柱二：架构约束（Architectural Constraints）

**解决的问题：** 防止 Agent 破坏代码库的结构边界

核心工具：
- 分层架构定义（Types → Config → API → Store → Hooks → Components → Pages）
- 自定义 ESLint 规则（机械执行架构规则）
- CI 流水线（违规代码物理上无法合并）

**类比：** 建筑的承重墙——不管谁来施工，都不能拆承重墙

### 支柱三：熵管理（Entropy Management）

**解决的问题：** 对抗 AI 代码随时间积累的混乱

核心工具：
- 黄金原则文档（定义什么是好代码）
- 周期性清理 Agent（后台持续扫描偏差）
- 质量度量体系（量化代码库的健康状态）

**类比：** 城市的环卫系统——不是一次性打扫，而是持续维护

## 四、Harness Engineering 工程师的日常

传统前端工程师的一天：写组件、调样式、写组件、调 bug...

Harness Engineering 工程师的一天：

### 早上：审查 Agent 的 PR

```
打开 PR 列表
├── PR #234：Agent 完成了 AI 对话消息气泡组件（CI 全绿）→ 30秒扫一眼，合并
├── PR #235：Agent 重构了请求拦截器（CI 全绿）→ 1分钟看关键改动，合并
└── PR #236：Agent 尝试在 Page 层直接调用 API（CI 报错：违反分层规则）→ 看错误信息，确认 ESLint 规则正确，关闭 PR，Agent 会自动重试
```

大多数 PR 在 1 分钟内完成审查。

### 上午：处理 Harness 改进

```
昨天 Agent 犯了一个错误：在 Component 层直接写了 fetch 请求
→ 这违反了"网络请求只能在 API 层"的规则
→ 但 ESLint 没有捕获到这个违规
→ 工程师：给 ESLint 加一条新规则，检测 Component 层的直接 fetch 调用
→ 更新 AGENTS.md，加一行提示
→ 这个错误以后永远不会再发生
```

### 下午：分配新任务给 Agent

```
把下一个功能需求拆解成 3 个独立任务
→ 为每个任务写清晰的任务描述
→ 分配给 3 个 Agent，在 3 个 worktree 里并行执行
→ 去做其他事，等 Agent 完成后审查 PR
```

### 偶尔：Harness Review

```
每隔 2 周，检查一次 Harness 本身的健康状态：
- AGENTS.md 是否还准确？
- 有没有过时的 ESLint 规则？
- 黄金原则是否需要更新？
```

**核心感受：** 工程师不再是代码的生产者，而是**代码质量系统的维护者**。

## 五、一个完整的例子串联起来

**任务：** 给 AI 内嵌 H5 应用添加"AI 回复流式渲染"功能

### 没有 Harness 的做法

```
工程师 → 告诉 AI "加流式渲染" → AI 生成代码
→ 代码里直接在 ChatPage 组件里调用了 AI SDK
→ 工程师没注意，合并了
→ 3 个月后，AI SDK 调用散落在 5 个不同的组件里
→ 没人知道改哪里
```

### 有 Harness 的做法

```
工程师 → 创建任务描述 → 分配给 Agent

Agent 执行：
1. 读 AGENTS.md → 了解项目架构
2. 读 docs/architecture.md → 发现"AI SDK 调用必须通过 AIProvider 封装"
3. 生成代码：在 API 层创建 aiApi.ts，封装流式请求
4. 在 Hooks 层创建 useAIStream Hook，调用 aiApi
5. 在 Component 层使用 useAIStream（而不是直接调用 SDK）

ESLint 检查：
→ 通过（符合分层规则）

CI 检查：
→ 通过

工程师审查：
→ 30秒确认结构正确，合并

结果：
→ AI SDK 调用集中在 API 层
→ 以后所有 AI 相关的改动都在同一个地方
→ 代码库结构保持整洁
```

## 本节小结

```
没有 Harness：
  AI 自由生成 → 风格混乱 → 熵快速增长 → 代码库腐化

有 Harness 的闭环：
  定义约束 → Agent 执行 → 机械反馈 → 自动修复 → 人工审查
                                              ↑
                              Agent 犯新错 → 更新 Harness（永久修复）

三大支柱：
  上下文工程（让 Agent 看到正确信息）
  架构约束（机械阻止 Agent 犯结构性错误）
  熵管理（持续对抗代码腐化）

工程师角色：
  不再是代码生产者，而是代码质量系统的维护者
```
