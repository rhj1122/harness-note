# 7.4　文档管理工具

> 本节目标：掌握 Markdown + Git 的文档版本化方案，以及如何自动化检测文档是否过时。

---

## 一、为什么文档要和代码放在一起

第二章讲过这个原则，这里从工具角度再强调一次：

```
文档放在外部系统（Confluence、Notion）的问题：
  - Agent 无法直接读取
  - 代码变了，文档没有跟着变（版本不同步）
  - 没有 diff，不知道文档改了什么

文档放在 Git 仓库的好处：
  - Agent 可以直接读取（read_file）
  - 文档和代码在同一个 PR 里更新（版本同步）
  - git blame 可以追溯文档的变更历史
  - git diff 可以看到文档的具体改动
```

## 三、文档版本化的最佳实践

### 实践一：文档和代码在同一个 PR

```
错误做法：
  PR #234：修改了 src/api/aiApi.ts（新增了 streamMessage 函数）
  PR #235（一周后）：更新 docs/modules/ai-chat.md

正确做法：
  PR #234：
    - 修改 src/api/aiApi.ts（新增 streamMessage 函数）
    - 同时更新 docs/modules/ai-chat.md（描述新函数）
```

在 PR 模板里加一条提醒：

```markdown
## PR 检查清单
- [ ] 如果修改了公开接口（Hook、组件 Props、API 函数），是否同步更新了对应文档？
- [ ] 如果修改了架构，是否更新了 docs/architecture.md？
```

### 实践二：设计决策记录（ADR）

每次做重要的技术决策，写一份 ADR（Architecture Decision Record）：

```markdown
# 决策 003：使用 AIProvider 模式封装 AI SDK

## 状态
已采纳（2026-03-15）

## 背景
项目需要调用 AI API 实现流式对话功能。
直接在 Hook 里使用 AI SDK 会导致：
- AI SDK 散落在多个 Hook 里
- 换 AI 服务商需要改多处
- 无法统一处理 Token 计费和错误

## 决策
创建 AIProvider（src/providers/AIProvider.tsx），
统一封装所有 AI SDK 调用。
所有需要调用 AI 的 Hook 通过 useAIProvider() 获取客户端。

## 原因
- 集中管理 AI SDK 初始化
- 换服务商只改 AIProvider
- 便于统一添加监控和错误处理

## 后果
- 所有 AI 调用必须通过 AIProvider（ESLint 规则强制）
- 新增 AI 功能时，先在 AIProvider 里添加方法
```

ADR 的价值：当 Agent 想"改进"某个设计时，它能看到当初为什么这样设计，避免推翻有充分理由的决策。

## 四、文档过时检测自动化

### 方法一：markdown-link-check（检测死链）

```bash
# 安装
npm install -D markdown-link-check

# 检测所有文档中的链接是否有效
npx markdown-link-check docs/**/*.md
```

在 CI 中使用：
```yaml
- name: 检测文档死链
  run: npx markdown-link-check docs/**/*.md --config .mlc-config.json
```

```json
// .mlc-config.json
{
  "ignorePatterns": [
    { "pattern": "^http" }  // 忽略外部链接（避免网络问题导致 CI 失败）
  ],
  "aliveStatusCodes": [200, 206]
}
```

### 方法二：检测代码引用是否还有效

写一个简单脚本，检测文档中引用的文件路径是否还存在：

```javascript
// scripts/check-doc-refs.js
import { readFileSync, existsSync } from 'fs'
import { glob } from 'glob'
import path from 'path'

const docFiles = glob.sync('docs/**/*.md')
let hasError = false

for (const docFile of docFiles) {
  const content = readFileSync(docFile, 'utf-8')

  // 匹配文档中引用的文件路径（如 src/hooks/useAIChat.ts）
  const fileRefs = content.match(/src\/[\w\/\-\.]+\.(ts|tsx|js|jsx)/g) || []

  for (const ref of fileRefs) {
    if (!existsSync(ref)) {
      console.error(`❌ ${docFile}: 引用了不存在的文件 ${ref}`)
      hasError = true
    }
  }
}

if (hasError) {
  console.error('\n文档中存在失效的文件引用，请更新文档。')
  process.exit(1)
} else {
  console.log('✅ 所有文档引用有效')
}
```

在 CI 中使用：
```yaml
- name: 检测文档文件引用
  run: node scripts/check-doc-refs.js
```

### 方法三：AGENTS.md 行数检查

```yaml
- name: 检查 AGENTS.md 行数
  run: |
    lines=$(wc -l < AGENTS.md)
    echo "AGENTS.md 当前行数：$lines"
    if [ "$lines" -gt 120 ]; then
      echo "⚠️ AGENTS.md 超过 120 行，建议精简（当前 $lines 行）"
      # 警告但不阻断
    fi
```

## 五、文档的 Git 工作流

```
文档变更的 commit 规范：

docs: 更新 architecture.md，新增 AIProvider 层说明
docs: 修复 conventions.md 中的过时命令
docs: 新增 ADR-003 关于 AIProvider 模式的决策记录
docs(skills): 更新 verify-changes.md，添加 Playwright 验证步骤
```

在 `.github/PULL_REQUEST_TEMPLATE.md` 里加文档检查项：

```markdown
## 文档更新
- [ ] 修改了公开接口 → 已更新对应模块文档
- [ ] 修改了架构 → 已更新 docs/architecture.md
- [ ] 做了重要技术决策 → 已在 docs/decisions/ 新增 ADR
- [ ] 修改了常用命令 → 已更新 AGENTS.md 和 docs/conventions.md
```

## 本节小结

```
文档版本化原则：
  文档和代码放在同一个 Git 仓库
  文档和代码在同一个 PR 里更新

完整 docs/ 结构：
  README.md（总索引）
  architecture.md / conventions.md / tech-choices.md（必备）
  decisions/（ADR，记录重要决策）
  modules/（模块详细说明）
  skills/（工具使用指南）
  tools/（工具目录）

文档过时检测：
  markdown-link-check（检测死链）
  自定义脚本（检测文件路径引用）
  AGENTS.md 行数检查

ADR 的价值：
  记录"为什么这样设计"
  防止 Agent 推翻有充分理由的决策
```
