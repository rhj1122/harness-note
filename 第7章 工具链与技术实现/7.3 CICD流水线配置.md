# 7.3　CI/CD 流水线配置

> 本节目标：掌握前端 Harness CI 的完整 GitHub Actions 配置，以及自动合并机器人的设置方法。

---

## 一、完整的 Harness CI 配置

```yaml
# .github/workflows/harness-ci.yml

name: Harness CI

on:
  pull_request:
    branches:
      - main
      - master
      - 'release-**'
  push:
    branches:
      - main
      - master
      - 'dev-**'
      - 'fix-**'
      - 'release-**'

jobs:
  # ─────────────────────────────────────────
  # 阶段一：快速检查（约 30-60 秒）
  # ─────────────────────────────────────────
  quick-check:
    name: 快速检查（格式 + 类型）
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'

      - name: 安装依赖
        run: pnpm install --frozen-lockfile

      - name: 格式检查（Prettier）
        run: pnpm prettier --check "src/**/*.{ts,tsx}"

      - name: TypeScript 类型检查
        run: pnpm tsc --noEmit

  # ─────────────────────────────────────────
  # 阶段二：架构约束检查（约 1 分钟）
  # ─────────────────────────────────────────
  arch-check:
    name: 架构约束检查（ESLint + 循环依赖）
    needs: quick-check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'

      - name: 安装依赖
        run: pnpm install --frozen-lockfile

      - name: ESLint 检查（含自定义 Harness 规则）
        run: pnpm eslint src --ext .ts,.tsx --max-warnings 0

      - name: 循环依赖检测
        run: |
          result=$(pnpm madge --circular src 2>/dev/null || true)
          if echo "$result" | grep -q "Found"; then
            echo "❌ 发现循环依赖："
            echo "$result"
            exit 1
          fi
          echo "✅ 无循环依赖"

  # ─────────────────────────────────────────
  # 阶段三：测试（约 2-5 分钟）
  # ─────────────────────────────────────────
  test:
    name: 测试（Vitest + 覆盖率）
    needs: arch-check
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'

      - name: 安装依赖
        run: pnpm install --frozen-lockfile

      - name: 运行测试
        run: pnpm vitest run

      - name: 覆盖率检查
        run: pnpm vitest run --coverage --coverage.thresholds.lines=80

  # ─────────────────────────────────────────
  # 阶段四：构建验证（约 1-2 分钟）
  # ─────────────────────────────────────────
  build:
    name: 构建验证
    needs: test
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'

      - name: 安装依赖
        run: pnpm install --frozen-lockfile

      - name: 构建
        run: pnpm run build

      - name: 上传构建产物（可选）
        uses: actions/upload-artifact@v4
        with:
          name: dist
          path: dist/
          retention-days: 3
```

## 二、周期性清理扫描配置

```yaml
# .github/workflows/weekly-cleanup.yml

name: 周期性质量扫描

on:
  schedule:
    - cron: '0 2 * * 1'  # 每周一凌晨 2 点
  workflow_dispatch:       # 支持手动触发

jobs:
  quality-scan:
    name: 代码质量扫描
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v4
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'pnpm'

      - name: 安装依赖
        run: pnpm install --frozen-lockfile

      - name: 死代码扫描
        run: pnpm ts-prune > reports/dead-code.txt 2>&1 || true

      - name: 重复代码扫描
        run: pnpm jscpd src --min-lines 10 --reporters json --output reports/ 2>/dev/null || true

      - name: 循环依赖扫描
        run: pnpm madge --circular src > reports/circular-deps.txt 2>&1 || true

      - name: 上传扫描报告
        uses: actions/upload-artifact@v4
        with:
          name: quality-report-${{ github.run_number }}
          path: reports/
          retention-days: 30
```

## 三、自动合并机器人配置

对于满足条件的 PR，配置自动合并：

```yaml
# .github/workflows/auto-merge.yml

name: 自动合并

on:
  pull_request:
    types: [opened, synchronize, reopened]
  check_suite:
    types: [completed]

jobs:
  auto-merge:
    name: 自动合并（满足条件的 PR）
    runs-on: ubuntu-latest
    if: github.actor == 'github-actions[bot]' || contains(github.event.pull_request.labels.*.name, 'auto-merge')
    steps:
      - name: 检查 CI 状态并自动合并
        uses: pascalgn/automerge-action@v0.16.2
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          MERGE_METHOD: squash
          MERGE_LABELS: auto-merge
          MERGE_REQUIRED_APPROVALS: 0
          UPDATE_METHOD: rebase
```

**使用方式：** 给 PR 打上 `auto-merge` 标签，CI 全绿后自动合并。

适合打 `auto-merge` 标签的 PR 类型：
- 文档更新
- 清理 Agent 生成的重构 PR
- 依赖 patch 版本升级

## 四、PR 状态检查配置

在 GitHub 仓库设置里，配置必须通过的状态检查：

```
Settings → Branches → Branch protection rules → main

Required status checks:
  ✅ 快速检查（格式 + 类型）
  ✅ 架构约束检查（ESLint + 循环依赖）
  ✅ 测试（Vitest + 覆盖率）
  ✅ 构建验证

Require branches to be up to date before merging: ✅
```

这样配置后，任何不满足条件的 PR 物理上无法合并。

## 五、CI 失败时的友好提示

让 CI 失败信息对 Agent 更友好：

```yaml
# 在测试步骤中添加失败摘要
- name: 运行测试
  run: pnpm vitest run
  continue-on-error: true
  id: test-step

- name: 输出测试失败摘要
  if: steps.test-step.outcome == 'failure'
  run: |
    echo "## ❌ 测试失败" >> $GITHUB_STEP_SUMMARY
    echo "" >> $GITHUB_STEP_SUMMARY
    echo "请运行 \`pnpm vitest run\` 查看详细失败信息。" >> $GITHUB_STEP_SUMMARY
    echo "修复后重新推送即可触发 CI 重新运行。" >> $GITHUB_STEP_SUMMARY
    exit 1
```

## 六、本地 pre-commit 钩子（可选）

在提交前自动运行快速检查，减少 CI 失败：

```bash
# 安装 husky
pnpm add -D husky lint-staged
pnpm husky init
```

```bash
# .husky/pre-commit
pnpm run check
```

```json
// package.json
{
  "lint-staged": {
    "src/**/*.{ts,tsx}": [
      "pnpm prettier --write",
      "pnpm eslint --fix"
    ]
  }
}
```

**注意：** pre-commit 钩子是可选的。如果 Agent 在 worktree 里工作，钩子可能会干扰 Agent 的提交流程。可以在 worktree 里禁用：

```bash
# 在 worktree 里临时禁用 pre-commit 钩子
HUSKY=0 git commit -m "..."
```

## 本节小结

```
完整 CI 流水线（四阶段）：
  阶段一：快速检查（Prettier + tsc）← 30-60秒
  阶段二：架构约束（ESLint + madge）← 1分钟
  阶段三：测试（Vitest + 覆盖率）← 2-5分钟
  阶段四：构建验证（vite build）← 1-2分钟

周期性扫描：
  每周一自动运行
  扫描死代码、重复代码、循环依赖
  上传报告供工程师查看

自动合并：
  打 auto-merge 标签 + CI 全绿 → 自动合并
  适合文档更新、清理 PR、依赖升级

分支保护：
  在 GitHub 设置里配置必须通过的状态检查
  违规 PR 物理上无法合并

pre-commit 钩子（可选）：
  本地提交前自动格式化和检查
  在 worktree 里可以用 HUSKY=0 临时禁用
```
