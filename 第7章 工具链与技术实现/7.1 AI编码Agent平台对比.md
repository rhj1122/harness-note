# 7.1　AI 编码 Agent 平台对比

> 本节目标：了解当前主流的 AI 编码 Agent 平台，理解各自的特点，掌握选型原则——尤其是与 Harness 设计的适配性。

---

## 一、为什么选型很重要

不同的 AI 编码平台，对 Harness 的支持程度差异很大：

```
关键问题：
  这个平台能读取 AGENTS.md 吗？
  这个平台能运行 ESLint 和测试吗？
  这个平台支持 Git Worktree 并行工作吗？
  这个平台的错误反馈能被 Agent 自动读取吗？

如果平台不支持这些，Harness 的很多设计就无法发挥作用。
```

---

## 二、主流平台对比

### Claude Code（Anthropic）

```
类型：CLI 工具 + IDE 扩展
运行方式：在本地终端运行，直接操作文件系统

Harness 适配性：
  ✅ 自动读取 AGENTS.md（原生支持）
  ✅ 可以运行任意 shell 命令（npm run check、vitest 等）
  ✅ 支持 Git Worktree
  ✅ 可以读取 ESLint 错误输出并自动修复
  ✅ 支持长任务自主执行

适合场景：
  本地开发环境
  需要深度集成 Harness 的项目
  个人开发者或小团队

注意：
  需要在本地安装和配置
  API 费用按 token 计费
```

### Cursor

```
类型：基于 VS Code 的 AI IDE
运行方式：在编辑器内运行，有 Agent 模式

Harness 适配性：
  ✅ 可以读取项目文件（包括 AGENTS.md）
  ✅ 可以运行终端命令
  ✅ Agent 模式支持多步骤自主执行
  ⚠️ AGENTS.md 需要手动引导 Agent 读取（不像 Claude Code 那样原生支持）
  ⚠️ 长任务稳定性不如专用 CLI 工具

适合场景：
  习惯 VS Code 工作流的工程师
  需要 AI 辅助 + 手动编码混合工作的场景

注意：
  订阅制收费
  Agent 模式在复杂任务上可能需要多次引导
```

### GitHub Copilot Workspace

```
类型：基于 GitHub 的云端 Agent
运行方式：在浏览器中运行，与 GitHub 深度集成

Harness 适配性：
  ✅ 与 GitHub PR 流程原生集成
  ✅ 可以读取仓库文件
  ⚠️ 对 AGENTS.md 的支持有限（不如本地工具）
  ⚠️ 无法直接运行本地命令（ESLint、测试等）
  ❌ 不支持本地 Git Worktree

适合场景：
  以 GitHub 为中心的工作流
  代码审查辅助
  简单的代码生成任务

注意：
  不适合需要深度 Harness 集成的场景
  更适合作为辅助工具，而非主力 Agent
```

### OpenAI Codex（2026 新版）

```
类型：云端 Agent，通过 API 调用
运行方式：在 OpenAI 的沙箱环境中运行

Harness 适配性：
  ✅ 原生支持 AGENTS.md（这是 Harness Engineering 的起源平台）
  ✅ 可以运行 shell 命令
  ✅ 支持 Git Worktree
  ✅ 与 GitHub 集成，可以自动提 PR
  ✅ 支持大规模并行任务

适合场景：
  需要大量并行 Agent 任务的团队
  OpenAI 生态的用户

注意：
  目前主要面向企业用户
  需要通过 API 或 ChatGPT 界面使用
```

---

## 三、选型原则

### 原则一：AGENTS.md 支持是必要条件

```
如果平台不能原生读取 AGENTS.md：
  → 每次任务都需要手动把 AGENTS.md 内容粘贴给 Agent
  → Harness 的自动化优势大打折扣
  → 不推荐作为主力平台
```

### 原则二：能运行本地命令是核心能力

```
Harness 的短回路依赖 Agent 能自己运行：
  npm run check（tsc + ESLint）
  npx vitest run
  npm run build

如果平台不能运行这些命令：
  → Agent 无法自我验证
  → 所有问题都要等 CI 才能发现
  → 反馈回路变长，效率下降
```

### 原则三：根据团队规模选择

```
个人开发者：
  Claude Code 或 Cursor（本地工具，灵活）

小团队（3-5人）：
  Claude Code（深度 Harness 集成）
  或 Cursor（如果团队习惯 VS Code）

中大型团队（5人以上）：
  OpenAI Codex（大规模并行）
  或 Claude Code + GitHub Actions（混合方案）
```

---

## 四、混合使用策略

实际工作中，不需要只用一个平台：

```
推荐的混合方案：

日常开发（本地）：
  Claude Code 或 Cursor
  → 快速迭代，深度 Harness 集成

大批量任务（云端）：
  OpenAI Codex 或 GitHub Copilot Workspace
  → 并行处理大量重复性任务（写测试、更新文档等）

代码审查辅助：
  GitHub Copilot（PR 审查建议）
  → 不替代 Harness，作为补充
```

---

## 本节小结

```
主流平台对比：
  Claude Code：本地 CLI，Harness 适配性最好
  Cursor：VS Code IDE，适合混合工作流
  GitHub Copilot Workspace：GitHub 集成，Harness 支持有限
  OpenAI Codex：云端大规模并行，Harness 起源平台

选型核心原则：
  必须支持读取 AGENTS.md
  必须能运行本地命令（ESLint、测试、构建）
  根据团队规模选择

推荐方案：
  个人/小团队：Claude Code 或 Cursor
  中大型团队：OpenAI Codex 或混合方案
```
