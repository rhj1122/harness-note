# 8.2　案例二：失败的 Harness 设计反例

> 本节目标：通过三个典型的失败案例，理解 Harness 设计中最常见的陷阱，以及如何避免。

---

## 一、反例一："一个巨型 AGENTS.md"

### 场景

一个前端团队刚开始使用 Harness Engineering，非常认真，把所有规则都写进了 AGENTS.md：

```
AGENTS.md（最终变成了 800 行）：
  - 项目概述（50行）
  - 完整的架构文档（200行）
  - 所有编码规范（150行）
  - 所有 ESLint 规则说明（100行）
  - 所有黄金原则（150行）
  - 常用命令（30行）
  - 模块说明（120行）
```

### 发生了什么

```
第一周：
  Agent 读取 AGENTS.md，表现良好
  工程师很满意

第三周：
  Agent 开始违反一些规则
  工程师困惑：规则明明写在 AGENTS.md 里

第六周：
  Agent 频繁违反规则
  工程师发现：Agent 在执行复杂任务时，
  AGENTS.md 的内容被推出了上下文窗口
  → Agent 根本没有"看到"那些规则
```

### 根本原因

```
上下文窗口的注意力分布：

开头 ████████████ 高注意力
中间 ████░░░░░░░░ 低注意力  ← 大量规则埋在这里
结尾 ████████████ 高注意力

800 行的 AGENTS.md：
  - 占用大量上下文空间
  - 关键规则被埋在中间，Agent 注意力低
  - 任务复杂时，早期读取的内容被"遗忘"
```

### 正确做法

```
AGENTS.md（约 30 行）：
  项目概述（3行）
  架构概览（5行，指向 docs/architecture.md）
  最关键的 5 条规则（直接写）
  常用命令（5行）
  文档入口（1行：docs/README.md）

docs/（详细内容）：
  architecture.md（完整架构）
  conventions.md（完整规范）
  golden-principles.md（黄金原则）
  ...
```

**教训：AGENTS.md 是地图，不是百科全书。越短越好，超过 100 行就要精简。**

## 二、反例二：缺少架构约束导致的代码混乱

### 场景

一个 AI H5 项目，工程师写了很好的 AGENTS.md，描述了分层架构，但没有配置 ESLint 规则来机械执行。

```
AGENTS.md 里写了：
  "Components 层禁止直接调用 fetch 或 AI SDK"
  "Pages 层禁止直接操作 Store"
```

### 发生了什么

```
第 1 周：Agent 遵守规则，代码结构良好

第 3 周：
  Agent 在赶一个紧急功能
  在 ChatMessage 组件里直接调用了 AI SDK
  工程师审查时没注意，合并了

第 5 周：
  另一个 Agent 看到了 ChatMessage 里的直接 AI SDK 调用
  认为"这是项目里的正常做法"
  在 3 个新组件里也直接调用了 AI SDK

第 8 周：
  代码库里有 7 个组件直接调用 AI SDK
  AI SDK 初始化逻辑散落各处
  换 AI 服务商时，需要改 7 个文件
  其中 2 个文件的初始化参数不一致，导致 bug
```

### 根本原因

```
文字规则 vs 机械约束：

文字规则（AGENTS.md）：
  依赖 Agent 的理解和记忆
  Agent 偶尔会"忘记"或"误判"
  一旦有先例违规，后续 Agent 会复制

机械约束（ESLint）：
  不依赖 Agent，代码提交时自动扫描
  违规立即报错，无法绕过
  没有先例效应
```

### 正确做法

```
光有文字规则不够，必须配套 ESLint 规则：

// eslint-rules/no-ai-sdk-in-components.js
// 检测 src/components/ 下是否直接导入 AI SDK
// 错误信息：告诉 Agent 应该通过 useAIProvider() 调用

这样：
  Agent 写了违规代码
  → ESLint 立即报错
  → Agent 读到错误信息，自动修复
  → 违规代码永远无法进入代码库
```

**教训：重要的架构规则必须机械化。光靠文字说明，迟早会被违反。**

## 三、反例三：没有熵管理的项目三个月后的样子

### 场景

一个团队搭建了不错的 Harness：有 AGENTS.md、有 ESLint 规则、有 CI。但没有配置任何熵管理（没有黄金原则、没有周期性清理）。

### 三个月后的代码库

```
重复实现：
  useAuth（第 1 周写的）
  useUserAuth（第 4 周，Agent 不知道上面那个）
  checkAuthStatus（第 7 周，又一个）
  isAuthenticated（第 10 周，工具函数版本）
  → 4 套认证逻辑，没有人知道该用哪个

死代码：
  src/hooks/ 目录下有 23 个 Hook
  其中 8 个从未被任何组件引用
  → 占用代码库空间，增加认知负担

风格混乱：
  有的组件用 const Foo: React.FC = () => {}
  有的用 function Foo() {}
  有的用 export default function Foo() {}
  → 同一个项目，三种写法并存

过度工程化：
  一个简单的"格式化时间"需求
  Agent 写了一个支持 12 种格式、带缓存、带国际化的"时间格式化引擎"
  → 200 行代码，实际只用了其中 10 行的功能
```

### 工程师的感受

```
"我们的代码能跑，CI 也是绿的，
 但每次要改一个功能，都要花很长时间
 搞清楚应该改哪个文件、用哪个 Hook。
 感觉代码库越来越难维护了。"
```

这就是**熵增失控**的典型症状：代码能跑，但越来越难维护。

### 根本原因

```
ESLint 只能检测"架构违规"（放错地方）
ESLint 无法检测"质量问题"（写得不好）

没有黄金原则：
  Agent 不知道"好代码"的标准
  → 每次生成风格不同

没有周期性清理：
  Slop 持续积累，没有对冲机制
  → 三个月后积重难返
```

### 正确做法

```
补充黄金原则（docs/golden-principles.md）：
  "开始实现新功能前，先搜索是否已有类似实现"
  "Hook 只做一件事，名称清晰描述功能"
  "避免过度抽象，先实现，再提取"

配置周期性清理（GitHub Actions 定时任务）：
  每周扫描死代码（ts-prune）
  每周扫描重复代码（jscpd）
  自动开清理 PR
```

**教训：ESLint 保证代码"放对地方"，黄金原则 + 清理流程保证代码"写得好"。两者缺一不可。**

## 四、三个反例的共同规律

```
反例一（巨型 AGENTS.md）：
  根因：把 Harness 当成"越详细越好"的文档
  教训：Harness 的每个部分都有最优规模，超过就适得其反

反例二（缺少架构约束）：
  根因：只用文字规则，没有机械执行
  教训：重要规则必须机械化，不能依赖 Agent 的自觉

反例三（没有熵管理）：
  根因：只关注"代码能跑"，忽视"代码好维护"
  教训：Harness 需要三大支柱齐全：上下文工程 + 架构约束 + 熵管理

共同规律：
  Harness 的每个组件都有其存在的理由
  缺少任何一个，都会在某个维度上失控
```

## 本节小结

```
反例一：巨型 AGENTS.md
  症状：Agent 开始忽略规则
  根因：上下文污染 + 注意力稀释
  修复：AGENTS.md 控制在 100 行，详细内容放 docs/

反例二：只有文字规则，没有 ESLint
  症状：违规代码积累，架构腐化
  根因：文字规则依赖 Agent 自觉，有先例效应
  修复：重要规则必须写成 ESLint 规则机械执行

反例三：没有熵管理
  症状：代码能跑但越来越难维护
  根因：Slop 持续积累，没有对冲机制
  修复：黄金原则 + 周期性清理流程

共同规律：
  Harness 三大支柱缺一不可
  上下文工程 + 架构约束 + 熵管理
```
