# 8.1　案例一：OpenAI 内部 100 万行代码实验

> 本节目标：深入理解 Harness Engineering 的起源案例，从中提取可以直接应用到自己项目的经验。

---

## 一、实验背景

2026 年 2 月，OpenAI 工程师 Ryan Lopopolo 在官方博客发布了一篇文章，记录了一个历时五个月的内部实验：

```
实验规模：
  团队：3 名工程师
  时间：5 个月
  代码量：约 100 万行（全部由 AI Agent 生成）
  手写代码：0 行
  PR 数量：约 1,500 个

效率数据：
  初期：人均等效 0.25 名工程师（比一个人还慢）
  成熟期：人均等效 3-10 名工程师
  估计节省：约 90% 的传统开发时间
```

这个实验直接催生了 Harness Engineering 这个概念的正式命名。

---

## 二、Harness 设计的关键决策

### 决策一：AGENTS.md 是目录，不是百科全书

```
他们踩过的坑：
  最初把所有规则都塞进一个 AGENTS.md
  → 文件越来越长，Agent 开始忽略中间部分的规则
  → 违规率反而上升

最终方案：
  AGENTS.md 约 100 行，只写入口信息
  详细内容分散在 docs/ 各文件里
  → Agent 每次都能完整读取 AGENTS.md
  → 需要深入了解时，按需读取 docs/
```

### 决策二：Linter 错误信息 = 修复指令

```
他们踩过的坑：
  普通 Linter 只说"这里有问题"
  → Agent 不知道怎么修
  → 反复尝试，浪费大量 token

最终方案：
  每条 Linter 规则的错误信息包含：
    - 违反了什么规则
    - 为什么不能这样
    - 应该怎么修
  → Agent 读到错误信息，直接修复，不需要人介入
```

### 决策三：架构分层 + 机械强制

```
他们踩过的坑：
  只在文档里写架构规则
  → Agent 偶尔"忘记"规则
  → 违规代码积累，越来越难清理

最终方案：
  把架构规则编码进 Linter
  CI 层面机械拒绝违规代码
  → 违规代码物理上无法进入主分支
```

### 决策四：每周 20% 时间清理 Slop → 自动化清理

```
他们踩过的坑：
  初期没有熵管理
  → 每周需要花 20% 时间（相当于一整天）手动清理 AI Slop
  → 严重影响开发节奏

最终方案：
  后台 Agent 持续扫描代码库
  自动开清理 PR
  → 人工清理时间降至接近 0
```

## 三、踩过的坑与解决方案

### 坑一：Agent 不知道"已有实现"

```
问题：
  Agent 写新功能时，不知道代码库里已经有类似的实现
  → 重复造轮子，同一个功能出现多个版本

解决：
  在 AGENTS.md 里加一条规则：
  "开始实现新功能前，先搜索代码库，确认是否已有类似实现"
  
  同时在 docs/ 里维护"已有功能索引"
  → Agent 开始工作前先查索引
```

### 坑二：文档和代码脱节

```
问题：
  代码在快速迭代，文档跟不上
  → Agent 读到过时文档，按旧架构写代码
  → 大量 PR 因架构违规被 CI 拒绝

解决：
  用后台 Agent 定期扫描文档
  发现文档与代码不一致时，自动开 PR 更新文档
  → 文档始终保持最新
```

### 坑三：并行 Agent 产生冲突

```
问题：
  多个 Agent 同时修改同一个文件
  → 合并时产生大量冲突
  → 解决冲突比写代码还费时间

解决：
  任务分解时确保每个任务的文件范围不重叠
  使用 Git Worktree 隔离工作目录
  → 冲突率大幅下降
```

## 四、效率数据背后的原因

为什么初期只有 0.25 名工程师的效率，后来能达到 3-10 名？

```
初期（0.25 名工程师效率）：
  Harness 还不完善
  Agent 频繁犯错
  工程师花大量时间：
    - 修复 Agent 的错误
    - 手动清理 Slop
    - 解决合并冲突
    - 更新过时文档
  → 工程师的时间被"维护 Agent 的错误"消耗殆尽

成熟期（3-10 名工程师效率）：
  Harness 已经完善
  Agent 犯错率大幅下降
  大部分 PR 自动通过 CI
  工程师的时间主要用于：
    - 架构决策
    - 审查 PR（1 分钟/个）
    - 偶尔更新 Harness
  → 工程师的时间被"高价值工作"占满
```

**关键洞察：Harness 的价值不是让 Agent 更聪明，而是让工程师不需要花时间处理 Agent 的错误。**

## 五、对你的前端项目的启示

这个案例里的经验，直接适用于你的 AI H5 项目：

```
可以直接复用的经验：

1. AGENTS.md 控制在 100 行以内
   → 你的项目也一样，超过就移到 docs/

2. ESLint 错误信息写清楚修复方法
   → 3.3 节已经讲了具体写法

3. 先搭最小 Harness，随项目演进完善
   → 不要等项目"足够大"才开始

4. 每次 Agent 犯错，立即更新 Harness
   → 这是最重要的习惯，不要拖延

5. 熵管理要自动化，不要靠人工清理
   → 4.3 节的周期性清理流程

需要根据前端特点调整的：

1. 分层架构不同
   → 后端是 Types→Repo→Service→Runtime
   → 前端是 Types→API→Store→Hooks→Components→Pages

2. Linter 工具不同
   → 后端可能用 pylint/rubocop
   → 前端用 ESLint（更成熟，规则更丰富）

3. 可观测性工具不同
   → 后端关注日志、指标、链路追踪
   → 前端关注 Web Vitals、错误监控、AI 调用质量
```

## 本节小结

```
实验规模：
  3 名工程师 × 5 个月 × 100 万行代码 × 0 行手写代码

关键决策：
  AGENTS.md 是目录（约 100 行）
  Linter 错误信息 = 修复指令
  架构规则机械化（不靠文字说明）
  熵管理自动化（不靠人工清理）

效率演进：
  初期 0.25 名 → 成熟期 3-10 名
  关键：Harness 完善后，工程师时间从"处理错误"转向"高价值工作"

对前端项目的启示：
  核心经验完全适用
  分层架构和工具需要根据前端特点调整
```
