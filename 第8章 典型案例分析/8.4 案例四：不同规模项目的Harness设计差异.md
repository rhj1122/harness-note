# 8.4　案例四：不同规模项目的 Harness 设计差异

> 本节目标：理解 Harness 不是一套固定的配置，而是根据项目规模和团队大小灵活调整的系统。

---

## 一、规模决定复杂度

Harness 的设计原则是：**刚好够用，不过度工程化。**

```
个人项目用企业级 Harness：
  → 花大量时间维护 Harness，而不是写功能
  → 得不偿失

企业级项目用个人项目的 Harness：
  → 约束不够，Slop 快速积累
  → 团队协作混乱

正确做法：
  根据项目规模，选择合适的 Harness 配置
  随规模增长，逐步升级 Harness
```

---

## 二、个人项目：最小 Harness

**适用场景：** 一个人开发，AI H5 应用，代码量 < 5000 行

### 配置清单

```
必须有：
  ✅ AGENTS.md（约 20-30 行）
  ✅ docs/architecture.md（分层说明，约 50 行）
  ✅ 3 条核心 ESLint 规则
  ✅ npm run check 脚本

可以没有：
  ⬜ 完整的 docs/ 体系（太小的项目不需要）
  ⬜ 黄金原则（规则少，直接写进 AGENTS.md）
  ⬜ CI 流水线（本地检查就够了）
  ⬜ 周期性清理（手动清理即可）
  ⬜ Git Worktree（单人不需要并行）
```

### 示例 AGENTS.md

```markdown
# ai-h5-demo

## 项目
AI 对话 H5 页面，React + TypeScript + Zustand + Vite。

## 架构
Types → Config → API → Store → Hooks → Components → Pages
依赖只能向下。详见 docs/architecture.md

## 规则
- Components 禁止直接 fetch 或调用 AI SDK
- Pages 禁止直接操作 Store
- AI SDK 统一在 src/providers/AIProvider.tsx

## 命令
npm run dev / npm run check / npx vitest run / npm run build
```

### 工作节奏

```
每天：
  给 Agent 分配 1-3 个任务
  审查 PR（5-10 分钟）

每周：
  手动检查有没有明显的 Slop
  如果有，花 30 分钟清理
  如果 Agent 犯了新错误，更新 AGENTS.md
```

## 三、小团队（3-10 人）：标准 Harness

**适用场景：** 3-10 人前端团队，AI H5 应用，代码量 5000-50000 行

### 配置清单

```
必须有：
  ✅ AGENTS.md（约 50-80 行）
  ✅ 完整的 docs/ 体系
      README.md / architecture.md / conventions.md
      tech-choices.md / golden-principles.md
  ✅ 10-15 条 ESLint 规则
  ✅ 完整的 CI 流水线（GitHub Actions）
  ✅ Vitest 测试配置（覆盖率 > 70%）

建议有：
  ✅ 黄金原则（docs/golden-principles.md）
  ✅ 周期性清理（每周自动扫描）
  ✅ 质量度量仪表盘（基础版）
  ✅ PR 模板

可以没有：
  ⬜ Git Worktree（3-5 人时可以不用）
  ⬜ 浏览器自动化（按需使用）
  ⬜ 复杂的自动合并机器人
```

### 团队分工

```
Harness Owner（1人）：
  维护 AGENTS.md 和 docs/
  审批 ESLint 规则变更
  执行 Harness Review（每 2 周）

其他成员：
  发现 Agent 犯错 → 提 PR 更新 Harness
  日常 PR 审查（1 分钟/个）
```

### 工作节奏

```
每天早上（10 分钟）：
  审查 Agent 提交的 PR
  分配新任务

每周（30 分钟）：
  查看质量报告
  处理积累的 Slop

每 2 周（1 小时）：
  Harness Review
  评估是否需要新增规则
```

## 四、企业级：完整 Harness + 治理体系

**适用场景：** 10 人以上团队，多个前端项目，代码量 > 50000 行

### 配置清单

```
必须有（在标准 Harness 基础上）：
  ✅ 分层 AGENTS.md（全局 + 模块级）
  ✅ 完整的 docs/ 体系 + 模块级文档
  ✅ 20-30 条 ESLint 规则
  ✅ 完整的 CI 流水线 + 自动合并
  ✅ Git Worktree 并行工作
  ✅ 周期性清理（自动化）
  ✅ 质量度量仪表盘（完整版）
  ✅ 浏览器自动化（E2E 测试）
  ✅ Harness 治理流程（变更审批、Review 机制）

治理体系：
  ✅ Harness 变更需要 PR + 审批
  ✅ 重大变更需要 ADR（设计决策记录）
  ✅ 定期 Harness Review（每 2 周）
  ✅ 新成员入职 Harness 培训
```

### 多项目的 Harness 共享

企业级项目通常有多个前端项目，可以共享部分 Harness：

```
共享层（所有项目通用）：
  基础 ESLint 规则（格式、类型安全）
  基础 CI 模板
  通用黄金原则

项目层（每个项目独有）：
  AGENTS.md（项目特定规则）
  架构文档（项目特定分层）
  自定义 ESLint 规则（项目特定约束）
```

实现方式：把共享配置发布为内部 npm 包：

```json
// package.json
{
  "devDependencies": {
    "@company/eslint-config-harness": "^1.0.0",
    "@company/ci-templates": "^1.0.0"
  }
}
```

## 五、三种规模的对比总结

| 维度 | 个人项目 | 小团队（3-10人）| 企业级（10人+）|
|------|---------|----------------|--------------|
| AGENTS.md | 20-30 行 | 50-80 行 | 分层，全局+模块 |
| docs/ | 基础 3 个文件 | 完整体系 | 完整体系+模块文档 |
| ESLint 规则 | 3 条 | 10-15 条 | 20-30 条 |
| CI | 可选 | 必须 | 必须+自动合并 |
| 测试覆盖率 | > 60% | > 70% | > 80% |
| 周期性清理 | 手动 | 自动化 | 自动化+报告 |
| Git Worktree | 不需要 | 可选 | 必须 |
| Harness Review | 随时 | 每 2 周 | 每 2 周+治理流程 |
| 搭建时间 | 2 小时 | 1-2 天 | 1-2 周 |

## 六、你现在在哪个阶段

根据你的情况（个人学习，前端工程师，AI H5 应用）：

```
现阶段：个人项目配置
  目标：把最小 Harness 跑通
  重点：AGENTS.md + 3条 ESLint 规则 + npm run check

未来（放到公司项目）：小团队配置
  目标：完整的 docs/ 体系 + CI 流水线
  重点：团队协作规范 + 自动化清理

再往后（如果团队扩大）：企业级配置
  目标：治理体系 + 多项目共享
  重点：Harness 的可维护性和一致性
```

**不需要一步到位。** 先把个人项目的 Harness 跑通，积累经验，再逐步升级。
