# 8.3　案例三：从 Vibe Coding 迁移到 Harness Engineering

> 本节目标：理解如何把一个已经存在 Slop 的项目渐进式迁移到 Harness Engineering，而不是推倒重来。

---

## 一、迁移的现实背景

很多人学习 Harness Engineering 时，面对的不是一个全新的项目，而是一个已经用 vibe coding 开发了一段时间的项目：

```
典型的迁移前状态：
  - 代码能跑，功能基本完整
  - 没有 AGENTS.md，没有 docs/
  - 没有 ESLint 架构规则
  - 代码库里有大量 Slop：
    重复的 Hook、风格不一致的组件、死代码
  - 工程师开始感觉"越来越难维护"
```

**不需要推倒重来。** 渐进式迁移是可行的，而且风险更低。

---

## 二、迁移时机的判断

什么时候应该开始迁移？

```
明确的信号（出现任意一个就该迁移了）：

  代码质量信号：
  ✅ 发现同一个功能有多个实现版本
  ✅ 改一个地方，不确定会影响哪里
  ✅ 新加入的工程师（或 Agent）需要很长时间才能上手

  效率信号：
  ✅ 每次 AI 生成的代码都需要大量修改才能用
  ✅ 花在"修复 AI 错误"上的时间超过了"写新功能"的时间
  ✅ PR 审查越来越费时，因为代码风格混乱

  规模信号：
  ✅ 项目超过 50 个文件
  ✅ 代码库超过 5000 行
```

---

## 三、迁移策略：渐进式，不是大爆炸

```
错误的迁移方式（大爆炸）：
  停止所有开发
  → 花两周时间清理所有 Slop
  → 重写所有文档
  → 配置所有 ESLint 规则
  → 然后重新开始

问题：
  两周内没有新功能交付
  清理过程中可能引入新 bug
  清理完成后，Slop 很快又积累回来（因为没有预防机制）

正确的迁移方式（渐进式）：
  边开发新功能，边逐步建立 Harness
  每周花 20% 的时间（约 1 天）做迁移工作
  4-6 周后，Harness 基本完善
  同时新功能也在持续交付
```

---

## 四、渐进式迁移的四个阶段

### 阶段一：建立基础（第 1 周，约 4 小时）

**目标：让 Agent 有基本的工作规范**

```
工作内容：

1. 写 AGENTS.md（约 1 小时）
   - 描述项目现状（不是理想状态）
   - 列出最关键的 3-5 条规则
   - 加上常用命令

2. 创建 docs/ 基础结构（约 1 小时）
   - docs/README.md（文档索引）
   - docs/architecture.md（描述现有架构，哪怕不完美）
   - docs/tech-choices.md（现有技术栈）

3. 配置基础 ESLint（约 2 小时）
   - 安装 @typescript-eslint
   - 配置 Prettier
   - 先不加自定义架构规则（避免大量报错）

完成后：
  Agent 有了基本的工作规范
  新生成的代码开始有一致性
  不需要清理任何旧代码
```

### 阶段二：建立约束（第 2-3 周，每周约 4 小时）

**目标：把最重要的架构规则机械化**

```
工作内容：

1. 识别最严重的违规模式（约 1 小时）
   运行：npx eslint src --ext .ts,.tsx
   看看现有代码有哪些问题
   选出最严重的 2-3 个模式

2. 为这些模式写 ESLint 规则（约 2 小时）
   注意：先设为 warn，不要设为 error
   → 让 Agent 知道问题，但不阻断现有工作流

3. 观察一周（约 1 小时）
   看 Agent 是否在遵守新规则
   如果遵守率 > 80%，把 warn 改为 error

关键原则：
  不要一次性加太多规则
  每周只加 2-3 条
  先 warn 后 error，给代码库一个适应期
```

### 阶段三：清理存量 Slop（第 3-5 周，每周约 4 小时）

**目标：逐步清理已有的 Slop，不影响正常开发**

```
清理优先级：

P0（立即清理，风险低）：
  死代码（直接删除，不影响功能）
  → 运行 ts-prune，找出未使用的导出
  → 逐个删除，每次删除后运行测试确认

P1（本月清理，风险中等）：
  重复实现（合并，需要测试）
  → 找出功能相似的 Hook/函数
  → 选一个最好的版本，更新所有引用
  → 删除其他版本

P2（下个月清理，风险较高）：
  风格不一致（统一，改动范围大）
  → 用 ESLint --fix 自动修复格式问题
  → 手动统一命名风格

清理节奏：
  每周清理一个 P0 问题
  每两周清理一个 P1 问题
  不要急，稳步推进
```

### 阶段四：完善 Harness（第 5-8 周，每周约 2 小时）

**目标：建立完整的 Harness，进入稳定运转状态**

```
工作内容：

1. 补充黄金原则
   把迁移过程中发现的"好的做法"写成黄金原则

2. 配置 CI 流水线
   把本地的检查搬到 GitHub Actions

3. 配置周期性清理
   设置每周自动扫描的 GitHub Actions

4. 完善 docs/
   补充 decisions/（记录迁移过程中的重要决策）
   补充 skills/（记录 Agent 常用的工作流程）

完成后：
  Harness 进入稳定运转状态
  新代码质量有保障
  存量 Slop 在持续减少
```

---

## 五、迁移过程中的常见问题

### 问题一：ESLint 规则一开始报了几百个错误

```
不要慌，这是正常的。

处理方式：
  1. 先把规则设为 warn（不阻断）
  2. 让 Agent 在做新功能时顺手修复遇到的 warn
  3. 配置周期性清理 Agent，专门处理这些 warn
  4. 4-6 周后，warn 数量降到可接受范围
  5. 把规则改为 error

不要做的事：
  ❌ 一次性让 Agent 修复所有 warn（风险太高）
  ❌ 把规则加进 .eslintignore 忽略（治标不治本）
```

### 问题二：清理 Slop 时引入了新 bug

```
预防措施：
  清理前：确保有足够的测试覆盖
  清理时：每次只改一个文件，改完立即运行测试
  清理后：运行完整测试套件

如果没有测试：
  先补测试，再清理
  "没有测试就不清理"是安全的原则
```

### 问题三：团队成员不理解为什么要迁移

```
沟通策略：
  不要说"我们要做 Harness Engineering"（太抽象）
  要说"我们要解决这些具体问题"：
    - "现在有 4 套认证逻辑，我们要统一成 1 套"
    - "现在改一个功能要找半天文件，我们要建立文档"
    - "现在 AI 生成的代码经常需要大改，我们要减少这个成本"

让结果说话：
  迁移 2 周后，展示具体改善：
    - "这周 AI 生成的代码，只需要改 20% 就能用，上周是 60%"
    - "这周没有发现重复实现，上周发现了 3 个"
```

## 本节小结

```
迁移时机：
  出现重复实现 / 改动影响范围不清 / AI 错误修复时间 > 新功能时间

迁移策略：
  渐进式（边开发边迁移），不是大爆炸（停下来重写）

四个阶段：
  阶段一（第1周）：建立基础（AGENTS.md + docs/ + 基础 ESLint）
  阶段二（第2-3周）：建立约束（架构 ESLint 规则，先 warn 后 error）
  阶段三（第3-5周）：清理存量（死代码 → 重复实现 → 风格统一）
  阶段四（第5-8周）：完善 Harness（CI + 周期清理 + 黄金原则）

常见问题：
  ESLint 报几百个错 → 先 warn，让 Agent 顺手修复
  清理引入 bug → 先补测试，再清理
  团队不理解 → 用具体问题说话，让结果说话
```
